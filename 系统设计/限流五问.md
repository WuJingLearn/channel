## 一、项目里用过限流吗？什么场景？阈值怎么定？

### 1.1 STAR 法则回答模板

**Situation（场景）**：
> 在我负责的对外 OpenAPI 系统中，被多个第三方业务方接入。某次大促期间一个业务方因 bug 死循环调用，接口 QPS 从 200 飙升到 8000，下游 MySQL 慢查询堆积导致服务雪崩。

**Task（任务）**：
> 复盘后需在 **网关层 + 应用层** 加入多级限流，保障核心接口 SLA 99.9%。

**Action（动作）**：
> 选型 Sentinel 做应用层限流，按 `appKey + interface` 维度配置规则，通过 Nacos 做动态规则推送。

**Result（结果）**：
> 上线后异常调用自动限流并触发降级返回缓存数据，下游 DB 压力下降 70%，可用性从 99.5% 提升到 99.95%。

### 1.2 真实业务的 5 大限流场景

| 场景 | 限流维度 | 典型阈值 |
|---|---|---|
| **OpenAPI 对外开放** | appKey + 接口 | 按合同约定，如 100 QPS/key |
| **登录 / 短信 / 验证码** | userId / 手机号 / IP | 60s 内最多 5 次 |
| **秒杀活动** | 商品 ID + 用户 ID | 库存 × 3（防止过度排队） |
| **第三方付费 API 调用** | 全局 | 对方配额的 80% |
| **消费 MQ 限速** | 消费者实例 | 按下游 DB 容量倒推 |

### 1.3 阈值怎么定 —— 四步定阈值法

**❌ 错误做法**：拍脑袋定 1000 QPS

**✅ 正确做法**：

#### 第一步：压测找出单机容量

```
单机基准压测 → 找到 RT 突增拐点
例：QPS=500 时 RT=20ms，QPS=800 时 RT=200ms（拐点）
→ 单机安全水位 = 拐点 × 70% = 560 QPS
```

#### 第二步：根据集群规模算总容量

```
集群限流阈值 = 单机水位 × 实例数 × 折扣
            = 560 × 10 × 0.8 = 4480 QPS
```

折扣考虑：负载不均、网络抖动、Full GC 等。

#### 第三步：根据业务峰值反向校验

```
日常峰值 1500 QPS（监控数据）
大促峰值 3500 QPS（历史数据 × 增长系数）
→ 阈值应 ≥ 3500 且 ≤ 4480 → 取 4000 QPS
```

#### 第四步：留缓冲 + 逐步收紧

```
首版：4000 QPS（宽松）→ 观察一周 → 收紧到 3500 QPS
```

### 1.4 加分点：自适应限流

> 我们没有写死阈值，而是接入 **自适应限流**：基于系统实时 Load、CPU、RT 动态计算阈值（Sentinel BBR 算法 / 类 TCP 拥塞控制）。临时扩容时阈值自动跟随容量，不必手改配置。

### 1.5 防追问

> Q：阈值定低了误伤业务怎么办？
> A：监控大盘 + 限流告警 + 业务方申诉通道；配合 **Warm Up（预热限流）** 让流量缓慢爬升而不是直接拒绝。

## 二、令牌桶 vs 漏桶：区别与适用场景

### 2.1 一图看懂两种算法

```
              【漏桶 Leaky Bucket】

请求 ┐ ┐ ┐ ┐ ┐ ┐ ┐ ┐
     ↓ ↓ ↓ ↓ ↓ ↓ ↓ ↓
    ┌─────────────────┐
    │ ▓ ▓ ▓ ▓ ▓ ▓ ▓ ▓ │  ← 桶（队列）
    └────────┬────────┘
             │ 恒定速率漏水
             ▼
        匀速输出


              【令牌桶 Token Bucket】

   定速放令牌
        ↓
    ┌─────────┐
    │ ● ● ● ● │  ← 桶（容量固定）
    └────┬────┘
         │ 请求来时拿令牌
         ▼
    拿到令牌 → 放行 / 拿不到 → 拒绝
```

### 2.2 核心差异对比

| 维度 | 漏桶 Leaky Bucket | 令牌桶 Token Bucket |
|---|---|---|
| 核心思想 | 出口匀速 | 入口取令牌 |
| 输出速率 | **恒定**（强制平滑） | **可突发**（桶满时） |
| 应对突发流量 | ❌ 不允许 | ✅ 允许 |
| 处理方式 | 排队 / 拒绝 | 拿令牌 / 拒绝 |
| 关注点 | 流量**整形** Shaping | 流量**控制** Throttling |
| 代表场景 | 网络包整形、消费 MQ | API 限流、秒杀 |
| 代表实现 | Nginx `limit_req` | Guava RateLimiter、Sentinel |

### 2.3 关键差异：能否应对突发

**漏桶**：上游再猛，下游永远恒速。
```
入口：100 QPS 突发 → 出口：固定 50 QPS（多余排队/拒绝）
```

**令牌桶**：桶里攒了令牌可以瞬间放行。
```
速率：50 个/秒，桶容量：100
攒满后突发：可瞬间通过 100 个，再恢复 50 QPS
```

### 2.4 适用场景

#### 漏桶适合
1. **下游绝对受不了突发**：如调第三方 API（对方明确配额）、写入容量有限的 DB。
2. **流量整形 Traffic Shaping**：如 MQ 消费者下游打 ES，需要恒定速率。

#### 令牌桶适合
1. **允许短时突发**：用户 API 调用、电商抢购开始瞬间。
2. **大多数互联网场景**：真实流量天然有峰谷，令牌桶更高效。

### 2.5 经验值

> 桶容量 = 速率 × 1~3 秒（攒 1~3 秒的令牌为宜）

### 2.6 真题陷阱

> Q：Guava RateLimiter 是漏桶还是令牌桶？
> A：**令牌桶**，提供两种模式：
> - `SmoothBursty`：标准令牌桶，允许突发
> - `SmoothWarmingUp`：预热模式，从慢到快（冷启动保护）

> Q：Nginx `limit_req` 是哪种？
> A：**漏桶**算法，但通过 `burst` 参数允许排队。

## 三、单机限流 vs 分布式限流：实现与精度

### 3.1 根本差异

| 维度 | 单机限流 | 分布式限流 |
|---|---|---|
| 统计范围 | 进程内 | 整个集群 |
| 实现 | JVM 内存（AtomicLong/Semaphore） | Redis / 中心化存储 |
| 性能 | 极快（纳秒级） | 有网络开销（毫秒级） |
| 精度 | 单机精确 | 全局精确（有损耗） |
| 典型工具 | Guava RateLimiter、Sentinel 单机 | Redis+Lua、Sentinel 集群限流 |

### 3.2 单机限流：3 种主流方案

#### 方案 1：Guava RateLimiter（令牌桶）

```java
// 100 QPS
private static final RateLimiter limiter = RateLimiter.create(100);

public Result handle() {
    if (!limiter.tryAcquire(50, TimeUnit.MILLISECONDS)) {
        return Result.error("限流");
    }
    return businessLogic();
}
```

**核心源码逻辑**：
- 不是真有"令牌池"，而是用 **时间戳** 计算"下次允许通过的时间"
- 关键变量 `nextFreeTicketMicros`（下个令牌可用时间）
- 优点：无锁（CAS）+ 无线程

#### 方案 2：Semaphore（并发数限流）

```java
private final Semaphore semaphore = new Semaphore(100);

public Result handle() {
    if (!semaphore.tryAcquire()) return Result.error("限流");
    try {
        return businessLogic();
    } finally {
        semaphore.release();
    }
}
```

> 注意：Semaphore 限的是 **并发数**（同时执行数），不是 QPS。

#### 方案 3：Sentinel 单机限流（基于滑动窗口，见第四节）

### 3.3 分布式限流：3 种主流方案

#### 方案 1：Redis + Lua（最常用）

**为什么必须 Lua**：保证 `INCR + EXPIRE` 原子性。

**固定窗口计数限流**：
```lua
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])

local current = redis.call('INCR', key)
if current == 1 then
    redis.call('EXPIRE', key, window)
end

if current > limit then
    return 0  -- 限流
else
    return 1  -- 通过
end
```

**升级版：基于 ZSet 的滑动窗口**：
```lua
local key = KEYS[1]
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local limit = tonumber(ARGV[3])

redis.call('ZREMRANGEBYSCORE', key, 0, now - window)  -- 删除过期
local count = redis.call('ZCARD', key)

if count >= limit then
    return 0
else
    redis.call('ZADD', key, now, now)
    redis.call('EXPIRE', key, window / 1000 + 1)
    return 1
end
```

#### 方案 2：Sentinel 集群限流

```
应用 1 ─┐
应用 2 ─┼──► Token Server（中心化，持有全局令牌池）
应用 3 ─┘
```
- **嵌入式模式**：某应用兼任 Token Server
- **独立部署模式**：Token Server 单独部署

#### 方案 3：Nginx + Redis（接入层）

OpenResty + lua-resty-limit-traffic 模块，请求进入应用前完成限流。

### 3.4 精度问题（核心难点）

#### 难点 1：单机均分误差

```
集群 1000 QPS / 10 台 → 单机 100 QPS？
现实：负载不均（A 机 150，B 机 50）
→ A 被限流，B 还有富余 → 整体未达 1000 已限流
```
**解决**：用集群限流而非均分。

#### 难点 2：Redis 网络开销

每次查 Redis 增加 1~5ms RT，且 Redis 可能成为瓶颈。

**优化：本地预扣 + 批量同步**
```
每台机器本地缓存 10 个令牌
本地令牌耗尽 → 批量从 Redis 申请下一批
减少 Redis 调用 10 倍
```
代价：精度下降（最多多放行 10 个）。

#### 难点 3：时钟漂移

分布式机器时钟不同步导致窗口边界混乱。
**解决**：以 Redis 时间为准（`redis.call('TIME')`）。

#### 难点 4：固定窗口的临界突刺

```
[0~1s]：100 个请求集中在 0.99s
[1~2s]：100 个请求集中在 1.01s
→ 0.99~1.01s 这 20ms 内通过了 200 个！
```
**解决**：滑动窗口或令牌桶。

### 3.5 精度 vs 性能 工程权衡

| 方案 | 精度 | 性能 | 复杂度 | 推荐场景 |
|---|---|---|---|---|
| 单机均分 | ★★ | ★★★★★ | ★ | 流量均匀 |
| Redis 同步限流 | ★★★★★ | ★★ | ★★★ | 严格精度（付费 API）|
| 本地预扣 + 批量同步 | ★★★★ | ★★★★ | ★★★★ | 大流量场景 |
| Sentinel 集群限流 | ★★★★ | ★★★★ | ★★ | 大多数业务 |

## 四、Sentinel 滑动窗口实现原理

### 4.1 为什么要滑动窗口

固定窗口（1 秒 1 格）存在临界突刺；滑动窗口（1 秒分 N 格、随时间向前滑）能平滑统计：

```
[0─0.1)[0.1─0.2)...[0.9─1.0)  ← 细分子窗口
   ▲          ▲
   实时滑动，每次取最近 1 秒所有 bucket 求和
```

### 4.2 核心数据结构 LeapArray

```
LeapArray（环形数组）
┌────┬────┬────┬────┬────┐
│ B0 │ B1 │ B2 │ ...│ Bn │  ← n 个 Bucket
└────┴────┴────┴────┴────┘
        每个 Bucket 即 WindowWrap：
        ┌────────────────────┐
        │ windowStart：起始时间 │
        │ windowLength：长度   │
        │ value：MetricBucket  │ ← 真正的统计数据
        └────────────────────┘
                  │
                  ▼
            MetricBucket
            ┌──────────────┐
            │ pass:    100 │ ← 通过数
            │ block:    5  │ ← 限流数
            │ exception: 1 │ ← 异常数
            │ rt:    2000  │ ← 总耗时
            └──────────────┘
            (基于 LongAdder 高性能累加)
```

### 4.3 核心算法（伪代码）

```java
public WindowWrap<T> currentWindow(long timeMillis) {
    // 1. 当前时间属于哪个 Bucket
    int idx = calculateTimeIdx(timeMillis); // (time / windowLength) % arrayLength
    long windowStart = calculateWindowStart(timeMillis);

    while (true) {
        WindowWrap<T> old = array.get(idx);

        if (old == null) {
            // 2. 第一次访问，CAS 初始化
            WindowWrap<T> window = new WindowWrap<>(windowLength, windowStart, newEmpty());
            if (array.compareAndSet(idx, null, window)) {
                return window;
            }
        } else if (windowStart == old.windowStart()) {
            // 3. 命中当前窗口，直接返回
            return old;
        } else if (windowStart > old.windowStart()) {
            // 4. 旧窗口过期，重置（环形复用）
            if (updateLock.tryLock()) {
                try {
                    return resetWindowTo(old, windowStart);
                } finally {
                    updateLock.unlock();
                }
            }
        }
    }
}
```

### 4.4 三大设计精髓

#### 精髓 1：环形数组复用

```
默认 1 秒 2 个 bucket，长度 = 2
时间轴：[0─0.5)[0.5─1.0)[1.0─1.5)[1.5─2.0)…
索引：    0      1       0       1     ← 复用
```
省内存、省 GC、O(1) 访问。

#### 精髓 2：LongAdder 高性能累加

```java
class MetricBucket {
    private final LongAdder[] counters;  // 不是 AtomicLong！
}
```
**为什么不用 AtomicLong**：高并发下 CAS 失败率高、性能下降；LongAdder 分散热点最后求和。

#### 精髓 3：懒重置 + CAS

旧 Bucket 过期不删除，下次访问时**懒重置**（reset），避免 GC 抖动。

### 4.5 QPS 计算逻辑

```java
public double pass() {
    long pass = 0;
    for (WindowWrap<T> w : list()) {
        if (!isWindowDeprecated(w)) {  // 排除过期
            pass += w.value().pass();
        }
    }
    return pass / getIntervalInSecond();
}
```
关键：**实时遍历有效 Bucket 求和**——这就是"滑动"。

### 4.6 加分：两种 LeapArray

| 类型 | 用途 |
|---|---|
| `BucketLeapArray` | 普通统计（通过/限流/异常） |
| `OccupiableBucketLeapArray` | 支持"未来配额借用"（排队限流） |

## 五、限流、熔断、降级的区别

### 5.1 一句话定义

| 概念 | 一句话 | 触发对象 |
|---|---|---|
| **限流** | 主动**拒绝超量请求**，控制入口流量 | 流量来源（自己） |
| **熔断** | 检测下游故障，**快速失败**避免雪崩 | 下游依赖（故障） |
| **降级** | 牺牲非核心功能，**保证核心可用** | 自身业务（取舍） |

### 5.2 触发条件

```
限流：QPS > 阈值                  ── 流量维度
熔断：错误率 > 阈值 / RT 飙升       ── 故障维度
降级：人为决策 / 自动判断           ── 业务维度
```

### 5.3 流程对比

```
【限流】
请求 → [阈值检查] → 超阈值 ─Yes→ 拒绝
                       │
                       └─No→ 业务执行

【熔断】
请求 → [熔断器状态]
        ├─ Closed（正常） → 业务执行 → 统计错误率 →（超阈值切 Open）
        ├─ Open（熔断）   → 直接失败（不调下游）
        └─ Half-Open（试探）→ 放小流量 → 恢复 / 继续熔断

【降级】
请求 → [系统状态]
        ├─ 正常       → 完整逻辑（推荐 + 评论 + 库存）
        └─ 异常/高峰  → 降级逻辑（推荐返兜底）
```

### 5.4 状态机

#### 熔断器三态

```
       ┌──────────┐  错误率超阈值   ┌──────────┐
       │  Closed  │ ──────────────► │   Open   │
       │ (闭合)   │                 │  (断开)  │
       └──────────┘                 └────┬─────┘
            ▲                            │ 等待时间窗口
            │  探测成功                  ▼
            └────────────────────  ┌──────────┐
                                   │ Half-Open│
                                   │ (半开)   │
                                   └──────────┘
```

- 限流：每次请求独立判断，**无状态机**
- 降级：开关式（开/关），**无中间态**

### 5.5 三者协作关系

```
                  用户请求
                     │
                     ▼
       ┌───────────────────────────┐
       │ ① 限流：QPS > 1000？拒绝   │
       └─────────────┬─────────────┘
                     ▼
       ┌───────────────────────────┐
       │ ② 熔断：下游错误率高？快失败 │
       └─────────────┬─────────────┘
                     ▼
       ┌───────────────────────────┐
       │ ③ 降级：调兜底逻辑          │
       └───────────────────────────┘
```

- **限流**："流量入口管控"，挡在最前
- **熔断**："下游故障保护"，挡在调下游前
- **降级**："失败兜底处理"，限流/熔断后的善后

### 5.6 真实场景：电商首页

```
首页接口（聚合）调用：
├── 商品服务（核心）
├── 推荐服务（非核心）
└── 评论服务（非核心）

【限流】首页 QPS > 5000 → 直接返回"系统繁忙"
       └─ 保护：自身机器、整条调用链

【熔断】调推荐服务连续超时，错误率 80%
       └─ 触发熔断 → 接下来 30 秒不再调推荐
       └─ 保护：避免线程池被推荐拖垮

【降级】熔断后，推荐位置返回"运营兜底列表"（缓存）
       └─ 保护：用户体验不崩，首页仍可用
```

### 5.7 大对比表（一图流）

| 维度 | 限流 | 熔断 | 降级 |
|---|---|---|---|
| **目的** | 防过载 | 防雪崩 | 保核心 |
| **触发** | 流量 > 阈值 | 错误率/RT 异常 | 主动开关或自动 |
| **作用对象** | 自身入口 | 下游依赖 | 自身功能 |
| **决策依据** | 计数器 | 错误统计 | 业务策略 |
| **结果** | 拒绝/排队 | 快速失败 | 走备用方案 |
| **状态** | 无状态 | 三态机 | 开关 |
| **类比** | 高速收费站 | 跳闸保险丝 | 战时配给 |

### 5.8 类比记忆法

> **限流是医院挂号——号挂完了你别来了**
> **熔断是医生病了——别派病人来这个科室了**
> **降级是非核心科室关门——保证急诊抢救不停**

---

## 总结速查表

| 问题 | 一句话核心 |
|---|---|
| **场景与阈值** | OpenAPI/秒杀/第三方调用；压测 + 集群规模 + 业务峰值 + 缓冲 = 四步定阈值 |
| **令牌桶 vs 漏桶** | 令牌桶允许突发（API、秒杀）；漏桶恒速整形（保护下游、流量整形） |
| **单机 vs 分布式** | 单机用 Guava/Sentinel；分布式用 Redis+Lua / Sentinel 集群；精度靠原子操作 + 本地预扣权衡 |
| **Sentinel 滑动窗口** | LeapArray 环形数组 + LongAdder + 懒重置 |
| **限流/熔断/降级** | 防过载 / 防雪崩 / 保核心，三者配合形成稳定性体系 |
