## 一、固定窗口计数器（Fixed Window Counter）

### 1.1 核心思想

最简单的限流算法：**把时间轴切成固定长度的"窗口"，每个窗口内维护一个计数器，超过阈值就拒绝**。

```
时间轴：
|─────── 1s ───────|─────── 1s ───────|─────── 1s ───────|
  counter=0→78         counter=0→100       counter=0→45
     ✅ 全放行          ✅ 刚好满           ✅ 全放行
```

### 1.2 算法步骤

```
1. 计算当前时间属于哪个窗口：windowId = currentTime / windowSize
2. 如果 windowId 变了（进入新窗口）：counter 归零
3. counter++
4. 如果 counter > threshold：拒绝
5. 否则：放行
```

### 1.3 Java 实现

```java
public class FixedWindowRateLimiter {
    private final long windowSize;   // 窗口大小（ms），如 1000
    private final int threshold;     // 阈值，如 100
    private final AtomicInteger counter = new AtomicInteger(0);
    private volatile long windowStart;  // 当前窗口起始时间

    public FixedWindowRateLimiter(long windowSizeMs, int threshold) {
        this.windowSize = windowSizeMs;
        this.threshold = threshold;
        this.windowStart = System.currentTimeMillis();
    }

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();

        // 进入新窗口：重置计数器
        if (now - windowStart >= windowSize) {
            counter.set(0);
            windowStart = now;
        }

        // 判断是否超阈值
        if (counter.get() < threshold) {
            counter.incrementAndGet();
            return true;
        }
        return false;
    }
}
```

### 1.4 Redis 实现（分布式）

```lua
-- 固定窗口限流 Lua 脚本
local key = KEYS[1]           -- 限流 key，如 "rate:api:/order"
local limit = tonumber(ARGV[1])  -- 阈值
local window = tonumber(ARGV[2]) -- 窗口（秒）

local current = redis.call('INCR', key)
if current == 1 then
    redis.call('EXPIRE', key, window)  -- 首次设置过期时间
end

if current > limit then
    return 0  -- 限流
else
    return 1  -- 通过
end
```

**Redis key 设计技巧**：

```
key = "rate_limit:" + resource + ":" + (currentSecond / windowSize)

例：rate_limit:orderApi:1716000000
```

### 1.5 优缺点

| 优点 | 缺点 |
|---|---|
| 实现极简 | **临界突刺问题**（致命） |
| 内存消耗小 O(1) | 窗口切换瞬间计数归零 |
| 适合简单防刷 | 不适合精确限流保护 |

## 二、固定窗口的致命缺陷：临界突刺

### 2.1 问题描述

> 两个相邻窗口的**边界处**，可能在极短时间内通过 **2 倍阈值** 的请求。

### 2.2 图示推演

```
阈值：100 QPS

窗口 1 [0ms ~ 1000ms]           窗口 2 [1000ms ~ 2000ms]
│                    ←100个→│←100个→                    │
│                   990~1000│1000~1010                   │
│                      ↓     ↓                          │
│        ─────────────[突刺区]─────────────              │
                  20ms 内通过 200 个！

实际瞬时速率 = 200 / 0.02s = 10000 QPS（阈值的 100 倍）
```

### 2.3 具体例子

```
假设阈值 = 100 QPS（每秒最多 100 个）

T = 0~990ms：    0 个请求 → counter = 0
T = 990~1000ms：100 个请求涌入 → counter = 100 ✅（刚好满）
─── 窗口切换 ─── counter 归零 ───
T = 1000~1010ms：100 个请求涌入 → counter = 100 ✅（新窗口没超）

结果：990ms ~ 1010ms 这 20ms 内通过了 200 个请求
     系统实际承受 200 QPS 的瞬时压力
     如果系统只能扛 120 QPS → 直接被打挂
```

### 2.4 为什么这是"致命"问题？

| 后果 | 说明 |
|---|---|
| **保护失效** | 限流设的 100 QPS，实际允许了 200 QPS 通过 |
| **下游被打挂** | DB、第三方 API 在突刺期承受 2 倍压力 |
| **数据不一致** | 秒杀场景下多发库存、多扣款 |
| **限流形同虚设** | 恶意攻击者可以利用边界绕过限流 |

### 2.5 根本原因

> 固定窗口在窗口切换时"**遗忘**"了上个窗口末尾的请求——新窗口从 0 计数，不关心上个窗口的状态。

## 三、滑动窗口计数器（Sliding Window Counter）

### 3.1 核心思想

> **不再以固定边界划分时间，而是"任意时刻向前看固定时长"**——窗口随时间滑动。

```
固定窗口：
|──── 1s ────|──── 1s ────|  ← 窗口边界固定不动
                ↑ 边界处可能突刺

滑动窗口：
   |──── 1s ────|              ← 窗口随时间连续滑动
      |──── 1s ────|
         |──── 1s ────|
→ 任意时刻，统计的都是"过去 1 秒"的请求
```

### 3.2 为什么能解决临界突刺？

```
T = 1010ms 时，滑动窗口看的是 [10ms, 1010ms) 这 1 秒
→ 包含了 T=990~1000ms 的 100 个请求！
→ sum = 100 + 新请求 > 100 → 拒绝

固定窗口在 T=1010ms 看的是 [1000ms, 2000ms) 这个新窗口
→ 只看到新窗口的请求，"遗忘"了上个窗口末尾的 100 个
→ sum = 0 + 新请求 < 100 → 放行（错误！）
```

### 3.3 算法步骤

```
1. 当前时刻 = now
2. 统计 [now - windowSize, now) 区间内的请求总数
3. 如果 总数 ≥ threshold → 拒绝
4. 否则 → 记录本次请求，放行
```

## 四、滑动窗口的两种实现

### 4.1 实现 1：细粒度子窗口（Sentinel 方案）

**思路**：把大窗口切成 N 个小格子（子窗口），每个格子独立计数。统计时累加最近 N 个格子。

```
1 秒 = 10 个格子（每格 100ms）

时间轴：
[0~100)[100~200)[200~300)...[900~1000)[1000~1100)
  15      8       12    ...    90         ?

T=1001ms 时，向前看 1s = [1, 1001)
涵盖 bucket: [100,200) + [200,300) + ... + [900,1000) + [1000,1100)
sum = 15 + 8 + 12 + ... + 90 + (当前 bucket 计数)
```

#### Java 实现（简化版）

```java
public class SlidingWindowRateLimiter {
    private final int windowSize;     // 窗口大小（ms）
    private final int bucketCount;    // 格子数量
    private final int threshold;      // 阈值
    private final int bucketSize;     // 每个格子的时长
    private final AtomicInteger[] buckets;
    private final long[] bucketStart;

    public SlidingWindowRateLimiter(int windowSizeMs, int bucketCount, int threshold) {
        this.windowSize = windowSizeMs;
        this.bucketCount = bucketCount;
        this.threshold = threshold;
        this.bucketSize = windowSizeMs / bucketCount;
        this.buckets = new AtomicInteger[bucketCount];
        this.bucketStart = new long[bucketCount];
        for (int i = 0; i < bucketCount; i++) {
            buckets[i] = new AtomicInteger(0);
            bucketStart[i] = 0;
        }
    }

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        int currentIdx = (int)((now / bucketSize) % bucketCount);

        // 重置过期的格子
        resetExpiredBuckets(now);

        // 累加所有有效格子
        int sum = 0;
        for (int i = 0; i < bucketCount; i++) {
            if (now - bucketStart[i] < windowSize) {
                sum += buckets[i].get();
            }
        }

        if (sum >= threshold) {
            return false;
        }

        // 更新当前格子
        bucketStart[currentIdx] = (now / bucketSize) * bucketSize;
        buckets[currentIdx].incrementAndGet();
        return true;
    }

    private void resetExpiredBuckets(long now) {
        for (int i = 0; i < bucketCount; i++) {
            if (now - bucketStart[i] >= windowSize) {
                buckets[i].set(0);
                bucketStart[i] = (now / bucketSize) * bucketSize;
            }
        }
    }
}
```

**特点**：
- 时间复杂度：O(N)，N = 格子数
- 空间复杂度：O(N)
- 精度：窗口长度 / 格子数（格子越多越精确）

---

### 4.2 实现 2：滑动日志（Redis ZSet 方案）

**思路**：记录每个请求的精确时间戳，统计窗口内的请求数。
1.请求进来，将1s前的数据移除
2.计算1s内总的请求数
3.如果超过limit则限流
4.没超过则加入到请求，通过。
这几步是一个原子操作。

```lua
-- Redis ZSet 滑动窗口限流
local key = KEYS[1]
local now = tonumber(ARGV[1])        -- 当前时间戳(ms)
local window = tonumber(ARGV[2])     -- 窗口大小(ms) 1000
local limit = tonumber(ARGV[3])      -- 阈值 100
 
-- 1. 删除窗口外的过期数据
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
 删除 ZSet 中 score 在 [min, max] 之间的所有元素

-- 2. 统计窗口内请求数
local count = redis.call('ZCARD', key)

-- 3. 判断是否超限
if count >= limit then
    return 0  -- 限流
else
    -- 4. 记录本次请求（score=时间戳，member 需唯一）
    redis.call('ZADD', key, now, now .. ':' .. math.random(1000000))
    redis.call('EXPIRE', key, math.ceil(window / 1000) + 1)
    return 1  -- 通过
end
```

**特点**：
- 精度：每个请求级别，**完美精确**
- 缺点：内存 O(请求数)，高 QPS 时 ZSet 暴增（10000 QPS → 每秒存 1 万条）
- 适用：低频但精度要求高的场景（如安全防刷：5 分钟 3 次）

---

### 4.3 两种实现对比

| 维度 | 子窗口（Sentinel） | 滑动日志（ZSet） |
|---|---|---|
| 精度 | 桶级精度（有误差） | **完美精确** |
| 内存 | O(格子数)，极小 | O(请求数)，随 QPS 线性增长 |
| 性能 | 极快（累加几个 int） | 较慢（ZCARD + ZREMRANGEBYSCORE）|
| 适用场景 | 高 QPS 系统保护 | 低频安全防刷 |
| 代表实现 | Sentinel LeapArray | Redis ZSet |

## 五、Sentinel 滑动窗口源码剖析（LeapArray）

### 5.1 数据结构全景

```
LeapArray（环形数组）—— Sentinel 统计的核心
┌────┬────┬────┬────┬────┐
│ B0 │ B1 │ B2 │ ...│ Bn │  ← N 个 Bucket（环形复用）
└────┴────┴────┴────┴────┘
     │
     ▼ 每个 Bucket = WindowWrap<MetricBucket>
     ┌──────────────────────────────┐
     │ windowStart : long           │ ← 起始时间
     │ windowLength: long           │ ← 格子时长
     │ value       : MetricBucket   │ ← 统计数据
     └──────────────────────────────┘
                    │
                    ▼
              MetricBucket
              ┌──────────────┐
              │ pass   : LongAdder │ ← 通过数
              │ block  : LongAdder │ ← 限流数
              │ exception: LongAdder│ ← 异常数
              │ rt     : LongAdder │ ← 总耗时
              └──────────────┘
```

### 5.2 默认配置

```java
// Sentinel 默认：1 秒窗口，分 2 个 bucket
private int sampleCount = 2;      // 格子数
private int intervalInMs = 1000;  // 窗口大小
// → 每个格子 500ms
```

### 5.3 核心方法：currentWindow()

```java
public WindowWrap<T> currentWindow(long timeMillis) {
    // 1. 计算当前时间属于哪个格子
    long timeId = timeMillis / windowLengthInMs;
    int idx = (int)(timeId % array.length());

    long windowStart = timeId * windowLengthInMs;

    while (true) {
        WindowWrap<T> old = array.get(idx);

        if (old == null) {
            // 2. 首次访问：CAS 初始化
            WindowWrap<T> window = new WindowWrap<>(
                windowLengthInMs, windowStart, newEmptyBucket(timeMillis));
            if (array.compareAndSet(idx, null, window)) {
                return window;
            }
            Thread.yield();
        } else if (windowStart == old.windowStart()) {
            // 3. 命中当前窗口：直接返回
            return old;
        } else if (windowStart > old.windowStart()) {
            // 4. 旧窗口过期：懒重置（环形复用关键）
            if (updateLock.tryLock()) {
                try {
                    return resetWindowTo(old, windowStart);
                } finally {
                    updateLock.unlock();
                }
            } else {
                Thread.yield();
            }
        }
        // windowStart < old.windowStart() 时钟回拨，忽略
    }
}
```

### 5.4 三大设计精髓

#### 精髓 1：环形数组复用

```
数组长度 = sampleCount = 2
窗口 = 1000ms，每格 500ms

时间轴：[0~500) [500~1000) [1000~1500) [1500~2000) ...
索引：     0       1         0          1        ← 环形

旧 bucket 不删除，到期后"懒重置"复用
→ 零 GC、零内存分配
```

#### 精髓 2：LongAdder 替代 AtomicLong

```java
class MetricBucket {
    private final LongAdder[] counters;  // 用 LongAdder
}
```

**为什么不用 AtomicLong？**

```
AtomicLong：所有线程 CAS 同一个地址 → 高并发下失败率高
LongAdder：分散到多个 Cell → 最后求和 → 高并发下几乎无竞争

性能对比（16 线程）：
AtomicLong: ~200ns/op
LongAdder:   ~30ns/op（快 6 倍）
```

#### 精髓 3：懒重置（Lazy Reset）

```java
// 旧 bucket 过期 → 不删除，不 new → 直接重置字段
private WindowWrap<T> resetWindowTo(WindowWrap<T> w, long startTime) {
    w.resetTo(startTime);
    w.value().reset();  // 清零计数
    return w;
}
```

> 避免频繁 `new Object()`，减少 Young GC。

### 5.5 QPS 统计逻辑

```java
public long pass() {
    long pass = 0;
    // 遍历所有有效 bucket，累加
    List<MetricBucket> list = values(TimeUtil.currentTimeMillis());
    for (MetricBucket bucket : list) {
        pass += bucket.pass();
    }
    return pass;
}

// 判断 bucket 是否在窗口内
private boolean isWindowDeprecated(long time, WindowWrap<T> windowWrap) {
    return time - windowWrap.windowStart() > intervalInMs;
}
```

> 每次统计都是"当前时刻向前看 intervalInMs"——这就是"滑动"的含义。

## 六、滑动窗口的精度边界问题

### 6.1 桶内突刺（无法完全避免）

> 滑动窗口虽然解决了"窗口边界"突刺，但**无法解决"桶内"突刺**。

```
阈值 100 QPS，10 格（每格 100ms）

T=901ms：突发 99 个 → bucket [900,1000) count=99 ✅
T=1101ms：突发 99 个 → bucket [1100,1200) count=99
  此时 bucket [900,1000) 已过期（1101-900=201 > ? 取决于实现）
  如果 [900,1000) 刚好过期 → sum=99 ✅ → 放行
  
→ 200ms 内通过 198 个，瞬时速率远超阈值
```

### 6.2 精度公式

```
最坏情况误差 ≈ 阈值 × (1 / 格子数)

格子数 = 2（Sentinel 默认）→ 误差 50%
格子数 = 10              → 误差 10%
格子数 = 100             → 误差 1%
```

### 6.3 如何提高精度？

| 方法 | 代价 | 效果 |
|---|---|---|
| 增加格子数 | 内存/CPU 略增 | 误差减小 |
| 减小窗口 | 阈值需换算 | 响应更快 |
| 改用滑动日志（ZSet） | 内存 O(N) | 完美精确 |
| 叠加漏桶/令牌桶 | 复杂度增加 | 消除突刺 |

### 6.4 核心认知

> **滑动窗口是"精确计数"工具，不是"流量整形"工具。**
>
> 它回答的是"过去 1 秒有多少请求"，而不是"请求之间是否匀速"。
>
> 如果你要"匀速输出"→ 用漏桶。
> 如果你要"允许突发但限制总量"→ 用令牌桶。
> 如果你要"精确统计是否超限"→ 用滑动窗口。

## 七、固定窗口 vs 滑动窗口对比

### 7.1 全维度对比表

| 维度 | 固定窗口 | 滑动窗口（子窗口） | 滑动窗口（日志） |
|---|---|---|---|
| **精度** | ❌ 有 2 倍突刺 | ⚠️ 桶级精度 | ✅ 完美 |
| **实现** | ★ 极简 | ★★ 中等 | ★★★ 较复杂 |
| **内存** | O(1) | O(N) 格子数 | O(M) 请求数 |
| **性能** | 极快 | 快 | 一般 |
| **适用** | 粗略防刷 | 系统保护 | 精确安全控制 |
| **代表** | 简单计数器 | Sentinel | Redis ZSet |

### 7.2 选型决策

```
需要精确限流保护系统？
├── 是 → 流量高吗？
│       ├── 高（>1000 QPS）→ 子窗口方案（Sentinel）
│       └── 低（<100 QPS）→ 滑动日志（ZSet）
└── 否 → 只是粗略防刷？
        └── 是 → 固定窗口（最简单）
```

### 7.3 实际系统中的组合使用

真实系统往往**不只用一种算法**：

```
Sentinel 架构：
┌─────────────────────────────────┐
│ 统计层：滑动窗口（LeapArray）     │  ← 精确计数
├─────────────────────────────────┤
│ 判断层：阈值比较                  │  ← 是否超限
├─────────────────────────────────┤
│ 行为层：                         │
│   - 直接拒绝（默认）              │
│   - 匀速排队（漏桶）             │  ← 流量整形
│   - 预热（令牌桶）               │
└─────────────────────────────────┘
```

> 滑动窗口负责"**看清流量**"，漏桶/令牌桶负责"**控制流量**"——分工明确。

## 八、面试金句

### Q：固定窗口限流有什么问题？

> 固定窗口的致命问题是**临界突刺**：两个相邻窗口的边界处，可能在极短时间内通过 2 倍阈值的请求。根本原因是窗口切换时计数器归零，"遗忘"了上个窗口末尾的流量。

### Q：滑动窗口怎么解决这个问题的？

> 滑动窗口把"固定边界"变成了"任意时刻向前看固定时长"，这样任何两个相邻时间点的统计都是连续的，不存在"遗忘边界"。
>
> 工业实现上，Sentinel 用 **LeapArray 环形数组**：把 1 秒切成 N 个格子，每个格子用 LongAdder 独立计数；统计时遍历有效格子求和。环形复用 + 懒重置避免 GC 抖动。

### Q：滑动窗口能做到完全精确吗？

> 不能。子窗口实现有"桶内突刺"问题——同一个格子内的请求被视为同时到达，无法区分格子内部的时间分布。误差约为 `阈值/格子数`。
>
> 如果需要完美精确，可以用滑动日志（Redis ZSet 记录每个请求时间戳），但代价是 O(N) 内存。
>
> 如果需要彻底消除突刺，应该叠加漏桶（恒速输出）或令牌桶（控制放行速率），因为**滑动窗口是计数工具，不是整形工具**。

### Q：为什么 Sentinel 默认只分 2 个格子？

> 工程取舍：**高性能 > 绝对精确**。2 个格子意味着最坏 50% 误差，但内存极小、CPU 消耗极低。对于大多数业务限流场景，50% 的误差（如 100 QPS 阈值，最坏通过 150 个）不会击垮系统——限流本就是"兜底保护"不是精确计量。
