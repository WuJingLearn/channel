## 一、先建立正确认知：限流的本质

### 1.1 限流到底在保护什么？

很多人一上来就讲算法、讲框架，其实 **限流的本质是"资源保护"**。任何系统的资源都是有限的：

| 资源类型 | 上限来自哪里 |
|---|---|
| **CPU** | 物理核数 × 单核计算能力 |
| **内存** | 物理 RAM + JVM 堆 |
| **线程** | 操作系统线程数 + Tomcat 线程池 |
| **数据库连接** | 连接池大小（HikariCP 默认 10） |
| **网络带宽** | 网卡速率 + 出口带宽 |
| **下游配额** | 第三方 API 给你的 QPS 配额 |
| **金钱成本** | 短信 / OCR / GPU 调用按次计费 |

**限流的核心使命**：在请求量超过资源承载力之前，主动拒绝多余请求，保住系统不崩。

### 1.2 限流的"前提假设"

限流不是万能药，它的前提假设是：

> **"少量请求被拒绝，比所有请求都失败要好。"**

如果业务上不能接受任何请求被拒绝（如银行核心转账），那就不该限流，而该 **扩容**。

## 二、什么情况下"必须"使用限流

### 2.1 决策树：判断要不要限流

```
                  ┌─────────────────────┐
                  │ 我的接口需要限流吗？   │
                  └──────────┬──────────┘
                             ▼
                  ┌─────────────────────┐
                  │ Q1：流量是否可预测？   │
                  └──────────┬──────────┘
                  不可预测   ◄┴► 可预测
                     │           │
                     ▼           ▼
              【强烈需要】   ┌──────────────┐
                            │ Q2：被打挂代价大吗？│
                            └──────┬───────┘
                              大  ◄┴►  小
                              │       │
                              ▼       ▼
                         【需要】  ┌────────────────┐
                                  │ Q3：下游能扛 N 倍
                                  └──────┬─────────┘
                                     不行 ◄┴► 能扛
                                      │       │
                                      ▼       ▼
                                 【需要】  【可不加】
```

### 2.2 必须加限流的 6 大场景

#### 场景 1：对外开放的 OpenAPI

**典型特征**：调用方不可控、流量不可预测、多租户。

**为什么要限流**：
- 防止单个调用方挤占公共资源
- 实现 SLA 分级（VIP / 普通 / 试用）
- 实现商业化计费（按调用次数收费）

**真实例子**：
- 微信支付：普通商户 100 QPS、VIP 商户 1000 QPS
- 高德地图 API：免费版每天 5000 次
- OpenAI API：按 tier 分级

#### 场景 2：调用付费 / 配额受限的第三方

**典型特征**：调用方是你，被调方有明确配额。

**为什么要限流**：
- 超限会被对方封号（如银行接口、微信接口）
- 超限会被对方计费（如短信、OCR）
- 超限的请求是"无效请求"，浪费自身线程、连接

**实战经验值**：
> 阈值 = 对方配额 × **80%~90%**（留余量应对时钟漂移、网络重传）

#### 场景 3：安全敏感接口（防刷防爬）

**典型接口**：
- 登录 / 注册 / 找回密码 → 防撞库
- 短信验证码 → 防刷成本
- 评论 / 点赞 → 防机器人刷量
- 抽奖 / 优惠券领取 → 防羊毛党

**限流维度**（重点）：

| 维度 | 适用 | 示例阈值 |
|---|---|---|
| **IP** | 通用 | 60s / 100 次 |
| **userId** | 登录态 | 60s / 5 次 |
| **手机号** | 短信 | 60s / 1 次，24h / 10 次 |
| **设备指纹** | 防小号 | 24h / 3 次 |

#### 场景 4：高资源消耗接口

**典型接口**：
- 报表导出 / 复杂统计查询（占用 DB 资源）
- AI 推理 / 大模型调用（占用 GPU）
- 文件上传下载（占用带宽 + IO）
- 全文搜索 / 聚合查询（占用 ES 资源）

**特点**：QPS 不高也要限流。即使每秒 10 次调用，也可能把 DB 打满。

#### 场景 5：营销活动 / 秒杀

**典型特征**：流量瞬时百倍增长，平时风平浪静。

**限流的两层意义**：
- **保护系统**：库存 1000，进来 10 万人没意义，徒增压力
- **业务上的"过滤器"**：让请求按某种规则有序进入

#### 场景 6：异步消费场景（MQ 消费限速）

**典型场景**：
- Kafka / RocketMQ 消费者拉数据写 DB
- Flink / Spark 流处理写 ES

**限流目的**：**保护下游** 而非自己。
> 消费者抢着拉消息，速率不控制，下游 DB 直接被打挂。这种限流又叫"消费整形"或"反压（Back Pressure）"。

### 2.3 "不需要限流"的场景（避免过度设计）

| 场景 | 为什么不需要 |
|---|---|
| **内部管理后台** | 用户数有限、QPS 极低 |
| **ToB 低频系统** | 调用模式稳定可控 |
| **已有上游网关统一限流** | 重复限流是过度设计 |
| **离线批处理任务** | 不直接面向用户请求 |
| **资源远超峰值** | 比如 QPS 10、容量 10000 |

> **过度设计的代价**：维护成本 + 复杂度 + 误伤业务的风险。

## 三、阈值怎么定 —— 系统性方法论

> 阈值定不好，比不限流危害更大。
> 定低了：误伤正常用户、业务投诉。
> 定高了：起不到保护作用、形同虚设。

### 3.1 面试回答模板：接口级阈值为什么不能直接用单接口压测水位？

> 接口级限流不能简单使用单接口压测出来的最大安全水位。因为单接口压测时，这个接口基本独占应用资源；但线上是多个接口同时运行，它们共享 Tomcat 线程、业务线程池、CPU、DB 连接池、Redis 连接池和下游 RPC 资源。如果每个接口都按单接口最大水位配置限流，多个接口同时打满时，总资源消耗会超过应用整体承载能力。
>
> 所以我理解限流应该是分层的。第一层是应用级总限流，控制应用整体入口流量；第二层是资源池级限流，比如 DB、Redis、下游 RPC、业务线程池；第三层才是接口级限流。接口级阈值应该取 `单接口安全水位`、`全局容量分配额度`、`下游资源容量分配额度` 三者中的最小值。
>
> 另外，只看 QPS 也不够，还要看接口资源成本。比如一个接口平均 RT 是 50ms，另一个接口平均 RT 是 200ms，同样是 1000 QPS，后者占用线程资源是前者的 4 倍。所以更合理的模型是 `Σ(QPS_i × RT_i) <= 系统可用并发容量`。对于核心接口，可以通过更高权重或独立线程池保护；对于低优先级接口，可以更早限流或降级。

一句话总结：

> **接口级限流只能防止单接口打爆系统，不能保证多接口组合流量下系统安全；真正可靠的限流应该是“应用总量控制 + 资源池保护 + 接口配额 + 优先级隔离”。**

### 3.2 ❌ 三种典型错误做法

| 错误做法 | 后果 |
|---|---|
| **拍脑袋定 1000 QPS** | 没有依据，不知道高低 |
| **抄竞品配置** | 系统容量不同，不可比 |
| **按业务期望定** | 系统扛不住，照样崩 |

### 3.3 ✅ 正确做法：四步定阈值法

```
┌─────────────────────┐
│ Step 1: 压测找容量   │  ← 知道系统能扛多少
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Step 2: 算集群总量   │  ← 单机 → 集群
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Step 3: 业务峰值校验  │  ← 既能扛业务，又能保护
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Step 4: 留缓冲+收紧   │  ← 灰度迭代
└─────────────────────┘
```

---

### 3.4 Step 1：压测找出单机容量（最关键）

#### 压测方法：阶梯递增

```
工具：JMeter / wrk / Locust

线程数从 50 → 100 → 200 → 500 → 1000 阶梯递增
观察指标：QPS / RT / CPU / Load / GC
```

#### 找"拐点"：RT 突增的位置

```
QPS    RT(ms)   CPU    Load   状态
100     5      20%    0.5    ✅ 健康
300    10      40%    1.2    ✅ 健康
500    20      60%    2.0    ✅ 接近水位
800    50      80%    3.5    ⚠️  开始劣化
1000  200      95%    6.0    ❌ 拐点！（RT 暴涨）
1200  超时     99%    8.0    ❌ 雪崩
```

**安全水位 = 拐点 QPS × 70%**
> 例：拐点 1000 → 安全水位 700 QPS / 单机

#### 为什么打 70% 折扣？

| 折扣原因 | 占比 |
|---|---|
| Full GC 抖动 | 10% |
| 网络波动 | 5% |
| 业务突发 | 10% |
| 监控误差 | 5% |

---

### 3.5 Step 2：根据集群规模算总容量

```
集群限流阈值 = 单机水位 × 实例数 × 集群折扣

例：单机 700 QPS × 10 实例 × 0.8 = 5600 QPS
```

**集群折扣 0.8 的来源**：
- 负载不均（LB 算法不完美）
- 实例间状态不一致
- 部分实例可能处于发布、Full GC 中

---

### 3.6 Step 3：业务峰值反向校验

**关键问题**：系统能扛 5600 QPS，业务实际需要多少？

```
监控数据采集（至少 1 个月）：
- 日常峰值     : 1500 QPS
- 周高峰        : 2000 QPS
- 大促/活动峰值 : 3500 QPS
- 未来 6 个月增长系数 : 1.5x

预估上限 = 3500 × 1.5 = 5250 QPS
```

**对比决策**：

```
系统容量：5600 QPS
业务上限：5250 QPS
        ↓
取较小值并留余量 → 阈值定 5000 QPS
```

#### 决策矩阵

| 情况 | 阈值取值 | 说明 |
|---|---|---|
| 系统容量 >> 业务上限 | 业务上限 × 1.2 | 防恶意攻击，超过即异常 |
| 系统容量 ≈ 业务上限 | 系统容量 × 0.9 | 主要为保护系统 |
| 系统容量 << 业务上限 | 先扩容再限流 | 限流是绝望中的选择 |

---

### 3.7 Step 4：留缓冲 + 灰度收紧

**绝对不要一上来就设最严阈值**，否则会误伤业务。

```
上线策略：
Week 1: 设宽松阈值 6000 QPS（仅做兜底）
Week 2-3: 观察真实流量分布
Week 4: 收紧到 5000 QPS
Month 2+: 根据监控数据动态调整
```

**配套基础设施**（建议）：

- ✅ 限流监控大盘（实时 QPS / 限流次数）
- ✅ 限流告警（触发限流即通知）
- ✅ 业务方申诉通道（避免误伤）
- ✅ 配置中心动态调整（Nacos / Apollo，不重启）

## 四、不同维度的阈值定法

不同限流维度，阈值思路完全不同：

### 4.1 全局阈值（保护系统）

按 Step 1-4 严格压测计算。

### 4.2 单用户阈值（防刷）

按业务"合理使用"反推：

```
登录接口   ：用户 1 分钟最多登录几次？ 5 次足够 → 60s/5
短信验证码 ：用户 1 分钟最多收几条？  1 次     → 60s/1，24h/10
搜索接口   ：用户 1 分钟最多搜几次？  30 次足够 → 60s/30
```

### 4.3 单 IP 阈值（防爬虫）

```
公司出口 IP（NAT）共享：放宽（如 1000/分钟）
家庭 IP             ：收紧（如 100/分钟）
机房 IP / 代理 IP    ：极严（如 10/分钟，或直接拉黑）
```

### 4.4 单 AppKey 阈值（OpenAPI）

按 **商业合同** 而非系统容量：

```
免费版：100 QPS
专业版：1000 QPS
企业版：10000 QPS（实际由系统容量 / 客户数倒推）
```

## 五、进阶：自适应阈值（动态限流）

写死的阈值无法应对"扩容、缩容、机器型号差异"，进阶方案是 **自适应限流**：

### 5.1 基于系统指标动态调整

```
指标：CPU Load + 平均 RT + 当前并发数

if (Load > 8.0 || RT > 200ms) {
    threshold = threshold × 0.7;  // 降阈值
} else if (Load < 3.0 && RT < 50ms) {
    threshold = threshold × 1.1;  // 升阈值
}
```

### 5.2 Sentinel 的 BBR 算法（参考 TCP）

核心公式：

```
最大并发数 = 系统最大处理能力 / 平均 RT
          = (CPU 核数 × 单核 QPS) × 平均 RT
```

实时动态计算，无需人工配置。

### 5.3 适用场景

| 方案 | 适用 |
|---|---|
| 固定阈值 | 稳定业务、可控环境 |
| 自适应阈值 | 弹性伸缩、容器化部署、流量波动大 |

## 六、面试金句 & 经验总结

### 6.1 一句话回答框架

> "我们在 XX 场景使用了限流。阈值不是拍脑袋定的，而是通过 **压测得到单机水位 → 集群折算 → 业务峰值反向校验 → 灰度收紧** 这四步确定的。同时配套监控告警和动态配置，保证阈值能随系统容量变化而调整。"

### 6.2 加分点话术

- "我们留了 20% 缓冲应对突发"
- "限流维度按 appKey + 接口聚合"
- "阈值通过 Nacos 动态推送，无需重启"
- "配合预热模式（Warm Up）避免冷启动时被限流"
- "对核心接口用自适应限流而非固定阈值"

### 6.3 避坑要点

| 坑 | 避坑 |
|---|---|
| 阈值定低误伤业务 | 灰度上线 + 监控告警 + 申诉通道 |
| 阈值定高形同虚设 | 配合压测数据校验 |
| 集群均分误差 | 用集群限流而非单机均分 |
| 配置写死无法调整 | 配置中心动态推送 |
| 限流没监控不知是否生效 | 限流指标必须可观测 |

## 七、给你的实战建议

如果你的项目目前还没有限流：

### 1. 从最危险的接口开始

- 第三方付费 API 调用（防钱包爆炸）
- 登录 / 短信接口（防刷）

### 2. 先用最简单的方式

- 单机：Guava RateLimiter（一行代码）
- 多机：Sentinel + Nacos（接入成本低）

### 3. 先松后紧

- 阈值定宽松（系统容量的 80%）
- 观察一周后再收紧

### 4. 必须有监控

- 没监控的限流 = 黑盒
- 至少做到：限流次数 + 触发告警
