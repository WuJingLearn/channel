## 一、为什么需要"流量整形"算法

### 1.1 计数窗口的局限

固定窗口、滑动窗口本质上都是**计数器**——回答"过去 X 秒有多少请求"，但**不关心请求之间的分布是否均匀**。

```
阈值 100 QPS，滑动窗口看到的合法情况：

情况 A：均匀流入   ▏▏▏▏▏▏▏▏▏▏  100 个 → ✅
情况 B：瞬时打满   ████░░░░░░    前 50ms 涌入 100 个 → ✅（同样合法）
                              下游接收到 2000 QPS 的瞬时压力！
```

> 计数窗口 = "看清流量"；流量整形 = "**控制流量节奏**"。

### 1.2 流量整形要解决的两个问题

| 问题 | 整形目标 | 适合算法 |
|---|---|---|
| 下游脆弱，要求**恒速** | 削峰、把毛刺磨平 | 漏桶 |
| 业务允许突发，但要**控总量** | 容忍短时突发，长期匀速 | 令牌桶 |

## 二、漏桶算法（Leaky Bucket）

### 2.1 物理模型

把请求想成水，系统是一个**底部带小孔的桶**。

```
   请求 ↓ ↓ ↓（任意速率流入）
   ┌─────────┐
   │   ░░░   │ ← 桶容量 capacity（排队缓冲）
   │  ░░░░░  │
   └────┬────┘
        ↓ 恒定速率漏出（QPS）
     下游服务
```

- 请求（水）从上方注入：速率任意
- 桶以**恒定速率**漏水：交给下游处理
- 桶满后再来的请求 → **直接拒绝（溢出）**

### 2.2 两个核心参数

| 参数 | 含义 | 决定什么 |
|---|---|---|
| **漏出速率 rate（QPS）** | 恒定流出速度 | 下游持续吞吐上限 |
| **桶容量 capacity** | 最大排队数 / 最大等待时间 | 突发缓冲能力 |

> ⚠️ 两个参数缺一不可。只配速率没桶容量 = 任何超速都直接拒绝（退化为严格限速）。

### 2.3 算法步骤

```
1. 当前时刻 = now
2. 按时间差计算应该漏掉的水量：leaked = (now - lastLeak) × rate
3. water = max(0, water - leaked)
4. lastLeak = now
5. 如果 water < capacity：water++，放行
6. 否则：拒绝
```

### 2.4 Java 实现

```java
public class LeakyBucketRateLimiter {
    private final long capacity;   // 桶容量
    private final double rate;     // 漏出速率（个/ms）
    private double water;          // 当前水量
    private long lastLeakTime;     // 上次漏水时间

    public LeakyBucketRateLimiter(long capacity, double ratePerSecond) {
        this.capacity = capacity;
        this.rate = ratePerSecond / 1000.0;
        this.lastLeakTime = System.currentTimeMillis();
    }

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        // 1. 漏水
        double leaked = (now - lastLeakTime) * rate;
        water = Math.max(0, water - leaked);
        lastLeakTime = now;
        // 2. 加水
        if (water < capacity) {
            water++;
            return true;
        }
        return false;
    }
}
```

### 2.5 漏桶的两种典型变体

| 变体 | 行为 | 代表 |
|---|---|---|
| **拒绝式漏桶** | 桶满即拒绝 | 网络流量整形 |
| **排队式漏桶** | 桶满才拒绝，未满则按恒速等待出桶 | Sentinel 匀速排队 |

排队式漏桶在限流场景更常见——它把"瞬时高峰"摊平到时间轴上：

```
入口（突发）：████████░░░░░░░░░░░░░░░░░
                ↓ 漏桶
出口（恒速）：▏▏▏▏▏▏▏▏▏▏▏▏▏▏▏▏▏▏▏▏▏▏
```

### 2.6 优缺点

| 优点 | 缺点 |
|---|---|
| 输出绝对平滑，下游永不超载 | 不能利用空闲期（即使下游闲着，仍按恒速放行） |
| 削峰填谷，能扛短时尖刺 | 排队会引入延迟 |
| 实现简单 | 不适合允许突发的业务 |

## 三、令牌桶算法（Token Bucket）

### 3.1 物理模型

桶里装的不是水，而是**令牌**：

```
  ↓ 恒定速率发放令牌（rate）
  ┌─────────┐
  │ T T T T │ ← 桶容量 capacity（最大突发量）
  │ T T T   │
  └────┬────┘
       ↓ 请求消耗令牌
   通过 / 拒绝
```

- 系统以**恒定速率**向桶中投放令牌
- 桶满则丢弃多余令牌
- 请求到来时**先取令牌**，取到通过，取不到则拒绝（或等待）

### 3.2 两个核心参数

| 参数 | 含义 | 决定什么 |
|---|---|---|
| **令牌生成速率 rate** | 每秒新增令牌数 | 平均吞吐 QPS |
| **桶容量 capacity** | 桶能存放的令牌上限 | **最大突发量** |

### 3.3 行为特征：允许"借用空闲期"

```
T=0~5s：    系统空闲，0 请求 → 桶累积到 capacity 个令牌
T=5s+1ms：  突发 capacity 个请求一起到达
            → 全部命中令牌 → 全部放行
            → 瞬时 QPS = capacity（远高于平均 rate）

之后：     桶被掏空，必须等 rate 速率慢慢恢复
```

> 这就是令牌桶的精髓：**长期匀速 + 短期允许突发**。突发上限 = capacity。

### 3.4 算法步骤

```
1. 当前时刻 = now
2. 按时间差补充令牌：tokens = min(capacity, tokens + (now - lastRefill) × rate)
3. lastRefill = now
4. 如果 tokens >= 1：tokens -= 1，放行
5. 否则：拒绝（或阻塞等待）
```

### 3.5 Java 实现

```java
public class TokenBucketRateLimiter {
    private final long capacity;        // 最大令牌数（突发上限）
    private final double rate;          // 每毫秒生成令牌数
    private double tokens;              // 当前令牌数
    private long lastRefillTime;

    public TokenBucketRateLimiter(long capacity, double ratePerSecond) {
        this.capacity = capacity;
        this.rate = ratePerSecond / 1000.0;
        this.tokens = capacity;         // 初始装满
        this.lastRefillTime = System.currentTimeMillis();
    }

    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        // 1. 补充令牌
        tokens = Math.min(capacity, tokens + (now - lastRefillTime) * rate);
        lastRefillTime = now;
        // 2. 消耗令牌
        if (tokens >= 1) {
            tokens -= 1;
            return true;
        }
        return false;
    }
}
```

### 3.6 优缺点

| 优点 | 缺点 |
|---|---|
| 允许突发，吞吐利用率高 | 不能保证恒速输出 |
| 业务体验好（空闲期累积令牌） | 突发期下游可能瞬时压力大 |
| 是 Guava RateLimiter 默认算法 | 需要谨慎设置 capacity 防击穿 |

## 四、漏桶 vs 令牌桶 全维度对比

### 4.1 对比表

| 维度 | 漏桶 | 令牌桶 |
|---|---|---|
| **桶里装的** | 请求（水） | 令牌 |
| **流入** | 请求按真实速率注入 | 令牌按固定速率注入 |
| **流出** | 恒定速率（QPS） | 由请求消耗驱动 |
| **突发流量** | 削峰排队 / 溢出拒绝 | **直接放过**（≤桶容量） |
| **空闲期** | 不累积"额度" | 累积令牌，为后续突发做准备 |
| **输出形态** | 严格匀速 | 平均匀速、瞬时可抖动 |
| **关注点** | 保护下游、抹平毛刺 | 提高吞吐、容忍业务抖动 |
| **代表实现** | Sentinel 匀速排队 | Guava RateLimiter / Sentinel WarmUp |

### 4.2 同样的流量，两种算法的输出对比

```
入口：[5个] [0] [0] [0] [10个突发] [0] [0]   （QPS 配置 = 5）

漏桶（capacity=10, rate=5/s）：
  → 第 1 秒放行 5 个
  → 第 2 秒放行剩 5 个排队 + 5 个新（其余溢出）
  → 输出：▏▏▏▏▏ ▏▏▏▏▏ ▏▏▏▏▏  恒速 5/s

令牌桶（capacity=10, rate=5/s）：
  → 前 2 秒桶满（10 个令牌）
  → 突发 10 个 → 瞬间放行（瞬时 QPS≈∞）
  → 之后慢慢恢复 5/s
  → 输出：░░░░ ██████████░░ ▏▏▏▏▏  允许突发 + 长期匀速
```

### 4.3 等价性视角

> 当 **漏桶 capacity = 0** 时，漏桶退化为严格限速器；
> 当 **令牌桶 capacity = 1** 时，令牌桶逼近恒速漏桶（无突发）。
>
> 两者本质都是"速率 + 容量"二维参数化的限流器，差别在于**容量在桶里装什么**。

## 五、工业实现：Sentinel 匀速排队（漏桶）

### 5.1 配置方式

```java
FlowRule rule = new FlowRule();
rule.setResource("createOrder");
rule.setGrade(RuleConstant.FLOW_GRADE_QPS);
rule.setCount(5);                                              // QPS = 5（漏出速率）
rule.setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_RATE_LIMITER);
rule.setMaxQueueingTimeMs(500);                                // 最大排队时间（漏桶容量）
```

| 字段 | 漏桶语义 |
|---|---|
| `count` | 漏出速率（QPS） |
| `maxQueueingTimeMs` | 桶容量（按时间换算的最大排队） |

### 5.2 核心源码：RateLimiterController

```java
public boolean canPass(Node node, int acquireCount, boolean prioritized) {
    long currentTime = TimeUtil.currentTimeMillis();
    // 1. 计算两次请求的最小间隔（恒速漏出）
    long costTime = Math.round(1.0 * (acquireCount) / count * 1000);
    // 2. 计算"我"应该在什么时候被处理
    long expectedTime = costTime + latestPassedTime.get();

    if (expectedTime <= currentTime) {
        // 不需要排队，直接通过
        latestPassedTime.set(currentTime);
        return true;
    } else {
        // 需要排队，检查排队时间是否超限（桶是否溢出）
        long waitTime = costTime + latestPassedTime.get() - currentTime;
        if (waitTime > maxQueueingTimeMs) {
            return false;   // 桶满，溢出拒绝
        }
        // CAS 抢占下一个时间槽，sleep 至轮到自己
        long oldTime = latestPassedTime.addAndGet(costTime);
        try {
            waitTime = oldTime - TimeUtil.currentTimeMillis();
            if (waitTime > maxQueueingTimeMs) {
                latestPassedTime.addAndGet(-costTime);
                return false;
            }
            if (waitTime > 0) Thread.sleep(waitTime);
            return true;
        } catch (InterruptedException e) { ... }
    }
}
```

### 5.3 设计精髓：虚拟队列

> Sentinel 没有真的把请求放进队列，而是用 `latestPassedTime` 记录"上一个请求被处理的虚拟时刻"——每个新请求按自己应该被处理的"虚拟时刻"决定**通过 / 等待 / 拒绝**。

```
QPS=5 → 每个请求间隔 200ms

T=0ms：req1 通过，latestPassedTime = 0
T=10ms：req2 来 → expectedTime = 0 + 200 = 200
        waitTime = 190 < maxQueueingTimeMs(500) → 排队 190ms 后通过
T=20ms：req3 来 → expectedTime = 200 + 200 = 400
        waitTime = 380 < 500 → 排队
...
T=20ms：req10 来 → waitTime = 1580 > 500 → ❌ 拒绝
```

## 六、工业实现：Guava RateLimiter（SmoothBursty）

### 6.1 使用方式

```java
RateLimiter limiter = RateLimiter.create(5.0);   // 5 QPS
if (limiter.tryAcquire()) {
    // 通过
} else {
    // 限流
}
```

> Guava 默认 `SmoothBursty`，桶容量 = `1 秒可累积令牌数 = rate × 1`，即支持最多 **1 秒空闲期** 的突发。

### 6.2 设计巧思：用"下一次允许时间"代替令牌计数

Guava 不维护 `tokens` 字段，而是维护 `nextFreeTicketMicros`：

```
请求到来：
1. 如果 nextFreeTicketMicros <= now：立即放行
2. 把 nextFreeTicketMicros 推进 stableInterval 微秒（= 1/rate）
3. 否则：sleep 至 nextFreeTicketMicros
```

桶容量通过"允许 nextFreeTicketMicros 落后于 now 多少"等价表达：

```
落后越多 = 累积令牌越多 = 后续突发能力越强
```

### 6.3 SmoothWarmingUp（冷启动模式）

```java
RateLimiter limiter = RateLimiter.create(100, 3, TimeUnit.SECONDS);
// 稳定 100 QPS，冷启动期间从慢速逐步预热到 100 QPS（耗时 3s）
```

适用场景：**应用刚启动 / 长期空闲后**，下游缓存未热、连接池未建立，不能直接打满 → **用预热曲线渐进放量**。

> Sentinel 的 `WarmUp` 模式（`CONTROL_BEHAVIOR_WARM_UP`）实现思路相同。

## 七、参数配置原则与避坑

### 7.1 漏桶配置

```
rate ≤ 下游持续承受 QPS × 80%
maxQueueingTimeMs：根据业务可接受延迟设定
  - 同步接口：≤ 用户超时容忍（如 500ms）
  - 异步任务：可较大（如 5s）
```

### 7.2 令牌桶配置（关键）

> ⚠️ **桶容量 capacity 必须确保下游能扛住瞬时 capacity 个请求的冲击**，否则突发期会击穿系统。

```
capacity 推荐范围：
  保守：rate × 0.1 ~ rate × 1   （允许 0.1~1s 空闲期突发）
  激进：rate × 5+                （需配合下游冗余 / 池化预热）

危险配置：capacity 远大于下游瞬时承载力
  → 业务沉睡 1 小时 → 累积海量令牌 → 醒来一拥而上 → 下游瞬间被打挂
```

### 7.3 三个常见错误

| 错误 | 后果 | 正确做法 |
|---|---|---|
| 只配 rate 不配 capacity | 漏桶退化为严格限速 / 令牌桶突发不可控 | 两参数必须同时配置 |
| 用令牌桶保护脆弱下游 | 突发击穿 | 改用漏桶（匀速排队） |
| 用漏桶限制对外 OpenAPI | 用户体验差（延迟） | 改用令牌桶（直接拒绝） |

## 八、与计数窗口的协作关系

> 三种算法不是互斥的，**真实系统通常组合使用**。

```
┌──────────────────────────────────────┐
│ 统计层：滑动窗口（看清流量）           │
│   → 精确计数过去 X 秒的 QPS          │
├──────────────────────────────────────┤
│ 决策层：阈值比较                      │
│   → 是否触发限流                      │
├──────────────────────────────────────┤
│ 行为层（流量整形，二选一或叠加）        │
│   - 直接拒绝（默认）                  │
│   - 漏桶：匀速排队                    │
│   - 令牌桶：允许突发 / 冷启动预热       │
└──────────────────────────────────────┘
```

| 算法 | 角色 | 一句话 |
|---|---|---|
| 滑动窗口 | 统计 / 决策 | 看清"过去发生了什么" |
| 漏桶 | 整形（恒速） | 控制"输出节奏" |
| 令牌桶 | 整形（突发） | 控制"放行预算" |

## 九、面试金句

### Q1：漏桶和令牌桶的核心区别是什么？

> 表面看，两者都是"速率 + 容量"二参数限流器。本质区别在于**输出形态**：
> - **漏桶**：恒定速率输出，**绝对平滑**——桶里装请求，请求按固定速率漏出。
> - **令牌桶**：平均速率受限但**允许瞬时突发**——桶里装令牌，空闲期累积令牌，突发期一次性消耗。
>
> 一句话：漏桶**整形**流量，令牌桶**配额**流量。

### Q2：为什么两个算法都需要两个参数？

> 单一速率参数无法描述"持续吞吐 + 突发缓冲"两个独立维度。
> - 漏桶：`rate` 控制持续吞吐，`capacity` 控制突发缓冲（排队数 / 排队时长）。
> - 令牌桶：`rate` 控制平均 QPS，`capacity` 控制最大突发量。
>
> 缺任一参数，算法行为不完整：漏桶失去缓冲能力 → 退化为严格限速；令牌桶失去突发上限 → 累积令牌可能击穿下游。

### Q3：什么场景用漏桶，什么场景用令牌桶？

> **下游脆弱、要求平滑** → 漏桶。例如：调用付费第三方 API、老旧服务、数据库批写。Sentinel 匀速排队就是漏桶。
>
> **业务允许突发、希望提升吞吐利用率** → 令牌桶。例如：API 网关日常流量、秒杀首屏、应用冷启动预热。Guava RateLimiter、Sentinel WarmUp 都是令牌桶。

### Q4：Sentinel 匀速排队是怎么实现漏桶的？

> Sentinel 没有真的维护一个队列，而是用 `latestPassedTime` 记录"上个请求被处理的虚拟时刻"，每个新请求按 `1/QPS` 计算自己的"应该处理时刻"：
> - 如果应该处理时刻 ≤ 当前时刻 → 直接通过；
> - 否则计算等待时间，如果超过 `maxQueueingTimeMs` → 拒绝（桶溢出）；
> - 否则 CAS 抢占时间槽，sleep 至轮到自己。
>
> 这是"虚拟队列"思想：**用时间戳做排队，O(1) 内存、无队列对象**。

### Q5：令牌桶的桶容量该怎么定？

> 桶容量决定**最大突发量**，必须满足：`capacity ≤ 下游瞬时承受能力`。
>
> 常见经验值：`capacity = rate × (0.1 ~ 1)`，即允许 0.1 ~ 1 秒空闲期累积突发。
>
> **危险配置**：`capacity` 远大于下游承载力 + 系统长期空闲——一旦突发到来，累积的海量令牌会瞬间击穿下游。Guava 默认只允许 1 秒突发就是这个原因。
