# Day 17-18：微服务架构与高可用

> 复习目标：掌握微服务拆分原则、注册中心、网关、限流熔断、配置中心和链路追踪。

---

## 一、微服务基础

**Q1：微服务拆分的原则有哪些？**

A：
1. **单一职责**：每个服务只做一件事（用户服务、订单服务、支付服务）
2. **领域驱动（DDD）**：按业务域（Bounded Context）拆分，而非按技术层拆
3. **高内聚低耦合**：服务内部高度相关，服务之间通过 API 交互，避免共享数据库
4. **演进式拆分**：先从单体中识别边界清晰的模块逐步拆出，不要一开始就过度拆分
5. **数据独立**：每个服务有自己的数据库（Database per Service）

**拆分粒度判断**：
- 太粗 → 修改一处需要整体部署，失去微服务优势
- 太细 → 分布式事务增多，运维复杂度爆炸
- 建议 3-5 人维护一个服务

---

**Q2：微服务间通信方式有哪些？**

A：
| 方式 | 协议 | 特点 | 场景 |
|------|------|------|------|
| **REST（HTTP/JSON）** | HTTP | 简单通用、跨语言 | 对外 API、内部简单调用 |
| **gRPC** | HTTP/2 + Protobuf | 高性能、强类型、双向流 | 内部高频调用 |
| **消息队列** | AMQP/自定义 | 异步解耦 | 事件驱动、最终一致性场景 |
| **Dubbo** | 自定义 TCP | 高性能 RPC、服务治理完善 | Java 体系内部调用 |

## 二、注册中心

**Q3：主流注册中心对比？**

A：
| 维度 | Nacos | Eureka | Consul | ZooKeeper |
|------|-------|--------|--------|-----------|
| CAP | **AP + CP 可切换** | AP | CP | CP |
| 一致性协议 | Raft（CP 模式） | 无（AP，对等复制） | Raft | ZAB |
| 健康检查 | TCP/HTTP/MySQL + 客户端心跳 | 客户端心跳 | TCP/HTTP/gRPC/脚本 | 会话心跳 |
| 配置管理 | 内置 | 无 | 内置 KV 存储 | 可实现（复杂） |
| 雪崩保护 | 支持（阈值保护） | 支持（自我保护） | 不支持 | 不支持 |
| 推荐场景 | Spring Cloud Alibaba | Spring Cloud Netflix | 多语言、Service Mesh | Dubbo 默认注册中心 |

**Nacos 的优势**：同时支持 AP 和 CP 模式，内置配置中心，社区活跃，阿里生产验证。

---

**Q3.5：注册中心场景为什么通常选 AP？**

A：注册中心的本质是维护一张"服务地址列表"，**首要职责是让调用方能找到服务**，而不是保证列表在每一刻都绝对精确。

**选 AP 的核心逻辑**：
- **选 CP 的代价**：网络分区时注册中心拒绝服务 → 调用方完全找不到下游 → 整个系统雪崩
- **选 AP 的代价**：分区时可能返回少量已下线的实例地址（旧数据），但调用方有重试和熔断兜底，影响可控

> 一句话：**宁可拿到一个可能过期的地址列表，也不要注册中心直接挂掉**。

**Nacos 临时实例 vs 持久实例**：

| 类型 | 模式 | 心跳方式 | 适用场景 |
|------|------|---------|---------|
| **临时实例**（默认） | **AP** | 客户端主动心跳，超时自动摘除 | 微服务实例（动态上下线） |
| **持久实例** | **CP** | 服务端主动探测，实例不会自动删除 | 固定基础设施（数据库节点、第三方服务） |

**各注册中心 CAP 选择对比**：

| 注册中心 | CAP 选择 | 原因 |
|---------|---------|------|
| **Eureka** | AP | 自我保护机制，宁可保留旧数据也不剔除实例 |
| **Nacos**（临时实例） | AP | 微服务场景优先可用性 |
| **ZooKeeper** | CP | 设计目标是分布式协调（锁、选主），强一致是必须的 |
| **Consul** | CP | 定位是 Service Mesh，强一致更重要 |

> **注意**：ZooKeeper 被用作注册中心（如早期 Dubbo）其实是"将就用"，CP 特性在注册中心场景下反而是缺点，这也是后来 Nacos 替代它的原因之一。

---

**Q4：Eureka 的自我保护机制是什么？**

A：
- 正常情况：Eureka Server 收到心跳（默认 30 秒/次），超过 90 秒未收到则剔除实例
- **自我保护触发**：15 分钟内心跳续约比例低于 85% → Eureka 认为是网络故障而非实例真的下线
- **保护行为**：不剔除任何实例（即使已下线），保护服务列表完整性
- **目的**：宁可保留已下线的实例（返回旧数据），也不误删正常实例（保可用性 → AP）

## 三、服务网关

**Q5：Spring Cloud Gateway 的核心原理？**

A：
**核心概念**：
- **Route（路由）**：匹配条件 + 转发目标 URI
- **Predicate（断言）**：匹配请求（路径、方法、Header、时间等）
- **Filter（过滤器）**：请求前/后的处理逻辑（认证、限流、日志、改写请求）

**执行流程**：
```
客户端请求 → Gateway Handler Mapping（匹配路由）
           → Gateway Web Handler（组装过滤器链）
           → Pre-Filters（前置处理：认证、限流）
           → 代理请求到下游服务
           → Post-Filters（后置处理：日志、响应改写）
           → 返回响应
```

**基于 Netty**：非阻塞式 IO，高性能。相比 Zuul 1.0（基于 Servlet 阻塞式），吞吐量提升数倍。

## 四、限流与熔断

**Q6：四种限流算法的原理和对比？**

A：

**1. 固定窗口计数器**：
- 每个固定时间窗口（如 1 秒）维护一个计数器
- 问题：窗口边界突发流量（前一窗口末尾 + 后一窗口开头可能瞬间超限 2x）

**2. 滑动窗口计数器**：
- 将窗口细分为多个小窗格（如 1 秒分 10 个 100ms 窗格）
- 统计最近 N 个窗格的总请求数
- 窗格越细精度越高，解决了固定窗口的边界问题

**3. 漏桶（Leaky Bucket）**：
- 请求进入桶中排队，以固定速率流出处理
- 优点：输出速率平滑
- 缺点：无法应对突发流量（即使系统有余力也只能匀速处理）

**4. 令牌桶（Token Bucket）**：
- 以固定速率向桶中放令牌，请求来了取令牌，有令牌就处理，没有就拒绝
- 优点：允许一定程度的突发（桶中积累的令牌可一次性消费）
- **Guava RateLimiter** 和 **Sentinel** 都基于令牌桶

---

**Q7：Sentinel 的核心原理和使用？**

A：Sentinel 是阿里开源的流量治理组件：

**核心功能**：
1. **流量控制**：QPS 限流、线程数限流、关联限流、链路限流
2. **熔断降级**：慢调用比例、异常比例、异常数
3. **热点参数限流**：对特定参数值限流（如某个用户 ID）
4. **系统自适应保护**：根据系统负载（CPU、线程数）动态调整

**熔断状态机**：
```
CLOSED（正常）→ 触发阈值 → OPEN（熔断，快速失败）
                            ↓ 经过熔断时长
                        HALF-OPEN（放行一个探测请求）
                            ↓ 成功 → CLOSED
                            ↓ 失败 → OPEN
```

**Q8：Hystrix、Sentinel、Resilience4j 的区别？**

## 六、链路追踪

**Q10：分布式链路追踪的原理？**

A：基于 Google Dapper 论文的核心概念：

- **Trace**：一次完整的请求链路（从入口到所有下游调用），用全局唯一的 TraceId 标识
- **Span**：链路中的一个调用单元（一次 RPC、一次 DB 查询），包含开始时间、结束时间、状态等
- **Parent Span**：Span 之间通过 ParentSpanId 形成树形结构

**数据流**：
```
请求入口生成 TraceId + SpanId
→ 调用下游时通过 Header 传递 TraceId + ParentSpanId
→ 每个服务记录自己的 Span 数据
→ 异步上报到收集器（Collector）
→ 存储（ES/ClickHouse）→ 查询展示
```

## 七、服务治理综合

**Q12：微服务架构下如何保证高可用？**

A：
| 层次 | 手段 |
|------|------|
| **服务实例** | 多实例部署 + 注册中心健康检查 + 自动摘除不健康节点 |

| **部署** | K8s 滚动更新 + 蓝绿/灰度发布 |
| **可观测** | 链路追踪 + 指标监控（Prometheus + Grafana）+ 日志聚合（ELK） |

**Q13：服务雪崩是什么？如何防止？**

A：
**雪崩**：一个服务不可用 → 上游调用阻塞/超时 → 上游线程耗尽 → 上游也不可用 → 级联故障

**防止手段**：
1. **超时控制**：所有远程调用设置合理超时（不要用默认的无限等待）
2. **熔断器**：下游错误率/慢调用率超过阈值 → 快速失败，不再调用
3. **线程池/信号量隔离**：每个下游服务独立的线程池，避免一个服务拖垮全部
4. **限流**：保护自身不被打垮
5. **降级兜底**：返回默认值/缓存数据/友好提示

---

**Q14：灰度发布和蓝绿部署的区别？**

A：
| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **蓝绿部署** | 维护两套环境（蓝=当前，绿=新版），验证通过后切换流量 | 回滚快（切回蓝） | 需要双倍资源 |
| **灰度发布（金丝雀）** | 新版本先接 5%→20%→50%→100% 的流量，逐步放量 | 风险可控，资源利用高 | 实现复杂（需要流量路由能力） |
| **滚动更新** | 逐个替换旧实例为新实例 | K8s 原生支持 | 新旧版本短时间共存 |

## 八、面试高频：如何做微服务治理？

> 回答思路：从"问题视角"出发，而不是单纯罗列技术栈，体现系统化思维。

**开场定调**：
> "我们的微服务治理主要围绕**服务发现、流量管控、容错保障、可观测性**四个维度来做，用的是 Spring Cloud Alibaba 技术栈。"

**1. 服务注册与发现**
- 用 **Nacos** 作为注册中心，服务启动时自动注册，客户端心跳保活
- Nacos 支持 AP/CP 切换，临时实例用 AP（高可用），持久实例用 CP（强一致）
- 健康检查：客户端心跳 + 服务端主动探测，不健康实例自动摘除
- 加分点：Nacos 内置配置中心，注册 + 配置统一一个组件，降低运维复杂度

**2. 流量管控（限流 + 熔断 + 降级）**
- 用 **Sentinel** 做流量治理，网关层和服务层双重规则
- **限流**：对核心接口设置 QPS 阈值，超出后快速失败而不是排队等待
- **熔断**：慢调用比例（响应时间 > 500ms 且比例 > 50%）触发熔断，走 Fallback 兜底
- **热点参数限流**：对高频参数（如热门商品 ID）单独限流，防止缓存击穿
- 加分点：Sentinel 规则持久化到 Nacos，服务重启后规则不丢失，支持动态推送

**3. 服务容错**
- **超时控制**：所有 RPC、DB、Redis、HTTP 调用都设置超时，绝不用默认无限等待
- **重试**：只对幂等接口开启重试（GET 查询、带幂等 Token 的写接口），非幂等不重试
- **隔离**：信号量隔离，每个下游服务独立并发限制，防止雪崩
- **降级兜底**：核心链路每个调用都有 Fallback，返回缓存数据或默认值

**4. 配置中心**
- 用 **Nacos Config** 管理配置，支持动态刷新（`@RefreshScope`）
- 按环境隔离：dev/test/prod 用不同 Namespace
- 敏感配置（数据库密码等）走加密存储

**5. 可观测性**
- **链路追踪**：SkyWalking Java Agent 无侵入接入，全链路耗时可视化
- **指标监控**：Prometheus 采集 + Grafana 看板，监控 QPS、RT（P99）、错误率、JVM 指标
- **日志**：ELK 日志聚合，日志里带 TraceId，从链路追踪直接跳到对应日志
- **告警**：错误率 > 1%、P99 > 1s、CPU > 80% 触发钉钉告警

**6. 发布安全**
- **灰度发布**：新版本先发 1 个实例，通过 Nacos 标签路由放 5% 流量，观察无异常后全量
- **优雅停机**：服务下线前先从注册中心摘除，等存量请求处理完再关闭
- **快速回滚**：保留上一版本镜像，出问题 1 分钟内可回滚

**总结收尾**：
> "整体来说，治理思路是：**入口层限流、调用层熔断、数据层降级、全链路可观测**，保证核心链路在任何情况下都有兜底。"

## 九、面试高频：如何保障系统稳定性？

> 回答思路：从**事前预防、事中应对、事后复盘**三个层次展开，体现系统性。

**开场定调**：
> "我们的稳定性保障体系分三个层次：**事前预防**（设计和测试阶段）、**事中应对**（运行时流量管控和容错）、**事后复盘**（可观测性和故障总结）。"

### 事前预防

**架构设计层面**：
- **服务拆分合理**：核心链路和非核心链路隔离，非核心故障不影响核心
- **异步解耦**：非实时操作走 MQ 异步处理（发短信、更新积分），减少主链路依赖
- **避免单点**：数据库主从、Redis 集群、MQ 集群，关键组件无单点

**容量规划**：
- 压测评估系统峰值 QPS，结合历史流量预估大促时的流量倍数
- 提前扩容，保留 **30%~50% 的水位余量**
- 限流阈值基于压测结果设定，而不是拍脑袋

**测试保障**：
- **全链路压测**：模拟真实流量压测，发现瓶颈点
- **混沌工程**：主动注入故障（杀进程、断网络、慢 SQL），验证系统容错能力
- **回归测试**：发布前自动化验证核心链路

### 事中应对（运行时）

**流量管控**：
- **限流**：Sentinel 在网关层和服务层双重限流，超出阈值快速失败
- **熔断**：下游慢调用/错误率超阈值自动熔断，防止雪崩
- **降级**：核心接口有 Fallback 兜底（缓存数据、默认值、友好提示）

**容错设计**：
- **超时控制**：所有远程调用设置超时，绝不用默认无限等待
- **重试**：只对幂等接口重试，非幂等接口不重试（避免重复扣款）
- **隔离**：核心业务和非核心业务线程池隔离，互不影响
- **幂等设计**：所有写接口支持幂等，防止重试导致数据重复

**发布安全**：
- **灰度发布**：新版本先放 5% 流量，观察错误率和响应时间无异常后再全量
- **优雅停机**：服务下线前先从注册中心摘除，等存量请求处理完再关闭
- **快速回滚**：发布后保留上一版本镜像，出问题 1 分钟内可回滚

### 事后复盘（可观测性）

**监控告警**：
- **指标监控**：Prometheus + Grafana，监控 QPS、RT（P99）、错误率、JVM 堆内存、GC 频率
- **告警规则**：错误率 > 1%、P99 > 1s、CPU > 80% 触发钉钉/电话告警
- **业务监控**：核心业务指标（下单量、支付成功率）异常也要告警，不只看技术指标

**链路追踪**：
- SkyWalking 全链路耗时可视化，快速定位是哪个服务、哪个 SQL 慢
- 日志里带 TraceId，从告警 → 链路 → 日志一键跳转，排查时间从小时级降到分钟级

**故障复盘**：
- 每次故障后写 **RCA（根因分析）**：故障时间线、根因、影响范围、修复措施、预防措施
- 建立故障等级制度（P0/P1/P2），P0 故障 24 小时内必须完成复盘

**总结收尾**：
> "整体思路是：**设计时考虑容错、运行时管控流量、出问题快速发现和恢复**。稳定性不是靠某一个技术点，而是整个研发流程的系统性保障。"

**面试官可能追问**：

| 追问 | 准备方向 |
|------|---------|
