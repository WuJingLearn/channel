## 一、数据结构与底层编码

**Q1：Redis 5 种基本数据类型及其底层编码？**

A：
| 数据类型 | 底层编码 | 转换条件 |
|---------|---------|---------|
| String | int（整数）/ embstr（≤44 字节短字符串）/ raw（长字符串） | 自动根据内容和长度选择 |
| List | quicklist（ziplist + 双向链表的混合体，Redis 3.2+） | — |
| Hash | ziplist（元素少且小）/ hashtable | 元素数 > 128 或单值 > 64 字节时转 hashtable |
| Set | intset（全是整数且数量少）/ hashtable | 元素数 > 512 或有非整数时转 hashtable |
| ZSet | ziplist（元素少且小）/ skiplist + hashtable | 元素数 > 128 或单值 > 64 字节时转 skiplist |

---

**Q2：SDS（Simple Dynamic String）相比 C 字符串有什么优势？**

A：
| 维度 | C 字符串 | SDS |
|------|---------|-----|
| 获取长度 | O(n) 遍历 | O(1)，直接读 `len` 字段 |
| 缓冲区溢出 | 手动管理，容易溢出 | 自动扩容检查 |
| 内存重分配 | 每次修改都要重分配 | 空间预分配 + 惰性释放 |
| 二进制安全 | 以 `\0` 判断结尾，不能存二进制 | 以 `len` 判断，可以存任意二进制数据 |

SDS 结构：`len`（已使用长度）+ `alloc`（分配长度）+ `flags`（类型标记）+ `buf[]`（实际数据）

---

**Q3：跳表（SkipList）的原理？为什么 ZSet 用跳表而不用红黑树？**

A：
- 跳表是多层有序链表，每层是下层的"快速通道"，查找时从顶层开始逐层下降
- 时间复杂度：查找/插入/删除 O(log n)，空间 O(n)

**为什么不用红黑树**：
1. 跳表实现更简单，代码量少，容易调试
2. 范围查询更方便：跳表底层是有序链表，直接遍历；红黑树需要中序遍历
3. 内存分配更灵活：跳表节点大小可变，红黑树每个节点固定两个指针
4. 并发友好：跳表可以用 CAS 实现无锁并发（虽然 Redis 单线程不需要）

## 二、持久化

**Q4：RDB 和 AOF 的原理和区别？**

A：
| 维度 | RDB（快照） | AOF（追加日志） |
|------|-----------|----------------|
| 原理 | 定时将内存数据全量快照到磁盘（二进制文件 dump.rdb） | 将每个写命令追加到文件末尾 |
| 触发 | `save`（阻塞）/ `bgsave`（fork 子进程）/ 配置自动触发 | 实时追加 |
| 文件大小 | 小（压缩的二进制） | 大（文本命令，会重写压缩） |
| 恢复速度 | 快 | 慢（逐条回放） |
| 数据安全 | 可能丢失上次快照到宕机之间的数据 | 取决于刷盘策略（最多丢 1 秒） |

```rdb
# 格式：save <seconds> <changes>
# 含义：在 seconds 秒内，如果有 changes 次写操作，就自动触发 bgsave

save 900 1       # 900 秒（15 分钟）内至少 1 次写操作
save 300 10      # 300 秒（5 分钟）内至少 10 次写操作
save 60 10000    # 60 秒（1 分钟）内至少 10000 次写操作

# 多个条件是 OR 关系，任意一个满足就触发
# 关闭自动 RDB：
save ""
```


**AOF 刷盘策略**（`appendfsync`）：
| 策略 | 说明 | 安全性 | 性能 |
|------|------|--------|------|
| always | 每次写命令都 fsync | 最安全（不丢数据） | 最慢 |
| **everysec** | 每秒 fsync 一次（推荐） | 最多丢 1 秒 | 折中 |
| no | 由 OS 决定何时 fsync | 可能丢较多数据 | 最快 |

**Q5：什么是 AOF 重写？RDB 的 bgsave 怎么做到不影响服务？**

A：
**AOF 重写**：AOF 文件越来越大 → 后台重写（`BGREWRITEAOF`）：读取当前内存数据，生成最小命令集替换旧 AOF 文件。不是对旧文件做分析，而是直接从内存生成。

**为什么需要重写？** AOF 记录每一条写命令，会产生大量冗余。例如对同一个 key 执行 `SET counter 1` → `INCR` × 3 → `DEL` → `SET counter 100`，6 条命令最终等价于 `SET counter 100` 一条，但 AOF 文件全部保留了。

**AOF 重写的详细执行流程**：

```
1. 触发重写（手动 BGREWRITEAOF 或 auto-aof-rewrite 自动触发）
       │
2. 主进程 fork() 子进程（和 bgsave 一样，利用 COW 机制）
       │
       ├──→ 子进程：遍历内存数据，将每个 key 的当前值转换为一条写命令，
       │     写入新的 AOF 临时文件（如 counter=100 → 只生成 "SET counter 100"）
       │
       ├──→ 主进程：继续处理客户端请求，新的写命令同时追加到：
       │     ① 旧 AOF 文件（保证安全，万一重写失败还有旧文件）
       │     ② AOF 重写缓冲区（aof_rewrite_buf）
       │
3. 子进程写完新 AOF 文件，通知主进程
       │
4. 主进程将重写缓冲区中积累的增量命令追加到新 AOF 文件末尾
       │
5. 用 rename() 原子性地替换旧 AOF 文件
```

**关键设计点**：
| 要点 | 说明 |
|------|------|
| 不分析旧文件 | 直接从内存生成，逻辑简单且结果最优 |
| fork + COW | 和 bgsave 相同的机制，子进程看到 fork 时刻的快照 |
| 重写缓冲区 | 解决重写期间新写命令的同步问题，保证数据不丢 |
| 双写保障 | 重写期间新命令同时写旧 AOF 和重写缓冲区，即使重写失败也不丢数据 |
| 原子替换 | 用 `rename()` 系统调用替换文件，保证切换的原子性 |

**bgsave 的 fork + COW**：
1. 主进程 `fork()` 子进程，子进程共享父进程的内存页（Copy-On-Write）
2. 子进程遍历内存数据，写入 RDB 文件
3. 主进程继续处理请求。如果有写操作，OS 会复制被修改的内存页（COW），子进程仍然看到 fork 那一刻的数据快照
4. 子进程写完后通知主进程替换旧 RDB 文件

---

**Q5.1：RDB bgsave 的详细执行过程？写入期间的新写命令会进入 RDB 文件吗？**

A：
**不会。RDB 是"时间点快照"，只保存 fork 那一瞬间的数据，fork 之后的写命令不会进入当前 RDB 文件。**

需要区分两个阶段：

| 阶段 | 耗时 | 是否阻塞主线程 | 说明 |
|------|------|---------------|------|
| `fork()` 系统调用 | 毫秒~秒级 | **是**（短暂阻塞） | 只复制页表，不复制实际数据，所以很快 |
| 子进程写磁盘 | 秒~分钟级 | **否** | 子进程在后台遍历 fork 时刻的内存快照，写入 RDB 文件 |

**写入耗时参考**（取决于数据量和磁盘性能）：
- 1GB 数据：SSD 约 1~2 秒，HDD 约 3~5 秒
- 10GB 数据：SSD 约 10~20 秒，HDD 约 30~60 秒
- 25GB+ 数据：可能需要几十秒到几分钟

**fork 之后的写命令数据去哪了？**
- **纯 RDB 模式**：只存在于内存中，Redis 崩溃则丢失（最多丢失两次快照间隔的数据）
- **AOF 模式**：每条写命令追加写入 AOF 文件，根据 `appendfsync` 策略保证持久化
- **RDB + AOF 混合模式**（Redis 4.0+）：AOF 重写时先以 RDB 格式写全量，再以 AOF 格式追加增量，兼顾恢复速度和数据安全

**生产建议**：
- 通过 `INFO persistence` 的 `latest_fork_usec` 监控 fork 耗时
- 单实例内存建议不超过 10~15GB，避免 fork 和写入时间过长
- 关闭 Linux THP（Transparent Huge Pages），否则 fork 会显著变慢
- 使用 SSD 磁盘

---

**Q5.2：COW（Copy-On-Write）的底层原理？**

A：
**COW 的核心思想：不在 fork 时复制数据，而是在"写"的时候才复制被修改的那一页。**

**阶段一：fork 瞬间 —— 只复制页表，不复制数据**

```
fork 之前：
  主进程页表 ──→ 物理内存页（数据 A=1, B=2）

fork 之后：
  主进程页表 ──┐
               ├──→ 物理内存页（数据 A=1, B=2）← 标记为"只读"
  子进程页表 ──┘
  （两个页表指向同一份物理内存，只复制了页表本身）
```

- 页表很小（只是虚拟地址→物理地址的映射），所以 fork 非常快
- 所有共享的物理内存页被 OS 标记为**只读**

**阶段二：有写操作时 —— 只复制被修改的那一页**

```
主进程执行 SET A=100：
  1. 主进程尝试写内存页 A → OS 发现该页是"只读" → 触发缺页中断
  2. OS 复制一份内存页 A 的副本
  3. 主进程页表指向新副本，在新副本上修改 A=100
  4. 子进程页表仍指向原始页，看到的仍是 A=1

  主进程页表 ──→ 新副本（A=100）  ← 主进程在这里改
  子进程页表 ──→ 原始页（A=1）    ← 子进程仍然看到旧值
  两个进程仍共享 ──→ 未修改的页（B=2）← 没人改过，不需要复制
```

**关键结论**：
- **不是所有数据都保存两份**，只有被修改的内存页才会被复制
- 如果 fork 后没有写操作，额外内存开销几乎为零
- 如果 fork 后所有数据都被修改，最坏情况下内存接近 2 倍
- 实际生产中，bgsave 期间额外内存开销一般在 **10%~30%**

> 类比：fork 就像"拍照"，是瞬间按下快门的动作。子进程写磁盘是"冲洗照片"的过程。照片内容在快门按下那一刻就定格了，之后场景怎么变化都不影响照片内容。COW 保证了子进程始终看到的是快门那一刻的数据。

**Q6：Redis 4.0 的混合持久化是什么？**

A：`aof-use-rdb-preamble yes`（Redis 4.0+）

AOF 重写时，先将当前数据以 RDB 格式写入 AOF 文件头部，再将重写期间的增量命令以 AOF 格式追加到文件尾部。

**优点**：兼顾 RDB 的快速恢复和 AOF 的数据安全性。恢复时先加载 RDB 部分（快），再回放 AOF 增量部分。

---

**Q6.1：生产环境如何选择持久化方式？**

A：

**推荐方案：RDB + AOF 混合持久化（Redis 4.0+，大多数生产环境首选）**

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 一般业务缓存（可从 DB 重建） | 纯 RDB 或关闭持久化 | 数据丢了可以从数据库重新加载，追求性能 |
| 重要业务数据（如会话、计数器） | RDB + AOF 混合 | 兼顾恢复速度和数据安全 |
| 金融/支付等强一致场景 | AOF always + 主从 | 不能丢数据，牺牲一定性能换安全 |
| 纯缓存，数据全在 DB | 关闭持久化 | 省磁盘 IO，性能最优 |

**生产环境典型配置**：
```
# 关闭自动 RDB（由从节点或定时任务负责，避免主节点 fork 阻塞）
save ""

# 开启 AOF
appendonly yes
appendfsync everysec

# 开启混合持久化（Redis 4.0+）
aof-use-rdb-preamble yes

# AOF 自动重写触发条件（两个条件同时满足才触发）
auto-aof-rewrite-percentage 100    # AOF 文件比上次重写后增长 100%（翻倍）
auto-aof-rewrite-min-size 64mb     # AOF 文件至少达到 64MB（防止小文件频繁重写）
```

**`auto-aof-rewrite` 触发逻辑**：`AOF增长比例 >= percentage` **AND** `AOF文件大小 >= min-size`，例如上次重写后 64MB，增长到 128MB 时触发。数据量大的实例可将 `min-size` 调大到 256MB。

**避坑建议**：
- **不要只用纯 AOF**：恢复速度慢，文件体积大
- **不要在主节点做 bgsave**：fork 阻塞影响线上服务，建议在**从节点**执行
- **不要用 `appendfsync always`**（除非强一致要求）：每次写都 fsync，性能下降严重

## 三、线程模型

**Q7：Redis 为什么用单线程还这么快？**

A：
1. **纯内存操作**：数据在内存中，操作耗时纳秒级
2. **IO 多路复用**：使用 epoll/kqueue，单线程监听大量连接，有事件才处理
3. **避免锁开销**：单线程无需加锁，无上下文切换
4. **高效数据结构**：hash、skiplist 等专门优化的内存数据结构
5. **单线程模型简单**：避免并发编程的复杂性和 bug

**瓶颈不在 CPU**：Redis 的瓶颈在网络 IO 和内存，不在 CPU 计算。

---

**Q8：Redis 6.0 为什么引入多线程？和单线程有什么区别？**

A：
- Redis 6.0 的多线程只用于**网络 IO 的读写和协议解析**
- **命令执行仍然是单线程**
- 默认关闭，需要配置 `io-threads-do-reads yes` + `io-threads 4`

```
客户端请求 → 多线程读取并解析 → 单线程执行命令 → 多线程写回响应
```

解决的问题：高并发场景下网络 IO 成为瓶颈（大量 read/write 系统调用），多线程分担 IO 压力，提升吞吐量。

## 四、缓存问题

**Q9：缓存穿透、击穿、雪崩的区别和解决方案？**

A：

### 缓存穿透（查不存在的数据）
**问题**：请求的数据在缓存和数据库中都不存在，每次都穿透到数据库。
**解决**：
1. **布隆过滤器**：在缓存前加一层布隆过滤器，不存在的 key 直接拦截
2. **空值缓存**：将不存在的 key 缓存为空值（设短 TTL，如 30 秒）
3. 参数校验，拦截非法请求

### 缓存击穿（热点 key 过期）
**问题**：某个热点 key 过期的瞬间，大量并发请求同时穿透到数据库。
**解决**：
1. **互斥锁**：只允许一个线程重建缓存，其他线程等待。`SETNX lock_key 1 EX 10`
2. **逻辑过期**：缓存不设 TTL，在 value 中记录逻辑过期时间，过期后异步更新

### 缓存雪崩（大量 key 同时过期）
**问题**：大量缓存同时过期或 Redis 宕机，请求全部打到数据库。
**解决**：
1. **随机 TTL**：过期时间加随机值，避免同时过期
2. **多级缓存**：本地缓存（Caffeine）+ Redis + 数据库
3. **集群高可用**：Redis Sentinel 或 Redis Cluster

---

**Q10：如何保证缓存与数据库的一致性？**

A：
| 策略 | 做法 | 问题 |
|------|------|------|
| **Cache Aside（旁路缓存）** | 读：先缓存 → 命中返回，未命中查 DB 写缓存。写：先更新 DB → 再删缓存 | 极小概率不一致（先删缓存被其他线程回填旧值） |
| 延迟双删 | 写：先删缓存 → 更新 DB → 延迟 N 毫秒再删缓存 | 延迟时间难确定 |
| 监听 Binlog | 通过 Canal 监听 MySQL Binlog → 异步更新/删除缓存 | 增加架构复杂度 |

**推荐方案**：Cache Aside + 消息队列重试删除（删缓存失败时发 MQ 重试）。

**为什么是删缓存而不是更新缓存**：
- 更新缓存可能存在并发写覆盖问题（A 更新 DB → B 更新 DB → B 更新缓存 → A 更新缓存 → 缓存是 A 的旧值）
- 删缓存保证下次读会从 DB 加载最新值（lazy 策略）

## 五、分布式锁

**Q11：Redis 分布式锁的标准实现？**

A：
```bash
# 加锁（原子操作）
SET lock_key unique_value NX EX 30
# NX：不存在才设置（互斥）
# EX 30：30 秒过期（防死锁）
# unique_value：每个客户端唯一标识（防误删）

# 释放锁（Lua 脚本保证原子性）
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

**注意事项**：
1. 加锁和设置过期时间必须是原子操作（`SET ... NX EX`）
2. 释放锁前必须校验是自己的锁（比较 unique_value）
3. 过期时间要合理（太短可能业务没执行完锁就释放了）

---

**Q12：Redisson 是如何解决锁续期问题的？**

A：Redisson 的 **WatchDog（看门狗）**机制：
1. 加锁成功后，启动一个后台线程每隔 `lockWatchdogTimeout / 3`（默认 10 秒）检查锁是否还持有
2. 如果持有则续期（重新设置过期时间为 30 秒）
3. 客户端宕机 → WatchDog 停止 → 锁到期自动释放 → 不会死锁
4. 支持可重入锁（hash 结构记录重入次数）

---

**Q13：RedLock 算法是什么？为什么有争议？**

A：
**RedLock 算法**（Antirez 提出）：
1. 获取当前时间
2. 依次向 N 个独立的 Redis 节点加锁（相同 key、value、过期时间）
3. 如果在**超过半数（N/2 + 1）**的节点上加锁成功，且总耗时 < 过期时间 → 加锁成功
4. 如果失败，向所有节点释放锁

**争议**（Martin Kleppmann vs Antirez）：
- Martin：分布式系统中进程暂停（GC、页面置换）可能导致锁过期后客户端仍认为持有锁。建议用 fencing token（递增序号）保证安全
- Antirez：反驳说时钟跳变和进程暂停的场景在实际生产中可以通过运维手段控制

**实际建议**：对强一致性要求高的场景用 ZooKeeper 分布式锁；一般场景用 Redisson 的单节点锁即可。

## 六、Redis Cluster

**Q14：Redis Cluster 的架构和数据分片原理？**

A：
- 无中心化架构，每个节点都保存数据和集群状态
- 使用**哈希槽（Hash Slot）**分片：共 16384 个槽，每个节点负责一部分槽
- Key 的槽定位：`CRC16(key) % 16384`
- 节点间通过 **Gossip 协议**交换状态信息（PING/PONG/MEET/FAIL）

**客户端重定向**：
- 客户端访问的 key 不在当前节点 → 返回 `MOVED slot ip:port` → 客户端重定向到目标节点
- 正在迁移中 → 返回 `ASK slot ip:port` → 临时重定向

---

**Q15：Redis Sentinel 和 Redis Cluster 的区别？**

A：
| 维度 | Sentinel | Cluster |
|------|---------|---------|
| 定位 | 高可用方案（主从自动故障转移） | 高可用 + 数据分片 |
| 数据分布 | 不分片，每个节点存全量数据 | 分片，每个节点存部分数据 |
| 扩展性 | 纵向扩展（增大单节点内存） | 横向扩展（增加节点） |
| 适用场景 | 数据量 < 单节点内存上限 | 数据量大，需要分片 |

## 七、其他高频题

**Q16：Redis 的内存淘汰策略有哪些？**

A：`maxmemory-policy` 配置，8 种策略：
| 策略 | 说明 |
|------|------|
| noeviction（默认） | 不淘汰，写满后写操作返回错误 |
| allkeys-lru | 从所有 key 中 LRU 淘汰 |
| allkeys-lfu | 从所有 key 中 LFU 淘汰（Redis 4.0+） |
| allkeys-random | 从所有 key 中随机淘汰 |
| volatile-lru | 从设了 TTL 的 key 中 LRU 淘汰 |
| volatile-lfu | 从设了 TTL 的 key 中 LFU 淘汰 |
| volatile-random | 从设了 TTL 的 key 中随机淘汰 |
| volatile-ttl | 淘汰 TTL 最短的 key |

**推荐**：缓存场景用 `allkeys-lru`；有热点数据用 `allkeys-lfu`。

**Q17：Redis 的大 key 问题怎么处理？**

A：

**大 Key 判断标准（官方/业界通用）**：

| 数据类型 | 大 Key 判断标准 |
|---------|----------------|
| **String** | value > 10KB 需关注，> 1MB 认为是大 Key |
| **Hash/Set/ZSet/List** | 元素数 > 5000 或 总大小 > 10MB |

**String 类型详细分级**：
| 大小 | 风险等级 |
|------|---------|
| < 10KB | 正常 ✅ |
| 10KB ~ 1MB | 需关注 ⚠️ |
| > 1MB | 大 Key 🚨 |
| > 10MB | 严重风险 🚨🚨 |

**集合类型详细分级**：
| 维度 | 大小/数量 | 风险等级 |
|------|----------|---------|
| 元素数量 | < 1000 | 正常 ✅ |
| 元素数量 | 1000 ~ 5000 | 需关注 ⚠️ |
| 元素数量 | > 5000 | 大 Key 🚨 |
| 总大小 | > 10MB | 大 Key 🚨 |

。**危害**：

| 问题 | 说明 |
|------|------|
| **阻塞主线程** | 大 Key 读写耗时较长（尤其删除），阻塞 Redis 主线程 |
| **网络带宽压力** | 大 Key 读写占用大量带宽，影响其他请求响应速度 |
| **内存不均** | Cluster 场景下大 Key 集中在单个节点，导致数据倾斜 |
| **主从同步延迟** | 主从全量同步时，大 Key 需要完整传输，增加同步时间 |

---

**Q17.1：为什么大 Key 删除会阻塞主线程？**

A：
**Redis 没有垃圾回收机制，删除集合类型的大 Key 需要遍历并逐个释放每个元素的内存。**

**不同数据类型的删除复杂度**：

| 数据类型 | 删除复杂度 | 原因 |
|---------|-----------|------|
| String | O(1) | 只需一次内存释放，即使 value 很大也很快 |
| List | O(N) | 需要遍历链表，逐个释放节点 |
| Hash | O(N) | 需要遍历哈希表，逐个释放 field-value 对 |
| Set | O(N) | 需要遍历哈希表，逐个释放元素 |
| ZSet | O(N) | 需要遍历跳表和哈希表，逐个释放元素 |

**为什么不用"设置引用为 null"？** Redis 采用主动内存管理，没有 GC 机制。好处是内存使用可预测、无 GC 停顿；代价是大 Key 删除需要逐个释放元素，会阻塞主线程。

**解决方案**：使用 `UNLINK`（Redis 4.0+）异步删除：

```
DEL mylist    → 主线程遍历释放 10 万元素，阻塞直到完成
UNLINK mylist → 主线程从字典移除 key（O(1)），后台线程异步释放内存
```

`UNLINK` 原理：如果 value 是集合类型且元素数超过 64，将释放任务交给后台线程，主线程立即返回。

---

**Q17.2：大 List 分页读取会有阻塞问题吗？**

A：**分页读取时不会阻塞，但大 Key 的其他问题仍然存在。**

**分页读取分析**：

```bash
# 每次只取 100 个元素，时间复杂度 O(100)，非常快
LRANGE mylist 0 99
LRANGE mylist 100 199
```

`LRANGE start stop` 的复杂度是 O(N)，N 是返回元素数量，不是总元素数。所以分页读取时单次操作很快，不会阻塞。

**仍然存在的问题**：

| 问题 | 是否存在 | 原因 |
|------|---------|------|
| 内存不均 | ✅ 存在 | Cluster 场景下 10 万元素集中在一个节点 |
| 主从同步延迟 | ✅ 存在 | 全量同步时大 Key 仍需完整传输 |
| 删除阻塞 | ✅ 存在 | 删除时仍需遍历释放所有元素 |
| 网络带宽压力 | ⚠️ 降低 | 单次响应数据量小 |

**发现**：`redis-cli --bigkeys` / `MEMORY USAGE key` / 监控慢日志。

**处理**：
1. **拆分**：大 Hash 按 field 前缀拆成多个小 Hash；大 List 按分页拆成多个小 List
2. **压缩**：序列化压缩 value（如 Snappy、LZ4）
3. **异步删除**：`UNLINK key`（Redis 4.0+，后台线程删除，不阻塞）
4. 分批删除：`HSCAN` + `HDEL` 分批删 Hash 字段

**Q18：Redis 的 Pipeline 和 Lua 脚本分别解决什么问题？**

A：
| 特性 | Pipeline | Lua 脚本 |
|------|---------|---------|
| 解决问题 | 减少网络 RTT（批量发送命令） | 保证多命令的原子性 |
| 原子性 | 不保证（中间可能插入其他命令） | 保证（整个脚本原子执行） |
| 用途 | 批量读写（如批量 GET/SET） | 分布式锁释放、库存扣减等需要原子操作的场景 |

---

**Q18.1：Pipeline 的原理是什么？为什么不保证原子性？**

A：
**Pipeline 的本质是客户端层面的网络优化，不是 Redis 协议的新功能。**

```
普通模式：客户端 → CMD1 → 等响应 → CMD2 → 等响应 → CMD3 → 等响应（3 次 RTT）
Pipeline：客户端 → CMD1 CMD2 CMD3 → 统一读取 → 响应1 响应2 响应3（1 次 RTT）
```

**原理**：普通模式下客户端 `write()` 后立刻 `read()` 阻塞等响应，人为制造了串行。Pipeline 模式下客户端连续 `write()` 多条命令到 TCP 发送缓冲区，最后统一 `read()` 所有响应，利用了 TCP 全双工能力。**服务端对此完全无感知**，仍然逐条解析、逐条执行。

**为什么不保证原子性？** Pipeline 的多条命令在 RESP 协议中是 N 条独立命令。如果数据量大导致 TCP 分包，Redis 第一次 `read()` 可能只读到部分命令并执行，回到事件循环后处理了其他客户端的命令，下一轮才读到剩余命令：

```
TCP 分包场景：
  事件循环第 1 轮：读连接A → 收到 CMD1 CMD2 → 执行
  事件循环第 1 轮：读连接B → 收到 CMD_X → 执行  ← 被其他客户端插入
  事件循环第 2 轮：读连接A → 收到 CMD3 → 执行
```

> 实际上数据量小时（一个 TCP 包），同一 Pipeline 的命令大概率会连续执行，但这是"碰巧如此"而非"协议保证"。

---

**Q18.2：Lua 脚本如果很大，也会 TCP 分包，如何保证原子性？**

A：
**根本区别：Pipeline 是 N 条独立命令，Lua 脚本是 1 条 EVAL 命令。**

| 对比 | Pipeline | Lua 脚本 |
|------|---------|---------|
| 服务端看到的 | N 条独立命令 | **1 条** EVAL 命令 |
| TCP 分包时 | 读到几条执行几条，剩下的下轮再读 | **等到 EVAL 命令完整后才执行** |
| 执行期间 | 可能被其他客户端命令穿插 | **单线程独占，不会被打断** |

**EVAL 命令的处理流程**：
1. Redis 从 TCP 缓冲区读取数据，解析到 EVAL 命令
2. RESP 协议中 `$5000\r\n` 明确标记了脚本内容的长度，Redis 知道数据还没收完
3. Redis **继续等待**，直到读取到完整的 EVAL 命令（不会提前执行不完整的命令）
4. 拿到完整命令后，一次性执行整个 Lua 脚本，期间不处理任何其他客户端的命令

> 原子性的保证来自"一条命令"的语义：Redis 单线程模型下，一条命令的执行过程不会被打断。Pipeline 的多条命令没有这个保证。

**需要原子性时的方案选择**：
| 方案 | 原子性 | 适用场景 |
|------|--------|---------|
| Pipeline | ❌ 不保证 | 批量读写，追求吞吐量 |
| MULTI/EXEC 事务 | ✅ 保证（但不支持回滚） | 简单的多命令原子执行 |
| Lua 脚本 | ✅ 保证（也不支持回滚） | 分布式锁释放、库存扣减等复杂原子操作 |
