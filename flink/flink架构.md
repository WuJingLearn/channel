# Flink 架构与部署模式

## 一、Flink 核心组件

### 1. JobManager（作业管理器）

集群的**主节点（Master）**，负责作业的管理和协调。内部包含三个关键子组件：

- **Dispatcher**：提供 REST 接口，接收作业提交请求，为每个作业启动一个 JobMaster
- **JobMaster**：负责管理单个作业（Job）的执行生命周期，包括：
  - 将 JobGraph 转换为 ExecutionGraph
  - 向 ResourceManager 申请资源（Slot）
  - 将任务调度到 TaskManager 的 Slot 上执行
  - 触发和协调 Checkpoint
  - 处理任务失败后的恢复
- **ResourceManager**：负责集群资源的分配与回收，与外部资源管理框架（YARN / K8s）交互，管理 TaskManager 的 Slot

### 2. TaskManager（任务管理器）

集群的**工作节点（Worker）**，负责实际的数据处理：

 - **任务执行**：运行 JobManager 分配的子任务（SubTask）
- **数据交换**：负责与其他 TaskManager 之间的网络数据传输（Shuffle）
- **心跳上报**：定期向 JobManager 发送心跳，报告自身状态和可用资源

#### Task Slot（任务槽）

每个 TaskManager 是一个 **JVM 进程**，它会将自己的资源（主要是内存）均匀划分成若干个 **Slot**。Slot 是 Flink **资源调度的最小单位**，可以理解为 TaskManager 上的一个"工位"。

```
TaskManager（JVM 进程，总内存 6GB）
┌──────────┐ ┌──────────┐ ┌──────────┐
│  Slot 0  │ │  Slot 1  │ │  Slot 2  │
│  2GB 内存 │ │  2GB 内存 │ │  2GB 内存 │
│  [Task]  │ │  [Task]  │ │  [Task]  │
└──────────┘ └──────────┘ └──────────┘
```

**核心特性**：

| 特性 | 说明 |
|:---|:---|
| **内存隔离** | 每个 Slot 拥有独立的内存空间，Slot 之间内存互不干扰 |
| **CPU 共享** | Slot 之间**不隔离 CPU**，共享 TaskManager 的所有 CPU 核心 |
| **运行内容** | 每个 Slot 中运行一个或多个 SubTask（通过算子链合并或 Slot 共享） |
| **数量配置** | 在 `flink-conf.yaml` 中通过 `taskmanager.numberOfTaskSlots` 配置，通常设为 CPU 核心数 |

**Slot 共享（Slot Sharing）**：

默认情况下，Flink 允许**不同算子的 SubTask 共享同一个 Slot**，但**同一个算子的不同 SubTask 不能在同一个 Slot 中**（否则就失去了并行的意义）。

```
✅ 合法（不同算子的 SubTask 共享 Slot）：
┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐
│     Slot 0       │ │     Slot 1       │ │     Slot 2       │
│  Source[0]       │ │  Source[1]       │ │  Source[2]       │
│  Map[0]          │ │  Map[1]          │ │  Map[2]          │
│  Sink[0]         │ │  Sink[1]         │ │  Sink[2]         │
└──────────────────┘ └──────────────────┘ └──────────────────┘
每个 Slot 包含一条完整的数据处理链路，只需 3 个 Slot

❌ 非法（同一算子的不同 SubTask 不能在同一 Slot）：
┌──────────────────┐
│     Slot 0       │
│  Map[0]          │  ← 同一个算子的两个 SubTask
│  Map[1]          │  ← ❌ 不允许！
└──────────────────┘
```

**Slot 共享的好处**：
- **减少所需 Slot 总数**：所需 Slot 数 = 作业中最大算子并行度（而非所有算子并行度之和）
- **资源利用更均衡**：每个 Slot 内包含一条完整的处理链路，避免某些 Slot 只跑轻量算子而浪费资源

**Slot 数量与并行度的关系**：
- **Slot 总数** = TaskManager 数量 × 每个 TaskManager 的 Slot 数
- **作业所需 Slot 数** = 作业中最大的算子并行度
- 如果集群可用 Slot 不足，作业将**无法启动**

**算子链 vs Slot 共享的区别**：

| 概念 | 含义 | 结果 |
|:---|:---|:---|
| **算子链（Operator Chaining）** | 多个算子的 SubTask 合并成**一个 Task**，在**同一个线程**中执行 | 减少线程切换和序列化开销 |
| **Slot 共享（Slot Sharing）** | 不同算子的 SubTask 放在**同一个 Slot** 中，但仍然是**独立的 Task/线程** | 减少所需 Slot 总数，提高资源利用率 |

### 3. Client（客户端）

- 将用户编写的 Flink 程序编译、优化为 **JobGraph**
- 将 JobGraph 提交给 Dispatcher
- 可选择保持连接获取执行结果（attached），或提交后断开（detached）

> **组件交互流程**：Client → Dispatcher → JobMaster → ResourceManager → TaskManager

## 二、Flink 部署模式

Flink 提供了三种部署模式，核心区别在于：**集群生命周期**、**资源隔离程度**、**main() 方法的执行位置**。

### 1. Session Mode（会话模式）

```
┌─────────────────────────────────────────┐
│          共享的 Flink 集群（长期运行）       │
│  ┌───────────┐                          │
│  │ JobManager│  ← Job A                 │
│  │           │  ← Job B                 │
│  │           │  ← Job C                 │
│  └───────────┘                          │
│  ┌────────┐ ┌────────┐ ┌────────┐       │
│  │  TM 1  │ │  TM 2  │ │  TM 3  │      │
│  │ Slot|Slot│ │ Slot|Slot│ │ Slot|Slot│  │
│  └────────┘ └────────┘ └────────┘       │
└─────────────────────────────────────────┘
```

- **特点**：预先启动一个长期运行的集群，多个作业**共享**同一组 JobManager 和 TaskManager
- **main() 执行位置**：Client 端
- **优点**：
  - 集群已就绪，作业提交后**启动速度快**
  - 资源复用，适合大量小作业
- **缺点**：
  - 资源竞争：多个作业共享 TaskManager，一个作业占用过多资源会影响其他作业
  - 稳定性风险：一个 TaskManager 崩溃可能影响上面运行的所有作业
- **适用场景**：开发测试、频繁提交的小型作业

### 2. Per-Job Mode（单作业模式）

> ⚠️ 从 Flink 1.15 开始已被废弃，推荐使用 Application Mode

```
┌──────────────────┐  ┌──────────────────┐
│  集群 A（Job A）   │  │  集群 B（Job B）   │
│  ┌───────────┐   │  │  ┌───────────┐   │
│  │ JobManager│   │  │  │ JobManager│   │
│  └───────────┘   │  │  └───────────┘   │
│  ┌────┐ ┌────┐   │  │  ┌────┐ ┌────┐   │
│  │ TM │ │ TM │   │  │  │ TM │ │ TM │   │
│  └────┘ └────┘   │  │  └────┘ └────┘   │
└──────────────────┘  └──────────────────┘
```

- **特点**：每提交一个作业，就在资源管理框架（如 YARN）上启动一个**独立的集群**，作业完成后集群自动销毁
- **main() 执行位置**：Client 端
- **优点**：
  - 资源隔离好，作业之间互不影响
  - 集群随作业生命周期自动创建和销毁
- **缺点**：
  - `main()` 在 Client 端执行，当提交大量作业时**客户端压力大**
  - 每次提交都要创建集群，启动开销较大
  - 仅支持 YARN，不支持 Standalone 和 K8s 原生部署
- **适用场景**：生产环境中需要资源隔离的长期运行作业（已废弃，建议迁移到 Application Mode）

### 3. Application Mode（应用模式）⭐ 推荐

```
┌──────────────────┐  ┌──────────────────┐
│  集群 A（App A）   │  │  集群 B（App B）   │
│  ┌───────────┐   │  │  ┌───────────┐   │
│  │ JobManager│   │  │  │ JobManager│   │
│  │ main()在此 │   │  │  │ main()在此 │   │
│  └───────────┘   │  │  └───────────┘   │
│  ┌────┐ ┌────┐   │  │  ┌────┐ ┌────┐   │
│  │ TM │ │ TM │   │  │  │ TM │ │ TM │   │
│  └────┘ └────┘   │  │  └────┘ └────┘   │
└──────────────────┘  └──────────────────┘
```

- **特点**：与 Per-Job 类似，每个应用一个独立集群，但 **main() 方法在 JobManager 上执行**而非 Client 端
- **main() 执行位置**：JobManager 端 ← 这是与 Per-Job 的**关键区别**
- **优点**：
  - 资源隔离好，作业之间互不影响
  - **减轻客户端负担**：main() 在集群内执行，Client 不再承担 JobGraph 生成的计算开销
  - 支持 Standalone、YARN、K8s 等多种部署平台
  - 一个 Application 内可以包含多个 Job（多次调用 `execute()`）
- **缺点**：
  - 每次提交需要启动新集群，有一定启动开销
- **适用场景**：**生产环境首选**，尤其是 K8s 上的部署

## 三、三种模式对比总结

| 对比维度 | Session Mode | Per-Job Mode | Application Mode |
|---------|-------------|-------------|-----------------|
| **集群生命周期** | 预先启动，长期运行 | 随作业创建和销毁 | 随应用创建和销毁 |
| **main() 执行位置** | Client | Client | **JobManager** |
| **资源隔离** | 低（共享集群） | 高（独立集群） | 高（独立集群） |
| **启动速度** | 快（集群已就绪） | 慢（需创建集群） | 慢（需创建集群） |
| **客户端压力** | 大 | 大 | **小** |
| **支持平台** | 全部 | 仅 YARN | 全部 |
| **状态** | 可用 | ⚠️ 已废弃 | ✅ **推荐** |
| **适用场景** | 开发测试 | - | 生产环境 |

## 四、部署平台

以上三种模式可以运行在不同的资源管理平台上：

- **Standalone**：独立部署，手动启动 JobManager 和 TaskManager 进程，适合开发测试
- **YARN**：借助 Hadoop YARN 进行资源管理，适合已有 Hadoop 生态的企业
- **Kubernetes（K8s）**：容器化部署，目前**最主流**的生产部署方式
  - Native K8s：Flink 主动与 K8s API Server 交互，动态申请 Pod
  - K8s Operator：通过 Flink Kubernetes Operator 声明式管理 Flink 集群
- **Mesos**：已逐步淘汰，不推荐使用
