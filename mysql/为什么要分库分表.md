# 为什么要分库分表

> 单机 MySQL 不是"撑不住才拆"，而是"在该拆的时候不拆，后面再拆代价指数级上升"。
> 本文从**性能、容量、并发、运维、可用性**五个维度，系统讲清分库分表的动机，并给出业界权威阈值。

---

## 一、单表数据量过大：B+ 树层高与查询性能

这是**最经典、最硬核**的原因，也是阿里巴巴《Java 开发手册》给出 500 万行阈值的底层依据。

### 1.1 权威阈值

| 来源 | 推荐阈值 | 性质 |
|------|---------|------|
| 阿里巴巴《Java 开发手册》（泰山版） | **单表行数 > 500 万行** 或 **单表容量 > 2GB** | 工程推荐值 |
| InnoDB B+ 树层高拐点 | 约 2000 万行（典型 1KB 行宽） | 性能拐点 |
| MySQL 官方文档（InnoDB Limits） | 单表空间最大 **64 TB**（默认 16KB 页）/ 256 TB（64KB 页） | 物理硬限制 |

阿里手册原文：
> **【推荐】单表行数超过 500 万行或者单表容量超过 2GB，才推荐进行分库分表。**
> 说明：如果预计三年后的数据量根本达不到这个级别，请不要在创建表时就分库分表。

### 1.2 为什么是 500 万 / 2GB —— InnoDB B+ 树原理

InnoDB 使用 B+ 树作为主键索引结构，**每个节点是一个 16KB 的数据页**（`innodb_page_size` 默认 16KB）。

**推导前提（明确假设）**：
- 主键类型 `BIGINT`（8 字节）+ 页内子节点指针（6 字节）= **14 字节/项**
- 单行数据大小约 **1 KB**（典型 OLTP 行：几十个字段的混合宽度）
- 数据页按理想满载估算（实际 InnoDB 即使将 `innodb_fill_factor` 设为 100，**聚簇索引页也会强制预留 1/16 空间**用于后续插入更新，即顺序插入时叶子页最多填充 15/16 ≈ 93.75%；随机插入还会触发页分裂，单页实际容量在 1/2 ~ 15/16 之间。同时页低于 `MERGE_THRESHOLD`（默认 50%）时会触发页合并）

**逐层推算**：
- **非叶子节点**：16384 / 14 ≈ **1170 个子节点指针**
- **叶子节点**：16384 / 1024 ≈ **16 行数据**

| B+ 树层高 | 可容纳行数（约） | 一次主键查询磁盘 IO |
|---------|----------------|------------------|
| 2 层 | 1170 × 16 ≈ **1.87 万行** | 2 次 |
| 3 层 | 1170 × 1170 × 16 ≈ **2190 万行** | 3 次 |
| 4 层 | 1170³ × 16 ≈ **256 亿行** | 4 次 |

**关键结论**：
- 数据量在 **2000 万行以内**时，B+ 树维持 3 层，主键查询稳定在 ~3 次 IO（且根节点常驻 Buffer Pool，实际只需 1~2 次物理 IO）
- 一旦突破，B+ 树变 4 层，**每次查询多一次磁盘 IO**，QPS 下降
- 500 万行是**留有缓冲的保守值**，避免业务增长后逼近临界点

**注意**：上述 2000 万只是基于"单行 1KB"假设的典型值。**实际上限随行宽变化**（以下数字为按相同公式 `1170² × 单页行数` 推算的理论估算，未计入页分裂、填充因子、off-page 溢出等真实损耗）：
- 窄表（如只存 `id + 状态`，行约 100 字节，单页 ≈ 163 行）：3 层理论上 ~2.2 亿行
- 宽表（含 `VARCHAR(2000)` 等大字段，行约 4 KB，单页 ≈ 4 行）：3 层理论上 ~547 万行
- 此外，InnoDB 对超长 `VARCHAR/TEXT/BLOB` 会启用 **off-page 存储**（行内只保留 20 字节指针，剩余数据放到独立溢出页），因此宽表的实际单页行数往往**高于**朴素估算
- 这也是阿里手册同时给出 **"500 万行 或 2GB"** 双阈值的原因——**容量（GB）比行数更本质**，能同时覆盖窄表和宽表

### 1.3 性能下降的连锁反应

单表数据量过大时，问题不只是"主键查询慢一点"：

- **二级索引查询**：回表代价被层高放大
- **范围扫描 / ORDER BY**：扫描页数线性增加，Buffer Pool 命中率下降
- **DDL 操作**：加索引、改字段动辄数小时，业务窗口难以承受
- **统计信息**：`ANALYZE TABLE` 采样不准，导致执行计划走偏
- **备份恢复**：`mysqldump` / xtrabackup 耗时随数据量近似线性增长，TB 级单表的全量备份与恢复窗口显著拉长

## 二、单库容量与硬件瓶颈

即使单表控制得不错，单库总量也会撞墙。

- **磁盘容量**：以阿里云 ESSD 云盘为例，单盘容量上限 **64 TiB**（PL0~PL3 各级别上限均为 65536 GiB）；物理机 SSD 阵列也有限。MySQL 官方文档（*Limits on Table Size*）建议：**单表超过 1 TB 时，将表分散到多个表空间文件**（`partitioning of tables into multiple tablespace files`，注意这里指物理文件拆分，不等同于 `PARTITION BY` 分区表语法）
- **Buffer Pool 命中率**：热数据集超过内存后，IO 抖动剧增
- **CPU**：单实例可用核数有限，复杂 SQL 排队
- **网络带宽**：万兆网卡在大查询/大批量写入下也会成为瓶颈

**分库**的核心动机就是把"单机硬件天花板"横向扩展为"集群硬件总和"。

---

## 三、并发写入瓶颈

MySQL 单实例的写入能力受多重限制：

- **行锁 / 间隙锁竞争**：热点行写入串行化（如秒杀场景下的库存行）
- **redo log + binlog 刷盘**：`innodb_flush_log_at_trx_commit=1` + `sync_binlog=1`（"双 1"配置）下，每次事务提交需分别对 redo log 与 binlog 各做一次 `fsync`。虽然 MySQL 引入了**组提交（Group Commit）**——通过 flush / sync / commit 三阶段队列，将多个并发事务的日志合并为一次 `fsync` 来摊薄开销——但单实例写 TPS 最终仍受限于底层磁盘的 `fsync` 能力，具体数值随硬件（SSD / NVMe）、行宽、并发度差异较大
- **主从复制延迟**：基于 binlog 的复制本质是逻辑回放，即使开启并行复制（MySQL 5.7 的 `slave_parallel_workers` / 8.0 重命名为 `replica_parallel_workers`，配合 `LOGICAL_CLOCK` 模式），高写入压力下从库仍可能追不上

**分库**通过把写入流量打散到 N 个实例，把 TPS 上限从单机级提升到 N 倍。

电商订单、IM 消息、IoT 设备上报等高并发写场景，几乎都走这条路。

---

## 四、连接数与单机资源上限

- MySQL 单实例 `max_connections` **默认仅 151**（MySQL 5.7 / 8.0），生产环境通常调到 1000~5000；继续上调会受限于每连接的内存开销（栈 + sort_buffer + join_buffer 等）和线程切换成本
- 应用集群规模扩大后，**连接池数 × 应用实例数** 很容易撑爆单库
  - 示例：每个应用实例 HikariCP 配 20 个连接，200 个实例就需要 4000 个连接
- 长事务、慢 SQL 占用连接，进一步压缩可用连接

分库后连接被分散到多个实例，单库连接压力线性下降。

---

## 五、可用性与故障爆炸半径

单库单表架构下：

- **一损俱损**：单实例宕机 = 全业务不可用
- **DDL 即停服**：大表加字段需要锁表或长时间 Online DDL，影响全部业务
- **备份/恢复窗口长**：TB 级单库恢复动辄数小时，RTO 不达标

分库分表后：

- 单分片故障只影响 1/N 流量，**爆炸半径可控**
- 可分片滚动升级、灰度发布
- 单分片体量缩小到 1/N，备份与恢复窗口同比例缩短，RTO 显著改善

---

## 六、业务隔离与垂直拆分

除了水平分库分表，**垂直拆分**也是常见动机：

- **垂直分库**：按业务域拆分（订单库、用户库、商品库），避免业务互相影响
- **垂直分表**：把大字段（如 `TEXT`、`BLOB`、商品详情 JSON）拆到副表，减小主表单行字节数，使单个 16KB 数据页能容纳更多行，从而降低 B+ 树层高、提升 Buffer Pool 缓存命中率

垂直拆分往往是水平分库分表的**前置步骤**。

---

## 七、什么时候**不**该分库分表

分库分表是"重武器"，引入后代价巨大：

- 跨分片 JOIN / 排序 / 聚合困难
- 分布式事务复杂度高（如 XA 协议、TCC 模式，或 Seata 等开源框架）
- 需要引入全局唯一 ID 方案（雪花算法、号段模式等，数据库单调自增主键不再可用）
- 跨分片分页（`LIMIT M, N`）几乎无解，需业务降级
- 扩容时数据迁移、双写、灰度切流成本高

**判断准则**：

| 场景 | 建议 |
|------|------|
| 三年内数据量 < 500 万行 | 不要分，做好索引和归档即可 |
| 单表 500 万 ~ 2000 万行 | 优先考虑**冷热分离 + 归档表 + 读写分离** |
| 单表持续增长且 > 2000 万行，或写入 TPS / 连接数已逼近单机硬件能力 | 启动分库分表 |
| 业务域明显独立 | 先做**垂直分库**，再视情况水平分 |

阿里手册的态度很明确：**"预计三年后达不到 500 万，请不要在建表时就分"**。过早分库分表是过度设计。

## 八、分库分表的常见策略（速览）

| 策略 | 适用场景 | 典型实现方式 |
|------|---------|---------|
| **范围分片**（按时间 / ID 区间） | 时序数据、归档场景 | Apache ShardingSphere 范围分片算法、手工分表 |
| **哈希分片**（按 user_id 取模 / 一致性哈希） | 通用 OLTP、订单 / 用户 | Apache ShardingSphere 内置 `MOD` / `HASH_MOD` / 一致性哈希算法 |
| **基因法**（将分片键基因嵌入业务 ID） | 一份数据多维度查询、避免冗余表 | 业务自研 ID 生成 + 自定义分片算法 |
| **垂直分库** | 业务域隔离 | 按 DDD 限界上下文划分库 |
| **分布式数据库**（业务免感知） | 希望保留单库 SQL 语义 | PolarDB-X、OceanBase、TiDB |

> **基因法**说明：将分片键（如 `user_id`）的末 N 位 bit 抽出作为"分片基因"，在生成另一业务 ID（如 `order_id`）时拼接到末 N 位。这样无论用 `user_id % 2^N` 还是 `order_id % 2^N`，都能路由到同一分片，**单条 SQL 即可在两种维度下定位数据，无需冗余的索引表**。
>
> **工具选型说明**：早期常见的 MyCat 社区已基本停止维护，新项目建议直接选用 **Apache ShardingSphere**（Apache 顶级项目，社区活跃），或在重交易、强一致场景下考虑原生分布式数据库（PolarDB-X / OceanBase / TiDB）。

多维度的查询,走搜索引擎。

## 九、总结

分库分表的本质是用**架构复杂度**换取**线性扩展能力**，常见动因可归纳为：

1. **单表数据量瓶颈**：B+ 树层高拐点 ~2000 万行，阿里推荐 500 万行 / 2GB 即启动评估（核心原因）
2. **单库容量瓶颈**：磁盘、内存、CPU 撞硬件天花板
3. **并发写入瓶颈**：单实例 TPS 上限 + 主从复制延迟
4. **连接数瓶颈**：应用集群扩张挤爆单库连接池
5. **可用性瓶颈**：缩小故障爆炸半径、降低 DDL/备份窗口
6. **业务隔离需求**：垂直分库实现业务域解耦

**一句话原则**：
> 不要为了分而分，但当 **500 万行 / 2GB / 单机写入或连接能力上限** 等任一阈值在可预见的未来会触达时，**越早规划，迁移代价越低**。
