# 共识算法（Consensus Algorithm）：从 Paxos 到 Raft

> 面向面试和工程实战的系统笔记。共识算法是分布式系统的理论基石——ZooKeeper、etcd、Kafka Controller、TiKV、CockroachDB 等几乎所有强一致组件的核心都是它。
> 本文以"问题驱动 → 经典算法 → 共同骨架 → 工程对比"的顺序展开。

---

## 一、什么是共识算法

**共识算法**是分布式系统中让一组节点就**某个值**达成一致意见的算法。这里的"值"在不同系统里有不同含义：

- ZooKeeper / etcd 中：这条写日志该不该被提交？
- Raft 集群中：当前任期的 Leader 是谁？
- 数据库主从架构中：哪一行更新应该被持久化？

形式化定义有三个核心属性：

| 属性 | 含义 |
|------|------|
| **Agreement（一致性）** | 所有正常节点最终对同一个值达成一致 |
| **Validity（有效性）** | 达成一致的值必须是某个节点真的提议过的，不能凭空造 |
| **Termination（终止性）** | 算法必须在有限时间内结束，不能死循环 |

通俗讲就是：**一群人投票决定一件事，最终所有人都同意同一个结果，且这个结果是有人提议过的，且大家迟早会得出结论**。

## 二、为什么需要共识算法

### 2.1 分布式系统的三大根本困难

1. **节点会宕机**：随时有机器挂掉、断电、被 kill
2. **网络会丢包 / 延迟**：消息可能丢失、重复、乱序、迟到
3. **无法区分"宕机"和"网络分区"**：你 ping 不通对方，你不知道是它死了还是网断了

在这些故障模型下，靠"广播 + 投票"这种朴素方案是不够的——会出现脑裂、双主、数据不一致等灾难。

### 2.2 故障模型分类

| 模型 | 含义 | 对应算法 |
|------|------|---------|
| **Crash-Stop**（崩溃停止） | 节点要么正常工作要么彻底死掉，不会捣乱 | Paxos / Raft / ZAB |
| **Byzantine**（拜占庭） | 节点可能撒谎、伪造消息（被攻破） | PBFT / PoW / PoS |

工业级分布式存储和协调系统（ZooKeeper、etcd 等）几乎都假设 **Crash-Stop 模型**——因为公司内部网络是可信的，不需要防恶意节点。

### 2.3 FLP 不可能定理（理论限制）

**FLP 定理**（Fischer-Lynch-Paterson, 1985）证明：

> 在**异步网络** + **允许节点崩溃**的模型下，**不存在**一个能同时满足 Agreement、Validity、Termination 的确定性共识算法。

通俗解释：在完全异步的网络里，你永远无法分辨"消息延迟"和"节点宕机"。

工业界的算法都在这个理论限制下做工程妥协：

- **Paxos / Raft**：放宽"异步假设"，引入超时机制（"超过 T 秒没回复就当宕机"）
- **PoW**：放宽"确定性"，引入随机化和概率收敛

## 三、Paxos：理论鼻祖

### 3.1 历史地位

- **作者**：Leslie Lamport，1989 年提出，1998 年正式发表
- **特点**：理论正确性最强，但极难理解和实现
- **代表系统**：Google Chubby、Megastore、Spanner

> Lamport 的原始论文用古希腊议会做比喻，业内调侃 "Paxos 是只有少数人真懂的算法"。

### 3.2 角色与核心思想

Paxos 有三类角色（一个节点可以同时担任多种角色）：

- **Proposer（提议者）**：发起提案
- **Acceptor（接受者）**：投票决定是否接受提案
- **Learner（学习者）**：从 Acceptor 那里学习已被接受的提案

**核心思想**：通过**递增的提案编号** + **多数派投票**，保证最终只有一个值被选中。

### 3.3 两阶段流程

#### 阶段 1：Prepare（准备阶段）

```
Proposer  ──Prepare(n=5)──▶  Acceptor 们
                              │
                              ├─ 如果 n > 之前见过的最大编号
                              │  → 承诺不再接受 n < 5 的提案
                              │  → 返回自己已接受的最高编号提案（如果有）
                              │
                              └─ 否则 → 拒绝
```

#### 阶段 2：Accept（接受阶段）

```
Proposer 收到多数派 Acceptor 的 Prepare 响应后：
  - 如果有 Acceptor 已接受过提案 → 用编号最高那个的值作为本次值
  - 否则 → 用自己原本想提议的值

Proposer  ──Accept(n=5, v=X)──▶  Acceptor 们
                                  │
                                  ├─ 如果没承诺过更高编号 → 接受 (n=5, v=X)
                                  └─ 否则 → 拒绝

多数派接受 → 提案被选中
```

### 3.4 为什么能保证一致

- **多数派交叉**：任意两个多数派必然有交集 → 第二个提案一定能"看到"第一个提案的值
- **编号递增**：旧提案被新提案天然压制
- **值锁定**：一旦某个值被多数派接受，后续所有提案都会用这个值（通过 Prepare 阶段的"返回最高编号"机制）

### 3.5 Paxos 的痛点

- **概念抽象**：Proposer/Acceptor/Learner 三个角色 + 编号 + 多数派交叉，新人很难一遍看懂
- **Multi-Paxos 没有正式定义**：原始 Paxos 只解决"对一个值达成共识"，工程上要连续提交日志（Multi-Paxos）的细节论文里语焉不详
- **活锁**：两个 Proposer 互相用更高编号 Prepare 抢占，可能永远走不到 Accept

这就是为什么 Raft 出现并迅速取代 Paxos 成为工业标准。

## 四、Raft：工程友好的标准答案

### 4.1 历史地位

- **作者**：斯坦福 Diego Ongaro 和 John Ousterhout，2014 年发表
- **设计目标**：**为可理解性而设计**——让普通工程师也能看懂、能写出正确实现
- **代表系统**：**etcd、Consul、TiKV、CockroachDB、Kafka KRaft**

> Raft 的正确性等价于 Paxos，但实现复杂度低一个数量级，是当前事实标准。强烈建议直接学 Raft，跳过 Paxos。

### 4.2 三种角色

任意时刻每个节点处于以下三种角色之一：

```
              选举超时
              ┌─────────┐
              ▼         │
        ┌─────────┐  ┌─────────┐  ┌─────────┐
        │ Follower│─▶│Candidate│─▶│ Leader  │
        └─────────┘  └─────────┘  └─────────┘
              ▲         │              │
              │         │              │
              └─────────┴──────────────┘
                  发现更高任期 / 落败
```

| 角色 | 职责 |
|------|------|
| **Follower** | 被动响应 Leader 和 Candidate 的请求，不主动发起请求 |
| **Candidate** | 选举期间的中间状态，主动拉票 |
| **Leader** | 处理所有客户端写请求，复制日志给 Follower，定期发心跳 |

**单 Leader 设计**：所有写都走 Leader，避免了 Paxos 那种多个 Proposer 互相冲突的复杂性。

### 4.3 任期（Term）：逻辑时钟

Raft 把时间切成连续的**任期**，每个任期最多有一个 Leader：

```
   Term 1        Term 2          Term 3      Term 4
┌──────────┬──────────────┬──────────────┬──────────┐
│  选举     |    Leader 任期        │ 无 Leader │ Leader 任期 │
│ + Leader │                      │ (选举失败)│            │
└──────────┴──────────────┴──────────────┴──────────┘
```

**任期号的作用**：

- 每个 RPC 都带 term，节点收到更小 term 的请求 → 拒绝并返回自己的 term
- 节点收到更大 term 的请求 → 立即降级为 Follower，更新自己的 term
- 这是 Raft 中"检测过期 Leader 并自动让位"的核心机制

### 4.4 Leader 选举

#### 触发条件

Follower **选举超时**（randomized 150~300ms）内没收到 Leader 心跳，就发起选举。

#### 流程

```
1. Follower 把自己 term + 1，转为 Candidate，给自己投一票
2. 并行向所有其他节点发送 RequestVote RPC
3. 收到响应后分三种情况：
   ① 拿到多数派的票 → 成为 Leader，立即发心跳
   ② 收到其他节点的心跳，且 term ≥ 自己 → 退回 Follower
   ③ 选举超时仍未决出 → term + 1，重新发起选举
```

#### 投票规则（保证安全性的关键）

Follower 收到 RequestVote 时只在以下条件**全部满足**才投票：

- 请求的 term ≥ 自己的 term
- 本任期还没投过票
- **Candidate 的日志至少和自己一样新**（关键！见 4.6）

#### 为什么用随机超时

如果所有 Follower 用同一个超时时间，Leader 挂了之后大家会**同时**发起选举，互相分票导致选不出来。**随机化超时（150~300ms 内随机）**让某个节点先超时、先抢到多数派票，避免活锁。

### 4.5 日志复制

#### 数据结构

每个节点维护一份日志数组，每个条目格式：

```
┌────────┬────────┬─────────────────┐
│  term  │ index  │   command       │
└────────┴────────┴─────────────────┘
   3        7       SET x = 5
```

#### 写入流程

```
客户端写请求
     │
     ▼
   Leader
     │
     ├─ 1. 在本地日志追加一条 entry（未提交）
     │
     ├─ 2. 并行向所有 Follower 发 AppendEntries RPC
     │
     ├─ 3. 收到多数派 ACK 后，标记这条 entry 为"已提交"
     │     → 应用到状态机 → 返回客户端成功
     │
     └─ 4. 后续心跳告诉 Follower "提交点已推进到 index=N"
           → Follower 也把 ≤ N 的日志应用到状态机
```

**关键点**：写入只要**多数派持久化**就返回成功，不需要全部节点 ACK。这是高可用 + 强一致的平衡点。

#### 日志一致性保证

Raft 保证：**如果两个日志在某个 index 处的 term 相同，那么从开头到这个 index 的所有日志都相同**。

这个性质通过 **AppendEntries 的一致性检查**实现：每次 RPC 都附带 `prevLogIndex` 和 `prevLogTerm`，Follower 校验自己这个位置的 term 是否匹配，不匹配就拒绝，Leader 退一格重试，直到找到匹配点后用自己的日志覆盖 Follower 后面的部分。

### 4.6 安全性：Election Restriction（选举限制）

**问题**：如果选出的新 Leader 没有最新已提交的日志，会导致已提交日志被覆盖丢失。

**解决**：Raft 规定 Candidate 拉票时必须证明**自己的日志至少和大多数节点一样新**：

- 比较"日志最后一条 entry 的 (term, index)"
- 先比 term，term 大的更新；term 相同时 index 大的更新

由于已提交的日志一定在多数派上，多数派至少有一个节点持有该日志，根据投票规则，**没有最新已提交日志的 Candidate 拿不到多数派的票**——保证当选 Leader 一定包含所有已提交日志。

### 4.7 一个完整的写入示例

假设 5 节点集群（N1~N5），N1 是 Leader，term=2：

```
T0  客户端发 SET x=5 给 N1
T1  N1 在 index=10 追加 entry (term=2, x=5)，未提交
T2  N1 并行发 AppendEntries 给 N2~N5
T3  N2、N3 持久化成功并 ACK（多数派 = 3 个节点：N1、N2、N3）
T4  N1 标记 index=10 已提交，应用 x=5 到状态机
T5  N1 返回客户端 OK
T6  N1 下一次心跳带 commitIndex=10
T7  N2~N5 把 index ≤ 10 的日志应用到状态机
```

整个过程的耗时 ≈ 一次磁盘写 + 一次网络 RTT，通常 10ms 量级。

## 五、ZAB：ZooKeeper 的共识协议

### 5.1 历史地位

- **ZAB**（ZooKeeper Atomic Broadcast）是 ZooKeeper 专用的共识协议
- **比 Raft 早提出**（ZooKeeper 2008 年开源，Raft 2014 年发表）
- **本质和 Raft 极其相似**：单 Leader、多数派写入、任期号、日志复制——大体思路一致

### 5.2 三种角色

| 角色 | 职责 | 对应 Raft |
|------|------|----------|
| **Leader** | 处理写请求，广播提案 | Leader |
| **Follower** | 响应 Leader、参与选举和投票、处理读请求 | Follower |
| **Observer** | 只同步数据不参与投票，用于扩展读吞吐 | Raft 中的 Learner（非投票成员） |

### 5.3 三大阶段

ZAB 把整个生命周期分成三个阶段，循环切换：

```
       崩溃 / 网络分区
   ┌────────────────────┐
   ▼                    │
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 选主阶段 │─▶│ 同步阶段 │─▶│ 广播阶段 │
│Election  │  │Recovery  │  │Broadcast │
└──────────┘  └──────────┘  └──────────┘
```

#### 阶段 1：Leader Election（选主）

集群启动时或 Leader 失联时触发，目前 ZooKeeper 默认用 **FastLeaderElection** 算法：

- 每个节点广播 `(myid, zxid)`：`myid` 是节点 ID，`zxid` 是自己见过的最大事务 ID
- 投票规则：**先比 zxid，大的胜；zxid 相同时 myid 大的胜**
- 收到多数派认同的票 → 成为准 Leader

> **为什么先比 zxid**：和 Raft 的"日志至少和大多数一样新"原则等价——保证新 Leader 一定有最新已提交的事务，不会丢数据。

#### 阶段 2：Discovery + Synchronization（发现 + 同步）

新 Leader 必须先把数据同步到 Follower 才能对外提供服务：

- Follower 把自己的 `lastZxid` 上报给 Leader
- Leader 根据差距决定同步策略：
  - **DIFF**：差距小，发送差异事务
  - **TRUNC**：Follower 有 Leader 没有的事务（旧 Leader 的"幽灵日志"），让 Follower 截断
  - **SNAP**：差距太大，直接发完整快照

#### 阶段 3：Broadcast（广播）

正常运行期，处理客户端写请求，是一个**两阶段提交**变种：

```
T0  客户端写请求 → Leader
T1  Leader 在本地生成事务 (zxid, data)，写本地日志
T2  Leader 广播 PROPOSAL 给所有 Follower
T3  Follower 持久化日志，回 ACK
T4  Leader 收到多数派 ACK 后广播 COMMIT
T5  Leader 和所有 Follower 应用事务到状态机（ZooKeeper 的内存数据树）
T6  Leader 返回客户端 OK
```

### 5.4 zxid：ZAB 的核心数据结构

zxid 是 64 位整数，分成两部分：

```
┌─────────────────┬──────────────────────┐
│  epoch (高 32)  │   counter (低 32)    │
└─────────────────┴──────────────────────┘
   选举轮次            本轮次内的事务序号
```

- **epoch**：每次选出新 Leader 时 +1，对应 Raft 的 term
- **counter**：本任期内事务编号，每次提交事务 +1

通过这套设计，**zxid 在全局严格递增、唯一**，比较两个 zxid 大小直接对应"事件先后顺序"。

### 5.5 ZAB vs Raft

| 维度 | ZAB | Raft |
|------|-----|------|
| 提出时间 | 2008 年（ZooKeeper） | 2014 年 |
| 设计目标 | 服务于 ZooKeeper 这个具体系统 | 通用、可理解、可教学 |
| 任期号 | epoch（zxid 高 32 位） | term |
| 日志匹配 | 通过 zxid 比对 + DIFF/TRUNC/SNAP | prevLogIndex + prevLogTerm 一致性检查 |
| 读请求 | Follower 可直接服务（弱一致） | 默认走 Leader（强一致） |
| 实现复杂度 | 中（角色多、阶段切换） | 低（论文目标就是简单） |

> **结论**：ZAB 和 Raft 是**同一类算法**——单 Leader + 多数派 + 任期号。差别在工程细节上，安全性证明的内核是一致的。

## 六、共识算法的共同骨架

把 Paxos、Raft、ZAB 抽象后，会发现它们共享一套核心套路。理解这套骨架就理解了 90% 的共识算法。

### 6.1 五个核心元素

| 元素 | 作用 | 在各算法中的体现 |
|------|------|----------------|
| **Quorum（多数派）** | 写入只要多数派持久化即可提交，容忍少数派故障 | 全部算法的根基 |
| **逻辑时钟** | 全局唯一递增的"轮次号"，用于检测过期和排序 | Paxos 的提案编号 / Raft 的 term / ZAB 的 epoch |
| **单一权威** | 同一时刻只有一个节点能"做决定"，避免分歧 | Paxos 的 Proposer / Raft 和 ZAB 的 Leader |
| **日志复制** | 通过追加日志 + 多数派确认实现状态机一致 | Multi-Paxos / Raft 日志 / ZAB 事务流 |
| **安全切换** | Leader 宕机后选出的新 Leader 必须含全部已提交日志 | Raft 的选举限制 / ZAB 的 zxid 比较 |

### 6.2 多数派为什么是 N/2 + 1

设集群有 N 个节点，多数派定义为 **⌊N/2⌋ + 1**：

| N | 多数派 | 容忍故障数 |
|---|-------|-----------|
| 3 | 2     | 1         |
| 5 | 3     | 2         |
| 7 | 4     | 3         |

**核心数学性质**：**任意两个多数派必有交集**。

```
N=5 集群，假设：
  写操作 1：被 {A, B, C} 多数派接受
  写操作 2：被 {C, D, E} 多数派接受

→ 交集 = {C}
→ "操作 2 的多数派"里一定有节点知道"操作 1"
```

这就是为什么共识算法在网络分区、节点宕机时仍能保证一致性——**新写入的多数派必然包含至少一个见过历史已提交日志的节点**。

### 6.3 为什么用奇数节点

| 集群规模 | 多数派 | 容忍故障 | 评价 |
|---------|-------|---------|------|
| 3 | 2 | 1 | ✅ 推荐：成本低，容忍 1 故障 |
| 4 | 3 | 1 | ❌ 浪费：多 1 个节点没多容忍能力 |
| 5 | 3 | 2 | ✅ 推荐：核心生产，容忍 2 故障 |
| 6 | 4 | 2 | ❌ 浪费 |
| 7 | 4 | 3 | ✅ 大型集群 |

> 偶数节点既不增加容错能力，又增大了写入延迟（要等更多 ACK），还增大了出现"刚好对半分"的脑裂风险。**永远部署奇数节点**。

### 6.4 共识算法解决的问题边界

**能解决**：

- 在 **Crash-Stop 故障模型** 下，让多个节点对一系列值（事务日志）达成强一致
- 自动检测和处理 Leader 宕机、网络抖动
- 保证已提交的数据不丢失

**不能解决**：

- **拜占庭故障**（节点撒谎、伪造消息）→ 需要 PBFT 等算法
- **网络完全分区且少数派需要服务** → CAP 定理决定了在分区时只能选 C 或 A，共识算法选 C
- **业务逻辑层面的一致性**（例如转账要扣 A 加 B 这种业务事务）→ 需要在共识层之上做应用逻辑

## 七、共识算法 vs Redis 分布式锁

理解了共识算法后，回头看 Redis 分布式锁就能看清它的"理论缺陷"到底在哪。

### 7.1 Redis 主从架构没有走共识

Redis 主从是**异步复制**的：主节点写入后**立即返回成功**，不等从节点确认。

```
T0  客户端 A 在主节点 M 上 SET 锁成功，M 返回 OK
T1  M 还没把这条 SET 同步给从节点 S 就宕机了
T2  哨兵把 S 提升为新主——新主上根本没这把锁
T3  客户端 B 来抢锁，成功
    → A 和 B 同时持有"同一把锁"
```

**根因**：Redis 把"被多数派持久化"省略了——它优先要的是**性能和可用性**（AP），而不是一致性。

### 7.2 ZooKeeper / etcd 怎么避免这个问题

ZooKeeper（ZAB）和 etcd（Raft）写入时强制走完整的共识流程：

```
T0  客户端 A 请求创建锁节点
T1  Leader 把这个写请求广播给所有 Follower
T2  Follower 持久化日志后回 ACK
T3  Leader 收到多数派（>N/2）ACK 才算提交
T4  Leader 返回成功给客户端 A
```

关键差异：**写请求必须经过多数派持久化才返回成功**。这意味着——

- 即便 Leader 立即宕机，**多数派节点上一定还有这条日志**
- 新选出的 Leader 必须包含全部已提交的日志（选举限制保证）
- **客户端看到的"加锁成功"必然不会丢失**

代价是性能：

| 维度 | Redis | ZooKeeper / etcd |
|------|-------|------------------|
| 写入路径 | 内存 + 异步复制 | **多数派持久化 + 落盘** |
| 单次加锁延迟 | < 1ms | 10ms+ |
| TPS | 几万 ~ 十万 | 几千 |
| 一致性 | AP（最终一致） | **CP（强一致）** |

### 7.3 Redlock 算法：Redis 的"伪共识"尝试

Antirez 为缓解 Redis 主从切换丢锁问题，提出了 **Redlock 算法**：部署 N（通常 5）个**完全独立的主节点**（注意：是各自独立，不是主从），客户端依次向它们 `SET NX EX`，**只有在 N/2+1 个节点上都成功**且总耗时小于 TTL，才算加锁成功。

Redlock 的本质：**把"多数派"思想搬到客户端协调**——但它仍然不是真正的共识算法，因为它没有 Leader 选举、没有日志复制、没有任期号。Martin Kleppmann 在 [How to do distributed locking](https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html) 中指出 Redlock 在 GC 停顿和时钟漂移场景下并不安全，至今争议未平息。

### 7.4 选型决策树

```
需要分布式锁？
   │
   ├─ 业务要求强一致（金额、库存绝对不超卖）？
   │     └─ ✅ 用 ZooKeeper / etcd（基于真共识算法）
   │
   ├─ 业务能容忍极小概率失效，能用幂等兜底？
   │     └─ ✅ 用 Redis + Redisson 看门狗（性能优先）
   │
   └─ 业务在区块链 / 不可信节点环境？
         └─ ✅ 用 PBFT / 拜占庭容错算法
```

> **一句话总结**：Redis 分布式锁不安全的根源是它没走共识。要强一致就得付出 ZK/etcd 的性能代价；要性能就得在业务层做幂等兜底。

## 八、CAP 视角下的总结

### 8.1 CAP 定理回顾

CAP 定理（Eric Brewer, 2000）说：**一致性（Consistency）、可用性（Availability）、分区容忍性（Partition Tolerance）三者不可兼得**，分布式系统在网络分区发生时必须从中选两个。

| 字母 | 含义 |
|------|------|
| **C** | 所有节点同一时刻看到的数据一致 |
| **A** | 每个请求都能在有限时间内得到非错响应 |
| **P** | 网络分区故障下系统仍能继续工作 |

由于网络分区在分布式系统中**不可避免**（机房断网、跨机房延迟、丢包），P 是必选项。**真正的取舍发生在 C 和 A 之间**。

### 8.2 共识算法的位置：CP 阵营

| 系统 | 类别 | 体现 |
|------|------|------|
| **ZooKeeper** | CP | 多数派写入才返回，分区时少数派直接拒绝服务 |
| **etcd** | CP | 同上 |
| **Consul** | CP | Raft 实现 |
| **HBase** | CP | 强一致读写，单 Region 分区时不可用 |
| **Redis 主从** | AP | 异步复制，分区时主可继续服务，可能丢数据 |
| **Cassandra** | AP（可调） | 默认 AP，可通过 quorum 配置接近 CP |
| **DynamoDB** | AP | 最终一致，高可用 |

**共识算法是 CP 系统的实现基础**——它通过"多数派确认"保证强一致性，代价是分区时少数派会暂时不可用（拒绝服务直到分区恢复或重新选主）。

### 8.3 一个具体例子：5 节点集群被分成 3+2

```
分区前：N1(Leader), N2, N3, N4, N5
分区后：{N1, N2, N3} | {N4, N5}
```

- **多数派分区 {N1, N2, N3}**：仍有 3 个节点 ≥ 多数派门槛，可以继续选主和写入 ✅
- **少数派分区 {N4, N5}**：只有 2 个节点，无法形成多数派 → 即便 N4 想当 Leader 也拿不到多数票 → **拒绝写入** ❌

这就是 CP 的真实表现：**牺牲少数派的可用性，换取整个集群永远不出现两个 Leader、永远不丢已提交数据**。

### 8.4 共识算法不能违反物理定律

值得强调：共识算法**不是魔法**——它通过工程手段在故障发生时做出"正确选择"，但不能让你同时拥有所有美好特性：

- 不能让分区下少数派也写入（违反 CAP）
- 不能让强一致读写做到 Redis 那种亚毫秒级延迟（违反物理）
- 不能在网络全断时还能服务（违反 P）

理解 CAP 是理解共识算法的代价的前提。

## 九、面试速答模板

### Q1：什么是共识算法？

**答**：共识算法是分布式系统中让一组节点对某个值达成一致意见的算法，需要满足三个属性：所有正常节点最终一致（Agreement）、达成一致的值是有人提议过的（Validity）、有限时间内结束（Termination）。它解决的是节点宕机、网络丢包、分区的故障下如何保证多副本一致这一根本问题。代表算法是 Paxos、Raft、ZAB。

### Q2：Raft 的核心机制？

**答**：Raft 通过**单 Leader + 任期号 + 日志复制 + 多数派确认**来达成共识：

1. **Leader 选举**：Follower 心跳超时后转为 Candidate 发起选举，拿到多数派票成为 Leader
2. **日志复制**：所有写请求由 Leader 接收并追加日志，并行复制给 Follower，**多数派持久化即可提交**
3. **安全性**：通过"Candidate 必须有最新日志才能当选"的选举限制，保证已提交日志永不丢失

### Q3：为什么共识算法要用多数派而不是全部节点确认？

**答**：

- **可用性**：要求全部节点确认意味着任何一个节点宕机都会让写入卡住，可用性极差
- **正确性足够**：任意两个多数派必有交集，保证新写入的多数派一定包含至少一个见过历史已提交日志的节点，不会丢数据
- **数学保证**：N 节点集群可容忍 ⌊(N-1)/2⌋ 个故障节点

### Q4：Raft 和 Paxos 的关系？

**答**：

- **Paxos** 是共识算法的理论鼻祖，由 Lamport 1989 年提出，正确性最强但极难理解和实现
- **Raft** 是 2014 年提出的工程化简化版，**正确性等价于 Paxos** 但实现复杂度低一个数量级
- Raft 的核心简化是**单 Leader 设计**——所有写都走 Leader，避免了 Paxos 多 Proposer 的复杂性
- 现代主流系统（etcd、Consul、TiKV、CockroachDB）几乎全部用 Raft

### Q5：ZAB 和 Raft 有什么区别？

**答**：本质是**同一类算法**——都是单 Leader + 多数派 + 任期号。差别在工程细节：

- ZAB 是 ZooKeeper 专用，Raft 是通用算法
- ZAB 用 zxid（高 32 位 epoch + 低 32 位 counter），Raft 用 term + log index
- ZAB 比 Raft 早提出 6 年
- Raft 论文以"可理解性"为核心目标，文档和实现门槛更低

### Q6：共识算法和 CAP 是什么关系？

**答**：共识算法是 CP 系统的实现基础。在网络分区场景下，共识算法**主动拒绝少数派的写入**，确保整个集群对外只呈现一致的数据视图——牺牲了少数派的可用性，换取了强一致性。AP 系统（如 Redis、Cassandra）则不走共识，分区时双方都能读写，但可能产生数据冲突需要后续合并。

### Q7：为什么 Redis 分布式锁理论上不安全？

**答**：因为 Redis 主从是异步复制的，**没有走共识**——主节点写入后立即返回成功，不等多数派 ACK。如果主节点在同步前宕机，新主上的数据会丢失，导致两个客户端可能同时持锁。强一致场景应使用基于真共识算法的 ZooKeeper 或 etcd。Redis 的优势是性能（亚毫秒级 vs 10ms+）和实现简单，99% 的业务场景配合业务幂等兜底就够用。

### Q8：为什么集群一般部署奇数节点？

**答**：

- N=3 多数派 2，容忍 1 故障；N=4 多数派 3，仍只能容忍 1 故障——多 1 个节点没多容忍能力且增加写入延迟
- 偶数节点在网络对半分区时更容易出现两边都凑不齐多数派的"双不可用"
- 业界标准：测试环境 3 节点，生产 5 节点，超大集群 7 节点

### Q9：Leader 宕机后如何保证不丢数据？

**答**：通过**选举限制**——Candidate 拉票时必须证明"自己的日志至少和大多数节点一样新"。由于已提交的日志一定在多数派上，多数派至少有一个节点持有该日志，根据投票规则没有最新日志的 Candidate 拿不到多数派的票。**保证当选 Leader 一定包含所有已提交日志**，从而已提交数据永不丢失。这是 Raft 安全性的核心证明。
