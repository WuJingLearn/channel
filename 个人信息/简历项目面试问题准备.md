# 简历项目面试问题准备

## 使用方式

这份文档用于准备简历中项目相关的面试追问。建议准备时遵循下面思路：

- **先讲背景**：为什么做这个项目，解决什么问题。
- **再讲方案**：你负责哪部分，系统链路怎么设计。
- **重点讲难点**：稳定性、幂等、一致性、扩展性、权限、安全、性能。
- **最后讲结果**：效率提升、成本下降、稳定性保障、经验沉淀。

面试时不要把答案背成流水账，要围绕“问题是什么、我怎么设计、为什么这么做、有什么取舍”来回答。

## 一、应用监控与错误日志智能查询助手

### 1. 你这个项目是做什么的？

**回答思路：**

这个项目是面向研发排障场景的智能查询助手。研发同学平时排查问题时，需要在监控平台、日志平台、应用平台之间来回切换，查询成本比较高。我们希望通过自然语言交互，让用户直接描述问题，比如“帮我看一下某个应用昨晚错误量有没有异常”，系统自动识别应用名、时间范围、指标类型或日志关键词，然后调用后端工具查询真实监控和日志数据，最后给出结构化摘要。

它不是简单 ChatBot，而是 **大模型 + MCP 工具调用 + 可观测数据查询** 的后端系统。

### 2. 它和普通 ChatBot 有什么区别？

**回答思路：**

普通 ChatBot 更多依赖模型直接回答，容易出现幻觉；这个项目的核心是工具调用。

- **事实来源不同**：答案来自监控平台和日志平台的真实数据，而不是模型凭空生成。
- **执行方式不同**：模型负责理解意图、抽取参数、选择工具，后端负责真实查询。
- **可控性要求更高**：监控和日志查询涉及权限、查询范围、成本控制，需要服务端做参数校验和权限控制。

所以它更像一个智能化后端工具平台，而不是简单套壳大模型。

### 3. MCP 在项目里解决了什么问题？

**回答思路：**

MCP 主要解决大模型和外部工具之间的标准化连接问题。我们把监控查询、日志检索、异常聚合等能力封装成标准化工具，每个工具包含：

- 工具名称
- 工具描述
- 参数 Schema
- 参数校验规则
- 权限约束
- 调用实现
- 结果格式化逻辑

这样后续要接入新的内部平台时，只需要新增工具定义，不需要重写主流程。模型根据工具描述选择合适工具，后端统一做安全校验、执行和结果处理。

### 4. 大模型生成的参数不准确怎么办？

**回答思路：**

不能完全信任模型生成的参数，我会做三层控制：

- **Schema 约束**：工具定义里明确参数类型、必填项、枚举值和描述。
- **服务端二次校验**：校验应用名、环境、时间范围、指标名、日志关键词是否合法。
- **多轮对话补全**：如果缺少应用名、环境、时间范围等关键参数，不直接查询，而是反问用户补充。

比如用户说“帮我看下昨天有没有异常”，系统会追问“请提供要查询的应用名称”。

### 5. 如何避免大模型幻觉？

**回答思路：**

主要通过“工具结果驱动回答”来避免幻觉：

- 模型不能直接编造监控和日志结论。
- Prompt 中要求模型只能基于工具返回结果回答。
- 工具无结果时，明确提示无法判断，而不是猜测原因。
- 回答中尽量带上数据依据，比如异常时间段、错误量、Top 异常类型。

本质上是让模型做交互层和总结层，不让模型成为事实来源。

### 6. 日志结果很多时怎么处理？

**回答思路：**

日志不能原样全部丢给模型，需要先降噪和结构化：

- 查询阶段限制时间范围、返回条数和关键词。
- 聚合阶段按异常类型、错误码、接口、时间窗口分组。
- 摘要阶段提取 Top 异常、首次出现时间、最近出现时间、样例日志。
- 用户需要细节时，再支持分页或继续追问。

这样模型处理的是结构化摘要，而不是海量原始日志。

### 7. 这个项目最大的难点是什么？

**回答思路：**

最大的难点是大模型输出不稳定，但监控日志查询又要求准确、可控。如果完全相信模型参数，可能出现查错应用、查询范围过大、越权查询、高成本查询等问题。

所以我重点做了工具调用链路的可控性设计：参数 Schema、服务端校验、权限控制、查询范围限制、异常兜底、结果截断和格式化。这个项目让我意识到，AI 应用落地不是简单调用模型 API，而是要把模型放在一个受控的工程系统里。

## 二、集团 JDK11 升级 RAG 答疑助手

### 1. 这个项目是做什么的？

**回答思路：**

这个项目是面向集团应用 JDK11 升级场景的 RAG 答疑助手。JDK8 升级到 JDK11 时，研发同学会遇到依赖兼容、构建失败、运行时异常、JVM 参数变化、非法反射访问等问题。我们把升级文档、历史答疑记录、FAQ 和典型案例沉淀成知识库，用户通过自然语言提问后，系统先检索相关资料，再由模型基于资料生成答案。

核心目标是减少重复答疑，让研发同学可以自助定位升级问题。

### 2. 为什么用 RAG，不直接用大模型？

**回答思路：**

JDK11 升级问题有很强的内部环境依赖，比如内部框架版本、构建插件、中间件适配情况，这些内容通用大模型并不知道。直接让模型回答容易出现两个问题：

- 答案太通用，不符合公司内部技术栈。
- 模型可能生成看似合理但实际不可执行的建议。

所以用 RAG，把内部文档和历史案例作为事实来源。模型负责理解问题和组织答案，具体解决方案来自检索到的知识内容。

### 3. 文档怎么切分？

**回答思路：**

不会简单按固定长度切，而是尽量按问题单元切分。对于 FAQ 或案例类文档，一个完整问题通常包含：

- 问题现象
- 错误堆栈
- 原因分析
- 解决方案
- 适用范围

这些信息最好保持在同一个 chunk 里，否则可能只召回错误现象，没有解决方案。对于长文档，可以先按标题层级切，再控制 chunk 长度，并补充元数据，比如问题类型、JDK 版本、框架名、错误关键词等。

### 4. 如何提升召回准确率？

**回答思路：**

主要从五个方面优化：

- **文档清洗**：去掉无关内容，保留错误现象、关键词和解决方案。
- **结构化切分**：按问题单元切分，保证 chunk 语义完整。
- **关键词增强**：异常类名、错误码、依赖名、插件名非常关键，要结合关键词匹配。
- **元数据过滤**：根据 JDK 版本、框架类型、问题分类过滤候选内容。
- **Prompt 约束**：要求模型只基于召回内容回答，召回不足时提示用户补充信息。

### 5. 如何减少 RAG 幻觉？

**回答思路：**

主要做三件事：

- Prompt 约束模型必须基于召回内容回答。
- 答案中带引用来源，方便用户查看原始资料。
- 当召回内容不足时，提示“当前知识库没有找到明确解决方案”，而不是强行回答。

对于 JDK 升级这种工程治理场景，可解释性很重要，不能只给一个结论。

### 6. JDK8 升级 JDK11 常见问题有哪些？

**回答思路：**

常见问题可以分几类：

- **模块移除**：JDK11 移除了部分 Java EE 模块，比如 JAXB、JAX-WS、javax.annotation，需要显式引入依赖。
- **非法反射访问**：JDK9 之后模块系统增强封装性，老框架反射访问 JDK 内部类可能出现 warning 或失败。
- **JVM 参数变化**：CMS、PermGen、GC 日志等部分参数废弃或不再支持。
- **依赖兼容问题**：老版本 Spring、Dubbo、Netty、ASM、CGLIB 等可能不兼容 JDK11。
- **构建插件问题**：Maven Compiler Plugin、Surefire Plugin 等版本过低时可能不支持 JDK11。
- **运行时行为差异**：TLS 默认版本、字符集、类加载行为等可能变化。

### 7. 怎么评估 RAG 效果？

**回答思路：**

可以从召回和答案两个层面评估：

- **召回层面**：TopK 结果里是否包含正确文档，是否召回了无关内容。
- **答案层面**：答案是否解决问题，是否有明确步骤，是否引用来源，是否出现幻觉。

实际可以沉淀一批典型升级问题作为测试集，每次调整切分策略、Embedding 模型或 Prompt 后，用测试集做回归验证。

## 三、淘特小游戏能量 PK 赛后端系统

### 1. 这个项目是做什么的？

**回答思路：**

这是淘特小游戏活动场景的后端项目。用户通过参与斗地主等小游戏完成任务，获得活动积分，并参与排行榜排名。后端主要负责活动任务配置、用户任务进度记录、积分领取、积分发放、排行榜查询和异常补偿等核心链路。

这个项目虽然业务规模不算特别复杂，但涉及活动系统里比较典型的问题：任务状态流转、积分发放幂等、外部系统调用异常处理、排行榜查询性能和数据一致性。

### 2. 积分领取如何保证幂等？

**回答思路：**

我会从业务幂等和数据幂等两层保证。

业务层面，每次领取对应唯一业务单号，比如活动 ID + 用户 ID + 任务 ID。用户重复点击时，业务单号相同。

数据层面，在领取记录表上建立唯一约束，比如 activity_id + user_id + task_id。请求进来后先尝试创建领取记录，只有创建成功的请求可以继续发放积分。如果唯一约束冲突，说明已经领取过，直接返回已有结果。

这样即使前端重复请求、接口重试或者网络超时，也不会重复发放积分。

### 3. 调用积分系统超时怎么办？

**回答思路：**

调用积分系统超时不能简单认为失败，因为积分系统可能已经发放成功，只是响应超时。

我会把这种情况标记为“处理中”或“结果未知”，然后通过业务流水号查询积分系统的发放结果。如果确认成功，就更新为已领取；如果确认失败，再进入重试或补偿流程。

关键点是重试时业务流水号必须保持一致，不能每次生成新的流水号，否则可能重复发放。

### 4. 任务状态怎么设计？

**回答思路：**

可以把任务流转抽象成状态机：

```text
未开始 -> 进行中 -> 已完成 -> 待领取 -> 已领取
                          -> 发放失败 -> 待补偿 -> 已领取
```

每次状态变更都要校验前置状态，不能随意更新。比如只有已完成或待领取状态才能发起领取，已领取状态不能再次领取。这样可以避免未完成任务直接领奖、重复领奖、发放失败后状态不一致等问题。

### 5. 排行榜怎么实现？

**回答思路：**

如果排行榜实时性要求比较高，可以使用 Redis ZSet：

- member 存用户 ID。
- score 存用户积分。
- 用户积分变化时更新 ZSet。
- 查询 TopN 用 ZREVRANGE。
- 查询用户排名用 ZREVRANK。
- 查询用户积分用 ZSCORE。

活动结束后的最终榜单可以异步落库，最终结算以数据库或离线核算结果为准。

### 6. Redis 排行榜和数据库积分不一致怎么办？

**回答思路：**

数据库应该作为最终事实来源，Redis 用于查询加速。

积分发放成功后，先更新数据库积分流水和用户积分，再更新 Redis 排行榜。如果 Redis 更新失败，可以通过异步任务或定时任务根据数据库积分重建排行榜。

对于最终发奖或结算，不应该完全依赖 Redis，而是以数据库或离线核算结果为准。

### 7. 活动系统如何防刷？

**回答思路：**

可以从几方面做：

- 资格校验：判断用户是否满足活动参与条件。
- 任务完成校验：任务完成事件必须来自可信业务系统，而不是前端直接上报。
- 领取频控：对用户维度做接口限流，避免频繁点击。
- 幂等控制：防止重复领取。
- 异常数据识别：识别短时间异常高积分、异常任务完成频次。
- 风控接入：如果奖励价值较高，可以接入风控系统。

## 四、内部 RPC 框架服务治理能力扩展

### 1. RPC 服务暴露流程是什么？

**回答思路：**

RPC 服务暴露大致分为几个步骤：

1. 业务方实现服务接口。
2. 通过注解或配置声明这是一个 RPC 服务。
3. 框架启动时扫描服务 Bean，解析接口、版本、分组等元数据。
4. 将服务实例注册到本地服务容器中，用于后续请求分发。
5. 启动网络服务器，监听端口。
6. 将服务元数据注册到注册中心。
7. 消费方从注册中心发现服务地址，通过代理发起远程调用。

核心是两件事：本地能根据请求找到服务实现，注册中心能让消费者发现服务。

### 2. 注解式 RPC 服务暴露怎么实现？

**回答思路：**

可以定义一个服务暴露注解，包含接口类型、版本号、分组等属性。应用启动过程中，框架扫描 Spring 容器中的 Bean，找到带有该注解的 Bean，解析注解元数据，构造服务定义对象。

之后把服务定义注册到 RPC 框架的本地服务注册表，同时把服务元数据上报到注册中心。这样业务方只需要在实现类上加注解，不需要写大量配置。

### 3. 可能用到哪些 Spring 扩展点？

**回答思路：**

可以结合具体实现选择不同扩展点：

- BeanPostProcessor：在 Bean 初始化后识别带注解的服务 Bean。
- ImportBeanDefinitionRegistrar：在启动阶段注册框架需要的 BeanDefinition。
- ApplicationContextAware：获取 Spring 容器上下文，扫描 Bean。
- ApplicationListener：监听容器刷新完成事件，在应用启动后执行服务发布。

核心思路是在 Spring 容器完成 Bean 创建后，扫描指定注解的 Bean，然后将服务实例和元数据交给 RPC 框架发布。

### 4. RPC 服务引用怎么实现？

**回答思路：**

服务引用通常通过动态代理实现。业务方在字段上加引用注解，框架启动时扫描这些字段，为对应接口生成代理对象并注入到 Spring Bean 中。

调用代理对象方法时，代理会把接口名、方法名、参数、版本等信息封装为 RPC 请求，然后通过负载均衡选择服务提供者，发起网络调用，最后反序列化响应结果返回给业务方。

### 5. Sentinel 如何接入 RPC？

**回答思路：**

Sentinel 接入 RPC 的核心是定义资源和处理限流异常。

在 provider 或 consumer 调用入口构造资源名，资源名可以是接口名、接口名 + 方法名，或者应用名 + 接口名 + 方法名。调用前进入 Sentinel 资源，如果触发限流，会抛出 BlockException，框架捕获后返回降级结果，或者转换成统一 RPC 异常。

调用完成后要正确退出 Sentinel entry，避免上下文泄漏。

### 6. 限流放在服务提供方还是消费方？

**回答思路：**

两边都可以放，但目标不同：

- 消费方限流：保护下游，避免自身发起过多请求，也可以做客户端侧熔断。
- 提供方限流：保护服务自身，避免被过大流量打垮。

如果只能选一个，我会优先在提供方做限流，因为提供方更清楚自身承载能力。实际系统中可以两边结合，消费方做熔断和超时控制，提供方做入口限流和降级保护。

### 7. RPC 调用超时怎么处理？

**回答思路：**

RPC 超时要从多个层面处理：

- 客户端设置合理超时时间，避免无限等待。
- 服务端监控慢调用，识别耗时方法。
- 失败重试要谨慎，只对幂等接口重试。
- 超时不一定代表服务端失败，可能服务端执行成功但响应没返回。
- 下游持续超时时要结合熔断降级，避免线程资源被拖垮。

## 五、综合追问准备

### 1. 你觉得最有技术含量的项目是什么？

**推荐回答：**

我会选择应用监控与错误日志智能查询助手。因为它不是简单 ChatBot，而是把大模型和内部可观测平台能力结合起来，通过 MCP 工具调用方式，让模型能够查询真实监控和日志数据。项目里涉及意图识别、参数抽取、工具调用、权限控制、查询结果摘要和异常兜底，比较能体现后端系统设计和 AI 应用落地能力。

### 2. 你做的 AI 助手和套壳大模型有什么区别？

**推荐回答：**

套壳大模型主要是把用户问题转发给模型，而我做的场景更偏工程工具化。模型不直接作为事实来源，而是负责理解问题、选择工具和总结结果。真实数据来自知识库、监控平台或日志平台。后端还需要负责工具注册、参数校验、权限控制、结果结构化和异常兜底，所以它更像一个智能化后端系统，而不是简单聊天页面。

### 3. 你项目经历比较分散，怎么体现你的优势？

**推荐回答：**

我的经历确实不是长期深耕单一业务领域，但我参与过几类不同系统：RPC 框架扩展、活动后端、JDK 升级治理、AI 排障助手。它们分别涉及框架扩展、业务一致性、工程治理和工具平台化。

所以我的优势是能在不同场景下抽象问题、设计稳定链路，并把后端能力落地。比如活动项目里关注幂等和补偿，RPC 项目里关注服务暴露和限流，AI 助手里关注工具调用可控性和数据可信度。

### 4. 你后续的职业方向是什么？

**推荐回答：**

**推荐回答：**

所以高级后端的重点是能识别系统风险，并设计机制去约束和兜底。

## 六、面试回答通用模板

### STAR 模板

- **S：背景**：当时业务或技术上遇到了什么问题。
- **T：任务**：你负责解决哪部分问题。
- **A：行动**：你怎么分析、设计和落地。
- **R：结果**：带来了什么效果，沉淀了什么经验。

### 技术项目回答模板

可以按照下面结构回答：

```text
这个项目的背景是……
我主要负责……
核心链路是……
这里面最大的难点是……
我当时的设计是……
这样做的好处是……
最后项目带来的结果是……
```

### 被追问细节时的回答原则

- **没做过的不要硬说做过**，可以说“这个点我没有直接负责，但我理解合理方案是……”。
- **不要只讲概念**，尽量结合项目场景讲。
- **不要过度夸大数据**，没有真实数据就说“减少重复人工查询成本”“提升常见问题自助定位效率”。
- **回答高级问题时要讲取舍**，比如一致性和性能、实时性和成本、模型能力和工程可控性。
