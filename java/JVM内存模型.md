# JVM 内存模型详解

> ⚠️ **概念澄清**：日常说的"JVM 内存模型"通常有两种含义，容易混淆：
>
> - **JVM 运行时内存结构（JVM Memory Structure / Runtime Data Areas）**：JVM 在运行 Java 程序时划分的内存区域（堆、栈、方法区等）—— **这是物理内存布局**
> - **Java 内存模型（JMM, Java Memory Model）**：定义多线程下变量的可见性、原子性、有序性的规范 —— **这是并发抽象规范**
>
> 本文将这两部分都讲清楚，并指出它们的关系。

---

## 一、JVM 运行时内存结构（物理布局）

JVM 规范将运行时内存划分为以下几个区域：

```
                ┌─────────────────────────────────────────┐
                │           JVM 运行时数据区              │
                ├─────────────────────────────────────────┤
                │  线程共享区域            │ 线程私有区域 │
                ├──────────────────────────┼──────────────┤
                │  ① 堆 (Heap)             │ ④ 虚拟机栈   │
                │  ② 方法区 (Method Area)  │ ⑤ 本地方法栈 │
                │  ③ 直接内存 (Off-Heap)   │ ⑥ 程序计数器 │
                └──────────────────────────┴──────────────┘
```

### 1.1 堆（Heap）⭐ 线程共享

- **存放内容**：所有对象实例、数组
- **GC 主战场**：垃圾回收的主要管理区域
- **分代结构**（JDK 8 前的传统分代）：

```
┌─────────────────────── 堆 ───────────────────────┐
│                                                  │
│  ┌─────── 新生代 (Young) ────────┐ ┌──老年代──┐ │
│  │ Eden  │ Survivor 0 │ Survivor 1│ │ (Old)    │ │
│  │  8    │     1      │     1     │ │          │ │
│  └───────────────────────────────┘ └──────────┘ │
└──────────────────────────────────────────────────┘
```

- **新生代**：新对象在 Eden 区分配，经历 Minor GC 后存活的进入 Survivor 区
- **老年代**：存活时间长（默认 15 次 GC）的对象晋升到此，由 Major GC / Full GC 回收

**异常**：内存不足抛 `OutOfMemoryError: Java heap space`。

---

### 1.2 方法区（Method Area）⭐ 线程共享

- **存放内容**：已加载的类信息、常量、静态变量、JIT 编译后的代码缓存
- **JDK 演进**：
  - JDK 7 及之前：**永久代（PermGen）**，在堆中实现
  - JDK 8 及之后：**元空间（Metaspace）**，使用**本地内存（Native Memory）**

```
JDK 7：永久代 → 堆内存
JDK 8：元空间 → 本地内存（不再受 -Xmx 限制）
```

**为什么改成元空间？**
- 永久代有大小限制，容易抛 `OutOfMemoryError: PermGen space`
- 元空间使用本地内存，扩容更灵活，几乎只受物理内存限制
- 字符串常量池也在 JDK 7 移到了堆中

**异常**：`OutOfMemoryError: Metaspace`。

---

### 1.3 程序计数器（PC Register）线程私有

- **存放内容**：当前线程执行的字节码**行号指示器**
- **作用**：线程切换后能恢复到正确的执行位置
- **唯一不会发生 OOM 的区域**

---

### 1.4 虚拟机栈（VM Stack）线程私有

- **存放内容**：每个 Java 方法执行时创建一个**栈帧（Stack Frame）**

```
栈帧结构：
┌─────────────────────────┐
│  局部变量表（Locals）   │  ← 基本类型值、对象引用
├─────────────────────────┤
│  操作数栈（Stack）      │  ← 字节码运算的工作区
├─────────────────────────┤
│  动态链接（Dynamic Link）│  ← 指向运行时常量池的方法引用
├─────────────────────────┤
│  方法返回地址（Return） │
└─────────────────────────┘
```

- 方法调用即栈帧入栈，方法返回即出栈
- **异常**：
  - 栈深度超限：`StackOverflowError`（如递归过深）
  - 栈扩容失败：`OutOfMemoryError`

---

### 1.5 本地方法栈（Native Method Stack）线程私有

- 与虚拟机栈类似，但服务于 **Native 方法**（C/C++ 实现的方法，如 `Object.hashCode()`）
- HotSpot 虚拟机将虚拟机栈和本地方法栈**合二为一**

---

### 1.6 直接内存（Direct Memory）非 JVM 规范定义

- **不是 JVM 运行时数据区的一部分**，但被频繁使用
- 通过 `ByteBuffer.allocateDirect()`、Netty、NIO 等申请
- 优点：**避免在 JVM 堆和 Native 堆之间复制数据**，提升 IO 性能
- 受 `-XX:MaxDirectMemorySize` 限制

---

## 二、JVM 各区域共享性总结

| 内存区域 | 是否线程共享 | 是否 GC 管理 | 异常类型 |
|---------|-------------|-------------|----------|
| 堆 | ✅ 共享 | ✅ 主要 GC 区 | OOM: Heap |
| 方法区/元空间 | ✅ 共享 | ⚠️ 少量 GC | OOM: Metaspace |
| 程序计数器 | ❌ 私有 | ❌ | 无 |
| 虚拟机栈 | ❌ 私有 | ❌ | StackOverflow / OOM |
| 本地方法栈 | ❌ 私有 | ❌ | StackOverflow / OOM |
| 直接内存 | ✅ 共享 | ⚠️ 少量回收 | OOM: Direct buffer |

## 三、Java 内存模型（JMM）⭐ 并发核心

> JMM 不是"内存的物理布局"，而是 **JVM 对多线程下共享变量的访问规则的抽象规范**，目的是屏蔽底层硬件和 OS 的内存差异，保证 Java 程序"一次编写，处处并发安全"。

### 3.1 JMM 抽象模型

JMM 把内存分为：
- **主内存（Main Memory）**：所有线程共享的数据（对应堆中的实例字段、静态字段）
- **工作内存（Working Memory）**：每个线程私有的数据副本（对应 CPU 缓存、寄存器）

```
        ┌──────────────────────────────┐
        │       主内存 (Main Memory)    │
        │       共享变量 x = 0          │
        └──────────────────────────────┘
              ↑↓             ↑↓
   ┌──────────┴──┐      ┌────┴───────┐
   │ 线程 A 工作内存 │   │ 线程 B 工作内存 │
   │  x 副本 = ?   │    │  x 副本 = ?   │
   └─────────────┘      └──────────────┘
```

**关键事实**：线程间不能直接访问对方的工作内存，所有变量交互都必须通过**主内存**。

---

### 3.2 JMM 解决的三大并发问题

#### ① 原子性（Atomicity）

一个操作要么全部完成，要么全部不执行，不被其他线程打断。

- 基本类型读写**大多数原子**（`long`/`double` 在 32 位 JVM 下不保证）
- `i++` 是 **3 步**（读、改、写），**非原子**
- **解决**：`synchronized`、`Atomic` 类（CAS）

#### ② 可见性（Visibility）

一个线程修改共享变量后，其他线程能立即看到。

- 普通变量的修改**不保证立即对其他线程可见**（缓存问题）
- **解决**：`volatile`、`synchronized`、`final`

#### ③ 有序性（Ordering）

程序按代码顺序执行，但 JIT 和 CPU 可能进行**指令重排序**优化。

- 单线程内：`as-if-serial` 保证最终结果与顺序执行一致
- 多线程间：可能出现重排序导致的并发问题（如 DCL 单例的"半初始化对象"）
- **解决**：`volatile`（插入内存屏障）、`synchronized`、`final`

---

### 3.3 8 种内存交互操作

JMM 定义了 8 种原子操作（了解即可，新版规范已简化）：

| 操作 | 作用域 | 含义 |
|------|--------|------|
| lock | 主内存 | 锁定变量 |
| unlock | 主内存 | 释放锁 |
| read | 主内存 → 工作内存 | 读取变量 |
| load | 工作内存 | 把 read 的值放入副本 |
| use | 工作内存 → 引擎 | 引擎使用变量 |
| assign | 引擎 → 工作内存 | 引擎赋值给变量 |
| store | 工作内存 → 主内存 | 把变量值传送到主内存 |
| write | 主内存 | 把 store 的值写入主内存 |

---

### 3.4 happens-before 原则 ⭐

JMM 通过 **happens-before** 来保证操作之间的可见性和有序性。**A happens-before B** 表示：A 的执行结果对 B 可见，且 A 在 B 之前发生。

**8 大天然 happens-before 规则**：

1. **程序顺序规则**：单线程内，前面的操作 happens-before 后面的操作
2. **监视器锁规则**：解锁 happens-before 后续对同一锁的加锁
3. **volatile 规则**：volatile 写 happens-before 后续对该变量的读
4. **线程启动规则**：`Thread.start()` happens-before 线程内的所有操作
5. **线程终止规则**：线程内所有操作 happens-before 其他线程检测到该线程结束（`Thread.join()` 返回）
6. **线程中断规则**：`interrupt()` 调用 happens-before 被中断线程检测到中断事件
7. **对象终结规则**：对象构造函数完成 happens-before `finalize()` 开始
8. **传递性**：A→B 且 B→C，则 A→C

---

### 3.5 内存屏障（Memory Barrier）

JMM 通过插入**内存屏障**实现有序性和可见性，4 种类型：

| 屏障类型 | 含义 |
|---------|------|
| LoadLoad | Load1; LoadLoad; Load2 → Load1 必须先于 Load2 完成 |
| StoreStore | Store1; StoreStore; Store2 → Store1 必须先刷到主内存，再执行 Store2 |
| LoadStore | Load1; LoadStore; Store2 → Load1 必须先于 Store2 完成 |
| StoreLoad | Store1; StoreLoad; Load2 → Store1 刷到主内存后，Load2 才能执行（**最重的屏障**） |

**应用**：
- `volatile` 写：前面插 StoreStore，后面插 StoreLoad
- `volatile` 读：前面插 LoadLoad，后面插 LoadStore
- `final` 字段写入：写后插 StoreStore（保证安全发布，详见 [final关键字.md](file:///Users/majin/IdeaProjects/channel/java/final%E5%85%B3%E9%94%AE%E5%AD%97.md)）

## 四、JVM 内存结构 vs JMM 对比

| 维度 | JVM 内存结构 | Java 内存模型（JMM） |
|------|-------------|---------------------|
| 关注点 | 物理存储区域划分 | 多线程共享变量访问规则 |
| 解决问题 | 内存如何分配/回收 | 原子性、可见性、有序性 |
| 范围 | 单 JVM 实例的内存布局 | 多线程之间的协作语义 |
| 典型概念 | 堆、栈、方法区、GC 分代 | volatile、final、happens-before、内存屏障 |

**关系**：JMM 是建立在 JVM 内存结构之上的**并发抽象规范**。JVM 的堆中存储的共享变量，正是 JMM 主内存的物理体现。

## 五、关键流程图：一个对象的内存生命周期

```
1. new 对象
   ↓
2. JVM 在【堆】上分配内存
   ↓
3. 类元数据从【方法区/元空间】加载
   ↓
4. 对象引用存入【虚拟机栈】的局部变量表
   ↓
5. 多线程访问时，受【JMM】规范约束（happens-before、volatile、final）
   ↓
6. 不再被引用 → 进入 GC Roots 不可达
   ↓
7. 被【堆】的 GC 回收（Minor GC / Full GC）
```

## 六、JDK 版本演进对内存结构的影响

| JDK 版本 | 关键变化 |
|---------|---------|
| JDK 6 | 永久代 + 字符串常量池在永久代 |
| JDK 7 | 字符串常量池**移到堆中**（解决永久代溢出常见问题） |
| JDK 8 | **永久代彻底移除，引入元空间**（使用本地内存） |
| JDK 9+ | 引入紧凑字符串、ZGC、Shenandoah 等新 GC |

## 八、面试加分要点

1. **概念区分**：JVM 内存结构（物理）≠ JMM（并发规范），面试时务必先澄清。
2. **JDK 8 元空间**：永久代被彻底移除，类元数据放到本地内存的元空间。
3. **栈帧结构**：局部变量表 + 操作数栈 + 动态链接 + 返回地址，是字节码执行的基础。
4. **JMM 三大特性**：原子性、可见性、有序性，分别对应 `synchronized/Atomic`、`volatile/synchronized/final`、`volatile/synchronized/final`。
5. **happens-before 是 JMM 核心**：理解了 happens-before，就理解了 Java 并发的本质。
6. **指令重排序的影响**：是 DCL 单例必须用 `volatile` 的根本原因。
7. **内存屏障是 JMM 落地的硬件抽象**：4 种屏障（LoadLoad/StoreStore/LoadStore/StoreLoad）是 `volatile`/`final` 实现的底层机制。

## 九、面试一句话总结 ⭐

> **"JVM 内存模型"在面试中通常指两个不同概念，需要澄清：**
>
> **第一个是 JVM 运行时内存结构**——JVM 把内存划分为**线程共享的堆和方法区（JDK 8 后的元空间）**、**线程私有的虚拟机栈、本地方法栈、程序计数器**，以及不属于规范但常用的**直接内存**。其中**堆**存放对象实例，是 GC 主战场；**虚拟机栈**为每次方法调用维护一个栈帧（局部变量表 + 操作数栈 + 动态链接 + 返回地址）。
>
> **第二个是 Java 内存模型（JMM）**——它是 JVM 对多线程下共享变量访问规则的抽象规范，把内存分为**主内存（共享）**和**工作内存（线程私有）**，并通过 `volatile`、`synchronized`、`final` 等关键字配合**内存屏障**和 **happens-before 原则**，解决多线程下的**原子性、可见性、有序性**三大问题。
>
> **关系**：JMM 建立在 JVM 内存结构之上——堆中的共享变量是 JMM 主内存的物理实现，CPU 缓存/寄存器是工作内存的物理实现。一个理解了 GC，一个理解了并发，二者结合才能完整把握 Java 程序的内存运行机制。
