# JMM（Java 内存模型）核心概念详解

## 一、JMM 是什么？

**JMM（Java Memory Model，Java 内存模型）** 是 Java 虚拟机规范中定义的一组**抽象规则**，用于规定**多线程环境下，一个线程对共享变量的修改何时、以何种方式对其他线程可见**。

> ⚠️ **关键澄清**：JMM 不是物理内存的布局，也不是 JVM 运行时的内存结构（堆、栈、方法区），而是**一套并发编程的规范契约**。
>
> 关于 JVM 运行时内存结构，详见 [JVM内存模型.md](file:///Users/majin/IdeaProjects/channel/java/JVM%E5%86%85%E5%AD%98%E6%A8%A1%E5%9E%8B.md)。

## 二、JMM 要解决什么问题？

### 2.1 现代计算机的硬件矛盾

```
        CPU                 主内存
   ┌──────────┐           ┌────────┐
   │ 寄存器    │ ←→ L1 ←→ │  RAM   │
   │ (纳秒级) │           │ (百纳秒)│
   └──────────┘           └────────┘
        ↑
   速度差距 100 倍以上
```

为弥补 CPU 与主内存的速度差距，引入了**多级缓存（L1/L2/L3）**和**寄存器**，但这带来了三大问题：

1. **缓存一致性**：每个 CPU 核心都有自己的缓存，对同一变量的修改可能不一致
2. **指令重排序**：编译器和 CPU 为了性能会调整指令执行顺序
3. **原子性破坏**：一条 Java 语句可能被拆成多条机器指令，中间可能被打断

### 2.2 JMM 的目标

> **屏蔽不同硬件、不同操作系统的内存访问差异，让 Java 程序在所有平台上都能保证一致的并发语义。**

这正是 Java"一次编写，处处运行"承诺在并发层面的延伸。

## 三、JMM 的抽象模型

JMM 定义了一个**两层内存抽象**：

```
        ┌──────────────────────────────────┐
        │      主内存 (Main Memory)         │
        │   存放所有共享变量（实例字段、     │
        │   静态字段、数组元素等）          │
        └──────────────────────────────────┘
              ↑↓ read/load/store/write ↑↓
   ┌──────────┴────────┐    ┌──────────┴────────┐
   │ 线程 A 的工作内存 │    │ 线程 B 的工作内存 │
   │ (变量副本 + 操作) │    │ (变量副本 + 操作) │
   └───────────────────┘    └───────────────────┘
```

### 3.1 主内存（Main Memory）

- 所有线程**共享**的内存区域
- 存放**共享变量**：实例字段、静态字段、数组元素
- **物理对应**：JVM 堆中的对象数据

### 3.2 工作内存（Working Memory）

- 每个线程**私有**的内存区域
- 存放该线程使用的**共享变量副本**
- **物理对应**：CPU 寄存器、L1/L2 缓存、栈中的局部变量

### 3.3 核心规则

> **线程之间不能直接访问对方的工作内存，所有共享变量的交互必须通过主内存中转。**

```
线程 A 修改变量 x：
  1. 在工作内存中改自己的 x 副本
  2. 把新值写回主内存

线程 B 读取变量 x：
  1. 从主内存读取 x 的最新值到自己的工作内存
  2. 在工作内存中使用该副本
```

## 四、JMM 关注的三大并发特性

JMM 的所有规则、关键字、屏障，最终都围绕这三大特性展开：

### 4.1 原子性（Atomicity）

**定义**：一个或多个操作要么**全部完成**，要么**全部不执行**，中间不被其他线程打断。

**JMM 提供的原子性保证**：
- 基本类型读写（除 `long`/`double` 在 32 位 JVM 下）
- `assign`/`store`/`write` 等 8 种基本操作
- `synchronized` 块内的所有操作（通过互斥实现"宏观原子性"）
- `Atomic` 系列类（基于 CAS）

**经典反例**：`i++` 不是原子操作（读 → 改 → 写）。

---

### 4.2 可见性（Visibility）

**定义**：一个线程修改了共享变量后，其他线程能**立即感知到**这个修改。

**为什么会有可见性问题？**
线程 A 修改变量后，可能只更新了自己的工作内存（CPU 缓存），还没刷回主内存；线程 B 读取的仍是主内存的旧值，或自己工作内存中的旧副本。

**JMM 提供的可见性保证**：
- `volatile`：写时立即刷主内存，读时强制从主内存加载
- `synchronized`：解锁前必须把变量同步回主内存
- `final`：构造完成后的值对所有线程可见

**经典反例**：

```java
class Worker {
    boolean running = true;        // 没有 volatile

    public void run() {
        while (running) { ... }    // ⚠️ 可能永远看不到 running=false
    }

    public void stop() {
        running = false;            // 其他线程修改
    }
}
```

---

### 4.3 有序性（Ordering）

**定义**：程序按代码编写的顺序执行。

**为什么会有有序性问题？**
- **编译器重排**：JIT 优化指令顺序
- **CPU 重排**：乱序执行（Out-of-Order Execution）
- **内存系统重排**：写缓冲区延迟写入主内存

**JMM 提供的有序性保证**：
- 单线程内：`as-if-serial` 原则保证最终结果与顺序执行一致
- 多线程间：通过 `volatile`、`synchronized`、`final` 配合**内存屏障**禁止重排

**经典反例**：DCL 单例的"半初始化"问题（详见 [懒加载单例模式.md](file:///Users/majin/IdeaProjects/channel/java/%E6%87%92%E5%8A%A0%E8%BD%BD%E5%8D%95%E4%BE%8B%E6%A8%A1%E5%BC%8F.md)）。

## 五、JMM 的核心机制：happens-before

happens-before 是 JMM 用来描述两个操作间**内存可见性 + 执行顺序保证**的核心抽象。

> **A happens-before B**：A 操作的结果对 B 可见，且 A 在 B 之前发生。

### 5.1 8 大天然规则

1. **程序顺序规则**：单线程内，前面的操作 happens-before 后面的操作
2. **监视器锁规则**：解锁 happens-before 后续对同一锁的加锁
3. **volatile 变量规则**：volatile 写 happens-before 后续对该变量的读
4. **线程启动规则**：`Thread.start()` happens-before 该线程内的所有操作
5. **线程终止规则**：线程内所有操作 happens-before 其他线程检测到该线程结束
6. **线程中断规则**：`interrupt()` happens-before 被中断线程检测到中断
7. **对象终结规则**：构造函数完成 happens-before `finalize()` 开始
8. **传递性**：A→B 且 B→C，则 A→C

### 5.2 直观理解

> **happens-before 的本质：JMM 给程序员的一份"承诺"——只要你的代码满足某条 happens-before 规则，JMM 就保证可见性和有序性，你不用关心底层的缓存、指令重排细节。**

## 六、JMM 的落地工具

### 6.1 三大关键字

| 关键字 | 解决问题 | 实现机制 |
|--------|---------|---------|
| `volatile` | 可见性 + 有序性 | 内存屏障（StoreStore/StoreLoad/LoadLoad/LoadStore） |
| `synchronized` | 原子性 + 可见性 + 有序性 | 互斥锁 + 内存屏障 |
| `final` | 安全发布（可见性 + 有序性） | StoreStore 屏障（写 final 后） |

详见 [final关键字.md](file:///Users/majin/IdeaProjects/channel/java/final%E5%85%B3%E9%94%AE%E5%AD%97.md) 关于 final 字段并发语义的深入剖析。

### 6.2 内存屏障（Memory Barrier）

JMM 通过插入 4 种内存屏障在底层硬件上实现禁止重排：

| 屏障 | 作用 |
|------|------|
| LoadLoad | 保证前面的读先于后面的读完成 |
| StoreStore | 保证前面的写先于后面的写刷新到主内存 |
| LoadStore | 保证前面的读先于后面的写完成 |
| StoreLoad | 全能屏障（最重），保证前面的写对后面的读可见 |

### 6.3 8 种内存交互操作

JMM 规范定义了 8 种原子操作（了解即可）：

| 操作 | 作用域 | 含义 |
|------|--------|------|
| lock | 主内存 | 锁定变量 |
| unlock | 主内存 | 释放锁 |
| read | 主内存 → 工作内存 | 读取变量 |
| load | 工作内存 | 把 read 的值放入副本 |
| use | 工作内存 → 引擎 | 引擎使用变量 |
| assign | 引擎 → 工作内存 | 引擎赋值给变量 |
| store | 工作内存 → 主内存 | 把变量值传送到主内存 |
| write | 主内存 | 把 store 的值写入主内存 |

## 七、JMM 与 JVM 内存结构的关系

| 维度 | JVM 内存结构 | Java 内存模型（JMM） |
|------|-------------|---------------------|
| 关注点 | 物理存储区域划分 | 多线程共享变量访问规则 |
| 解决的问题 | 内存如何分配/回收 | 原子性、可见性、有序性 |
| 典型概念 | 堆、栈、方法区、GC 分代 | 主内存、工作内存、happens-before、内存屏障 |
| 是否实体 | 真实的内存区域 | 抽象规范（不对应任何具体硬件） |

**关系**：JMM 抽象层建立在 JVM 内存结构的物理层之上：
- **JMM 主内存** ≈ JVM 堆中的共享变量（实例/静态字段）
- **JMM 工作内存** ≈ CPU 寄存器 + 缓存 + 虚拟机栈中的局部变量副本

## 八、一个完整的例子：理解 JMM 的工作流程

```java
class Example {
    int a = 0;                       // 共享变量
    volatile boolean flag = false;    // volatile 变量

    // 线程 A
    public void writer() {
        a = 1;                        // ① 普通写
        flag = true;                  // ② volatile 写
    }

    // 线程 B
    public void reader() {
        if (flag) {                   // ③ volatile 读
            int i = a;                // ④ 普通读，i 的值一定是 1
        }
    }
}
```

**JMM 的保证**：
- 由于 ② 是 volatile 写，JMM 会在 ② 之前插入 **StoreStore 屏障** → ① 不能重排到 ② 之后
- 由于 ③ 是 volatile 读，JMM 会在 ③ 之后插入 **LoadLoad 屏障** → ④ 不能重排到 ③ 之前
- 通过 happens-before **传递性**：① → ② → ③ → ④
- **结论**：线程 B 进入 if 后，读到的 `a` **一定是 1**，不会是 0

这就是经典的 **"volatile 写 happens-before volatile 读"** 应用场景，常用于状态标志位、双重检查锁等。

## 十、面试加分要点

1. **JMM 不是物理内存布局**：是抽象规范，要与"JVM 运行时内存结构"严格区分。
2. **三大特性**：原子性、可见性、有序性，是 JMM 解决的全部问题。
3. **JMM 通过 happens-before 给出统一抽象**，让程序员不必关心底层硬件细节。
4. **`volatile` 解决可见性 + 有序性，但不解决原子性**（常见陷阱：`volatile int i; i++` 仍非原子）。
5. **JSR-133 是分水岭**：JDK 5 重构了 JMM，重新定义了 `volatile` 和 `final` 的语义，DCL 才真正安全。
6. **内存屏障是 JMM 落地的硬件抽象**：4 种屏障对应 CPU 的真实指令（如 x86 的 `mfence`/`lfence`/`sfence`）。
7. **as-if-serial**：单线程内任意重排，但最终结果不变；多线程下需要 happens-before 来约束。

## 十一、面试一句话总结 ⭐

> **JMM（Java 内存模型）是 Java 在多线程环境下定义的一套抽象规范，目的是屏蔽不同硬件和操作系统的内存访问差异，保证 Java 程序在任意平台上都有一致的并发语义。**
>
> **它把内存抽象成两层：所有线程共享的"主内存"和每个线程私有的"工作内存"，规定线程对共享变量的所有操作必须通过主内存中转，不能直接访问其他线程的工作内存。**
>
> **JMM 解决三大并发问题：原子性（不被打断）、可见性（修改对其他线程可见）、有序性（禁止有害的指令重排）。**
>
> **它通过 `volatile`、`synchronized`、`final` 三大关键字配合底层的内存屏障来落地，并提供 happens-before 原则作为程序员理解可见性和有序性的"统一契约"——只要满足 happens-before，就不必关心底层的缓存与重排细节。**
>
> **简单说：JMM 是 Java 并发的"宪法"，是所有锁、原子类、并发工具类得以正确工作的底层基石。**
