# Java 应用线上问题排查

## 总览：五大类线上问题

| 问题类型 | 典型表现 | 核心工具 |
|---------|---------|---------|
| **CPU 飙高** | 负载高、响应慢、风扇狂转 | `top`、`jstack`、Arthas |
| **内存溢出/泄漏** | OOM、频繁 Full GC、响应越来越慢 | `jmap`、MAT、`jstat` |
| **线程问题** | 死锁、线程池打满、请求卡住 | `jstack`、Arthas `thread` |
| **GC 问题** | STW 过长、吞吐下降 | GC 日志、`jstat`、GCEasy |
| **接口响应慢** | RT 升高、超时告警 | Arthas `trace`/`watch`、链路追踪 |

---

## 一、CPU 飙高
优先去看监控大盘，重点关注请求访问量/GC次数这两个指标。
这两个指标通常是引起CPU升高的主要原因。

### 1.1 第一步：先判断是流量问题还是代码问题（**最重要**）

> **核心原则**：拿到 CPU 告警后，**不要急于登机器执行 top/jstack**，先看监控判断 CPU 升高是「合理的」还是「异常的」。

**判断流程**：

```
CPU 告警
   │
   ├─► 看监控大盘（QPS / RT / 业务量）
   │      │
   │      ├─► QPS 同步上升 → 流量驱动型 → 扩容 / 限流 / 看是否符合容量规划
   │      │
   │      └─► QPS 平稳但 CPU 飙高 → 代码异常型 → 进入下一步定位
   │
   └─► 看其他维度
          ├─► 是否近期有发布？ → 回滚验证
          ├─► 是否有定时任务？ → 排查批处理逻辑
          └─► 是否有上游重试风暴？ → 看入口流量构成
```

**两类问题的处理路径完全不同**：

| 类型 | 表现 | 处理方案 |
|------|------|---------|
| **流量驱动（正常）** | QPS、入口请求数同步上升，多实例 CPU 一起升高 | 扩容、限流、提前做容量评估 |
| **代码异常（非正常）** | QPS 平稳甚至下降，但 CPU 持续高位；或个别实例异常 | 进入下面的四步法定位线程堆栈 |

**应重点关注的监控指标**：

- **流量层**：网关 QPS、各接口调用量、上游服务调用量
- **应用层**：CPU 使用率、Load、线程数、GC 频率
- **业务层**：订单数、支付数等业务计数器（避免被网关刷量误导）
- **发布信息**：最近一次发布时间、变更内容（80% 的线上问题与近期变更有关）

> **面试话术**：CPU 告警后我的第一反应不是登机器，而是先看监控大盘。如果 QPS 同步上升，说明 CPU 升高是流量驱动的，方案是扩容或限流；如果 QPS 平稳但 CPU 飙高，才是代码异常问题，这时候才进入 top/jstack 四步法。同时还会看最近的发布记录——很多问题都是发版引入的，回滚比定位更快止血。

### 1.2 第二步：定位异常线程（经典四步法）

确认是代码异常型问题后，再按以下步骤定位：

```bash
# 第一步：找到 CPU 最高的 Java 进程
top -c                    # 找到目标 PID，假设为 12345

# 第二步：找到该进程中 CPU 最高的线程
top -Hp 12345             # 找到线程 TID，假设为 12346

# 第三步：将线程 ID 转为十六进制
printf "%x\n" 12346       # 输出 303a

# 第四步：用 jstack 查看该线程在干什么
jstack 12345 | grep '303a' -A 30
```

### 1.3 Arthas 一键排查（推荐）

```bash
# 进入 Arthas
java -jar arthas-boot.jar

# 直接查看最忙的线程及其堆栈
thread -n 3               # 显示 CPU 占用最高的 3 个线程
thread -b                 # 检测死锁
```

### 1.4 火焰图：热点代码的"杀手锏"

> **为什么需要火焰图？**
> jstack / Arthas `thread` 是**某一瞬间的快照**，对于间歇性占 CPU、多个热点叠加、调用链较深的场景容易漏看。火焰图是基于**一段时间的高频采样（默认 99Hz）**聚合出来的，能够准确反映“哪些函数累计占用了最多 CPU”。

#### 火焰图怎么看

```
   y 轴：调用栈深度（上面是被调者，下面是调用者）
   x 轴：采样次数占比（**不代表时间顺序**，按函数名字母序排列）
   定位关键：看**平顶**——越宽的顶部平台，说明该函数占用 CPU 越多
```

**三种典型火焰图形状**：

| 形状 | 含义 | 处理 |
|------|------|------|
| **宽而矮的平顶** | 某个单一函数占用大量 CPU（如正则、JSON 序列化） | 直接优化该函数 |
| **高而窄的尖塔** | 调用链过深，但各层耗时均匀 | 考虑减少调用层次或合并逻辑 |
| **多个宽顶平台** | 多个热点共存 | 按占比依次优化 |

#### 生成火焰图的三种方式

**方式一：Arthas profiler（推荐，开箱即用）**

```bash
# 进入 Arthas
java -jar arthas-boot.jar

# 开始采样
profiler start                    # 默认采样 CPU
profiler start --event alloc      # 或采样内存分配（排查 GC 压力）

# 采样 60 秒左右，足够覆盖热点
profiler stop --format html       # 生成 HTML 火焰图
# 输出例：OK
#         profiler output file: /tmp/arthas-output/xxxxxx.html

# 下载到本地用浏览器打开即可交互
```

> Arthas profiler 底层封装的就是 **async-profiler**，这是目前 Java 生态最主流的低开销采样器（利用 perf_events + AsyncGetCallTrace，**不会发生 safepoint bias**，采样更准确）。

**方式二：async-profiler（独立使用）**

```bash
# 下载后直接运行
./profiler.sh -d 60 -f flame.html <PID>

# 常用参数
-d <秒>          # 采样时长
-e cpu|alloc|lock # 采样事件类型（CPU/内存分配/锁竞争）
-f <文件>        # 输出文件
```

**方式三：JFR + JMC（JDK 自带，生产环境首选）**

```bash
# JDK 11+ 官方推荐，开销极低（< 1%）
jcmd <PID> JFR.start duration=60s filename=/tmp/profile.jfr

# 下载后用 JDK Mission Control (JMC) 打开分析
# 或转换为火焰图：jfrconv profile.jfr flame.html
```

#### 三种采样方式对比

| 方式 | 开销 | 准确性 | 适用场景 |
|------|------|--------|---------|
| **jstack** | 低 | 差（快照） | 快速看某个线程在干什么 |
| **async-profiler / Arthas profiler** | 低 | 高（无 safepoint bias） | 临时排查 CPU 热点 |
| **JFR** | 极低 | 高 | 生产长期开启、事后复盘 |

#### 为什么不用 jstack “手动多次 dump”代替火焰图？

传统的“jstack 多次采样 + 人看”方法有三个缺陷：

1. **safepoint bias**：jstack 只能在线程到达安全点时采样，导致**循环体内的热点被低估**
2. **频率低**：手动采样频率远低于 99Hz，统计意义不足
3. **难以聚合**：人脑30 份堆栈难以看出热点，火焰图能自动聚合成可视化占比

### 1.5 常见原因

| 原因 | 堆栈/火焰图特征 | 解决方案 |
|------|---------|---------|
| **死循环** | 线程状态 `RUNNABLE`，堆栈停在循环逻辑 | 检查循环退出条件 |
| **正则回溯** | 火焰图中 `java.util.regex.Pattern` 出现宽顶 | 优化正则表达式，避免贪婪匹配嵌套 |
| **频繁 GC** | 火焰图中 `GC` 相关帧占比高 | 转到内存/GC 问题排查 |
| **序列化/加密** | `Jackson`、`Fastjson`、`Cipher` 帧宽顶 | 减少不必要的序列化，缓存结果 |
| **自旋锁竞争** | `Unsafe.park`/CAS 相关帧高位 | 降低并发竞争，考虑分段锁 |
| **日志打印过多** | `Logger.info`/`String.format` 宽顶 | 调低日志级别、异步 Appender |

> **面试话术（完整版）**：CPU 飙高的排查我会分三层走。
> **第一层先判断性质**：看监控大盘的 QPS 和业务量，如果 QPS 同步上升说明是流量驱动，处理方案是扩容或限流；如果 QPS 平稳但 CPU 飙高，说明是代码异常型。同时会看最近的发布记录，很多问题回滚就能立刻止血。
> **第二层快速定位线程**：使用 `top → top -Hp → printf → jstack` 四步法，或 Arthas `thread -n` 一键看最忙线程。适用于快速看「某一瞬间」在干什么。
> **第三层火焰图定位热点代码**：如果是间歇性 CPU 高、jstack 看不出明显热点，或需要准确定位“哪些函数累计耗了最多 CPU”，就用 `Arthas profiler` / `async-profiler` / `JFR` 采样 60 秒生成火焰图。看**宽顶平台**直接定位到热点函数。主流采样器底层是 async-profiler，低开销且无 safepoint bias。

## 二、内存溢出（OOM）

### 2.1 常见 OOM 类型

| OOM 类型 | 含义 | 常见原因 |
|----------|------|---------|
| `Java heap space` | 堆内存不足 | 大对象、内存泄漏、堆设置过小 |
| `Metaspace` | 元空间不足 | 动态生成类过多（CGLib、反射） |
| `GC overhead limit exceeded` | GC 耗时超过 98% 但回收不到 2% 内存 | 严重内存泄漏 |
| `unable to create new native thread` | 无法创建新线程 | 线程数过多，超过系统限制 |
| `Direct buffer memory` | 直接内存溢出 | NIO ByteBuffer 未释放 |

### 2.2 排查步骤

```bash
# 1. 先加上 JVM 参数，OOM 时自动 dump
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/tmp/heapdump.hprof

# 2. 如果应用还活着，手动 dump
jmap -dump:live,format=b,file=/tmp/heapdump.hprof <PID>

# 3. 查看堆内存概况
jmap -heap <PID>

# 4. 查看对象占用排行
jmap -histo:live <PID> | head -20
```

### 2.3 MAT 分析 dump 文件

核心分析路径：

1. **Leak Suspects Report** → MAT 自动给出泄漏嫌疑
2. **Dominator Tree** → 查看哪个对象占用内存最多
3. **对象的 GC Roots 引用链** → 右键 → Path to GC Roots → exclude weak/soft references → 找到是谁持有了引用

### 2.4 常见内存泄漏场景

| 场景 | 原因 | 解决方案 |
|------|------|---------|
| **静态集合** | `static Map` 只往里放不清理 | 使用弱引用或定期清理 |
| **未关闭资源** | Connection、Stream 未 close | try-with-resources |
| **ThreadLocal** | 线程池中 ThreadLocal 未 remove | 用完后 `finally` 中 remove |
| **监听器/回调** | 注册了监听器但没注销 | 生命周期结束时注销 |
| **缓存无上限** | 自建缓存没有淘汰策略 | 用 Caffeine/Guava Cache 设置 maxSize |

> **面试话术**：排查 OOM 的核心是拿到 heap dump 文件。生产环境建议始终加上 `-XX:+HeapDumpOnOutOfMemoryError`，用 MAT 的 Dominator Tree 找到大对象，再通过 GC Roots 引用链定位持有者。

## 三、线程问题

### 3.1 死锁排查

```bash
# jstack 直接检测死锁
jstack <PID> | grep -A 20 "Found one Java-level deadlock"

# Arthas
thread -b    # 一键检测阻塞线程（含死锁）
```

**jstack 死锁输出示例**：
```
Found one Java-level deadlock:
=============================
"Thread-1":
  waiting to lock monitor 0x00007f... (object 0x000000076ab28..., a java.lang.Object),
  which is held by "Thread-0"
"Thread-0":
  waiting to lock monitor 0x00007f... (object 0x000000076ab30..., a java.lang.Object),
  which is held by "Thread-1"
```

### 3.2 线程池打满

**表现**：`RejectedExecutionException`、请求超时

```bash
# 查看线程数
jstack <PID> | grep "pool-" | wc -l

# 查看线程状态分布
jstack <PID> | grep java.lang.Thread.State | sort | uniq -c
```

**常见原因与解决**：

| 原因 | 排查方向 | 解决方案 |
|------|---------|---------|
| 下游服务慢 | 大量线程 `WAITING`/`TIMED_WAITING` 在 HTTP 调用 | 加超时、熔断降级 |
| 任务堆积 | 队列满了触发拒绝策略 | 调整队列大小、增加核心线程 |
| 线程泄漏 | 每次 new 线程池而不复用 | 全局复用线程池 |

### 3.3 线程状态速查

| 状态 | 含义 | 排查方向 |
|------|------|---------|
| `RUNNABLE` | 正在运行或等待 CPU | CPU 高 → 看它在执行什么逻辑 |
| `BLOCKED` | 等待 synchronized 锁 | 锁竞争激烈，找锁持有者 |
| `WAITING` | 无限等待（`wait()`、`park()`） | 等待条件未被满足 |
| `TIMED_WAITING` | 限时等待（`sleep()`、`wait(timeout)`） | 一般正常，但大量出现可能是下游慢 |

> **面试话术**：线程问题用 `jstack` 就够了。死锁直接搜 "deadlock"，线程池问题看状态分布——如果大量线程卡在 WAITING 且堆栈指向 HTTP 调用，基本就是下游响应慢。

## 四、GC 问题

### 4.1 GC 监控

```bash
# 实时查看 GC 情况（每 1 秒输出一次）
jstat -gcutil <PID> 1000

# 输出示例：
#  S0     S1     E      O      M     CCS    YGC   YGCT    FGC   FGCT     GCT
#  0.00  45.23  67.89  82.34  95.12  92.45   234  2.345    12   8.234  10.579
```

**关键指标**：

| 指标 | 含义 | 告警阈值参考 |
|------|------|------------|
| `FGC` | Full GC 次数 | 频繁增长（> 1次/小时）需关注 |
| `FGCT` | Full GC 总耗时 | 单次 > 1s 需优化 |
| `O` | Old 区使用率 | 持续 > 80% 且 FGC 后不降 → 泄漏 |

### 4.2 GC 日志分析

```bash
# JDK 8 开启 GC 日志
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:/tmp/gc.log

# JDK 9+ 统一日志
-Xlog:gc*:file=/tmp/gc.log:time,uptime,level,tags
```

**分析工具**：GCEasy（https://gceasy.io）上传 GC 日志即可可视化分析。

### 4.3 常见 GC 问题

| 问题 | 表现 | 原因 | 解决方案 |
|------|------|------|---------|
| **频繁 Young GC** | YGC 次数快速增长 | 对象分配速率高，Eden 区太小 | 增大 Young 区或优化对象创建 |
| **频繁 Full GC** | FGC 频繁，O 区使用率居高不下 | 大对象直接进 Old 区、内存泄漏 | 排查泄漏、调整晋升阈值 |
| **Full GC 后 Old 区不降** | O 区使用率 GC 前后无变化 | 内存泄漏 | dump 分析引用链 |
| **CMS Concurrent Mode Failure** | 退化为 Serial Old | CMS 回收速度跟不上分配速度 | 降低触发阈值 `CMSInitiatingOccupancyFraction` |
| **G1 长时间 Mixed GC** | 停顿超过目标 | Region 中存活对象过多 | 调整 `MaxGCPauseMillis`、排查大对象 |

> **面试话术**：GC 问题排查的第一步是 `jstat -gcutil` 看 FGC 频率和 Old 区变化趋势。如果 Full GC 后 Old 区不降，基本确认是内存泄漏，需要 dump 分析。

## 五、接口响应慢

### 5.1 Arthas 方法级耗时分析

```bash
# trace：追踪方法内部调用耗时
trace com.example.OrderService createOrder

# 输出示例（自动高亮耗时最长的分支）：
# +---[3.2ms] com.example.OrderService:validateParam()
# +---[856.3ms] com.example.OrderService:queryInventory()  ← 瓶颈在这里
# +---[2.1ms] com.example.OrderService:saveOrder()

# watch：观察方法入参、返回值、异常
watch com.example.OrderService createOrder '{params, returnObj, throwExp}' -x 2
```

### 5.2 分层排查思路

```
接口慢
├── 应用层
│   ├── 代码逻辑（Arthas trace）
│   ├── 锁等待（jstack 看 BLOCKED）
│   └── GC 停顿（jstat -gcutil）
├── 中间件层
│   ├── 数据库慢查询（慢 SQL 日志）
│   ├── Redis 大 Key / 热 Key
│   └── MQ 消费积压
└── 基础设施层
    ├── 网络延迟（ping / telnet）
    ├── 磁盘 IO（iostat）
    └── CPU / 内存不足（top / free）
```

### 5.3 数据库慢查询排查

```sql
-- 查看慢查询是否开启
SHOW VARIABLES LIKE 'slow_query%';

-- 查看当前正在执行的SQL
SHOW FULL PROCESSLIST;

-- 分析执行计划
EXPLAIN SELECT * FROM orders WHERE user_id = 123;
```

**常见慢 SQL 原因**：

| 原因 | 排查手段 | 解决方案 |
|------|---------|---------|
| 缺少索引 | `EXPLAIN` 看 `type=ALL` | 加索引 |
| 索引失效 | `EXPLAIN` 看 `key=NULL` | 检查函数/类型转换/`OR`/`LIKE '%x'` |
| 大事务/锁等待 | `SHOW ENGINE INNODB STATUS` | 拆分事务、减小锁粒度 |
| 深分页 | `LIMIT 100000, 10` | 改用游标分页 |

> **面试话术**：接口慢我会分层排查。先用 Arthas trace 定位到耗时最长的方法调用，如果指向数据库就看慢 SQL 日志和执行计划，如果指向下游服务就检查网络和对方响应时间。

## 七、面试高频问题

### Q1：线上 CPU 100% 怎么排查？

四步法：`top` 找进程 → `top -Hp` 找线程 → `printf "%x"` 转十六进制 → `jstack` 看堆栈。或直接用 Arthas `thread -n 3`。常见原因：死循环、正则回溯、频繁 GC。

### Q2：线上发生 OOM 怎么办？

1. 首先确保 JVM 参数已加 `-XX:+HeapDumpOnOutOfMemoryError`
2. 拿到 dump 文件后用 MAT 分析 Dominator Tree 找大对象
3. 通过 GC Roots 引用链定位持有者
4. 常见泄漏点：静态集合、ThreadLocal 未 remove、资源未关闭

### Q3：如何排查死锁？

`jstack <PID>` 直接搜索 "deadlock"，JVM 会自动检测并输出死锁链。Arthas 可以用 `thread -b` 检测阻塞。预防手段：统一加锁顺序、使用 `tryLock` 设置超时。

### Q4：Full GC 频繁怎么排查？

1. `jstat -gcutil` 观察 Old 区变化趋势
2. 如果 FGC 后 Old 区不降 → 内存泄漏 → dump 分析
3. 如果 FGC 后能降但很快又满 → 大对象频繁进入 Old 区 → 检查对象大小和晋升阈值
4. 如果是 CMS 的 Concurrent Mode Failure → 降低触发阈值

### Q5：接口突然变慢怎么排查？

分层排查：
1. **应用层**：Arthas `trace` 定位耗时方法，`jstat` 排除 GC 影响
2. **中间件层**：慢 SQL 日志、Redis 大 Key、MQ 积压
3. **基础设施层**：CPU/内存/磁盘 IO/网络

### Q6：生产环境有哪些必须加的 JVM 参数？

```bash
# OOM 自动 dump（必加）
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/logs/heapdump.hprof

# GC 日志（必加）
-Xlog:gc*:file=/data/logs/gc.log:time,uptime,level,tags

# OOM 时执行脚本（可选，用于告警）
-XX:OnOutOfMemoryError="kill -9 %p"
```
