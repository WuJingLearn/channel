# Java 懒加载单例模式（Lazy Initialization Singleton）

> **懒加载（Lazy Loading）**：单例对象在**第一次被使用时**才创建，而非类加载时就创建，目的是**节省内存、加快启动速度、避免不必要的初始化开销**。
>
> 与之相对的是**饿汉式（Eager Loading）**：类加载时就立即创建实例。
>
> 本文重点剖析**懒加载单例的 5 种实现方式**，并结合 JMM、JVM 类加载机制等底层原理，分析每种实现的**线程安全性、性能、适用场景**。

---

## 一、单例模式核心要素

无论哪种实现方式，单例必须满足以下三点：

1. **构造方法私有**（`private`）：防止外部 `new`
2. **持有自身的唯一实例**（`static` 字段）
3. **提供全局访问点**（`static` 方法）

懒加载单例还需额外满足：
- **延迟创建**：仅在第一次调用 `getInstance()` 时才实例化
- **线程安全**：多线程并发调用 `getInstance()` 时必须保证唯一性

---

## 二、五种懒加载实现方式对比

| 实现方式 | 线程安全 | 性能 | 推荐度 | 核心原理 |
|---------|---------|------|-------|---------|
| 1. 普通懒汉式 | ❌ 不安全 | 高 | ❌ 不推荐 | 直接判空 |
| 2. synchronized 方法 | ✅ 安全 | 低（每次加锁） | ⚠️ 一般 | 整个方法加锁 |
| 3. 双重检查锁（DCL） | ✅ 安全 | 高 | ⭐⭐⭐⭐ | volatile + 双重判空 |
| 4. 静态内部类（Holder） | ✅ 安全 | 高 | ⭐⭐⭐⭐⭐ | JVM 类加载机制 |
| 5. 枚举（Enum） | ✅ 安全 | 高 | ⭐⭐⭐⭐⭐ | JVM 枚举特性 |

## 三、实现方式详解

### 3.1 普通懒汉式（线程不安全，反例）

```java
public class LazySingleton {
    private static LazySingleton instance;

    private LazySingleton() {}

    public static LazySingleton getInstance() {
        if (instance == null) {
            instance = new LazySingleton();   // ⚠️ 多线程下可能创建多个实例
        }
        return instance;
    }
}
```

**问题**：
- 线程 A 和 B 同时通过 `instance == null` 判断
- 两者都执行 `new`，导致**创建出多个实例**，违反单例

**适用场景**：仅适用于单线程环境（基本不存在）。

---

### 3.2 synchronized 同步方法（线程安全，性能差）

```java
public class SyncSingleton {
    private static SyncSingleton instance;

    private SyncSingleton() {}

    public static synchronized SyncSingleton getInstance() {
        if (instance == null) {
            instance = new SyncSingleton();
        }
        return instance;
    }
}
```

**优点**：实现简单，线程安全。

**问题**：
- **每次调用 `getInstance()` 都要加锁**，但实际上只有第一次创建时需要同步
- 高并发下锁竞争严重，性能极差

**适用场景**：调用频率极低且实例化开销不大的场景。

---

### 3.3 双重检查锁 DCL（Double-Checked Locking）⭐

```java
public class DclSingleton {
    private static volatile DclSingleton instance;   // ⭐ 必须 volatile

    private DclSingleton() {}

    public static DclSingleton getInstance() {
        if (instance == null) {                       // 第一次检查（无锁，快速路径）
            synchronized (DclSingleton.class) {
                if (instance == null) {               // 第二次检查（加锁后再确认）
                    instance = new DclSingleton();    // 非原子操作！
                }
            }
        }
        return instance;
    }
}
```

#### 原理拆解

**① 为什么要双重判空？**
- **第一次判空**：避免每次进入都加锁，提升性能（**只有第一次创建时才会进入 synchronized 块**）
- **第二次判空**：进入同步块后再次确认，防止多个线程同时通过第一次判空后**重复创建**

**② 为什么必须用 `volatile`？**

`new DclSingleton()` 是**非原子操作**，JVM 将其拆为 3 步：

```
1. 分配内存空间
2. 调用构造方法初始化对象
3. 将引用赋值给 instance
```

由于 JIT 与 CPU 的**指令重排序**，2 和 3 可能被调换为 `1 → 3 → 2`。此时：

```
时刻 T1：线程 A 执行 1 分配内存
时刻 T2：线程 A 执行 3 将引用赋给 instance（此时对象未初始化）
时刻 T3：线程 B 调用 getInstance()，发现 instance != null，直接 return
时刻 T4：线程 B 使用 instance → ⚠️ 字段是默认值 / NPE
时刻 T5：线程 A 才执行 2 初始化对象
```

**`volatile` 的作用**：
- 禁止步骤 2 和步骤 3 之间的重排序（插入内存屏障）
- 保证 `instance` 的写入对其他线程立即可见
- 让其他线程要么看到 `null`，要么看到**完整初始化**的实例

> ⚠️ **JDK 5 之前**，由于 JSR-133 未发布，即使加 `volatile` 也无法完全解决 DCL 问题；**JDK 5 及之后**才安全。

**优点**：兼顾**懒加载 + 线程安全 + 高性能**。
**缺点**：实现复杂，容易写错（漏 `volatile`、忘记双重判空）。

### 3.4 静态内部类（Holder 模式）⭐⭐⭐⭐⭐

```java
public class HolderSingleton {

    private HolderSingleton() {}

    // 静态内部类：只有调用 getInstance() 时才会被加载
    private static class Holder {
        private static final HolderSingleton INSTANCE = new HolderSingleton();
    }

    public static HolderSingleton getInstance() {
        return Holder.INSTANCE;
    }
}
```

#### 原理：JVM 类加载机制天然保证

**① 懒加载机制**：

根据 JVM 规范，**类只有在被主动使用时才会被加载**。

- `HolderSingleton` 类加载时，**不会**触发 `Holder` 内部类加载
- 只有第一次调用 `getInstance()` 访问 `Holder.INSTANCE` 时，`Holder` 类才被加载并初始化
- 此时 JVM 才会执行 `new HolderSingleton()`，实现真正的懒加载

**② 线程安全机制**：

JVM 在执行类的 `<clinit>`（类初始化方法）时，会**自动加锁**：

> JVM 规范明确规定：**多个线程同时初始化同一个类时，只有一个线程会执行 `<clinit>`，其他线程必须等待**。

这意味着：
- `Holder.INSTANCE` 的初始化是**JVM 层面保证的线程安全**
- **无需手动加锁、无需 volatile、无需 synchronized**
- 性能等同于直接读取静态字段，零开销

**优点**：
- ✅ 实现简洁，代码可读性好
- ✅ 线程安全由 JVM 保证，无锁开销
- ✅ 真正的懒加载（调用时才初始化）

**缺点**：
- 无法防止**反射攻击**（可通过反射调用私有构造方法创建多个实例）
- 无法防止**反序列化攻击**（反序列化会创建新对象）

**推荐场景**：**绝大多数场景的最佳选择**。

### 3.5 枚举单例（Effective Java 推荐）⭐⭐⭐⭐⭐

```java
public enum EnumSingleton {
    INSTANCE;

    // 业务方法
    public void doSomething() {
        System.out.println("Hello Singleton");
    }
}

// 使用
EnumSingleton.INSTANCE.doSomething();
```

#### 原理：JVM 对枚举的天然保护

**① 线程安全**：

枚举常量本质是 `public static final` 字段，在类加载时由 JVM 保证**只初始化一次**，与静态内部类同理。

**② 防反射攻击**：

JVM 在 `Constructor.newInstance()` 中对枚举类型做了**特殊检查**：

```java
// JDK 源码：Constructor#newInstance
if ((clazz.getModifiers() & Modifier.ENUM) != 0)
    throw new IllegalArgumentException("Cannot reflectively create enum objects");
```

→ 反射创建枚举会直接抛异常。

**③ 防反序列化攻击**：

枚举的序列化机制由 JVM 特殊处理，反序列化时**只返回已存在的枚举实例**，不会创建新对象。

**优点**：
- ✅ **绝对单例**：JVM 保证天然线程安全、防反射、防反序列化
- ✅ 实现最简洁，仅一行代码
- ✅ 由 Joshua Bloch 在《Effective Java》中**强烈推荐**

**缺点**：
- ❌ **严格来说不是懒加载**：枚举常量在类加载时就初始化（除非配合 `Holder` 模式延迟枚举类加载）
- 无法继承其他类（枚举类已隐式继承 `java.lang.Enum`）

## 五、核心原理总结

### 5.1 对象实例化非原子性（DCL 的核心）

```
new Singleton() 拆解为：
  1. 分配内存
  2. 初始化对象（执行 <init> 构造方法）
  3. 将引用赋值给变量
```

步骤 2 与 3 之间可能被**重排序**，导致其他线程看到**半初始化对象**。
→ 解决方案：**`volatile`** 禁止重排序，保证安全发布。

### 5.2 类加载机制（Holder 模式的核心）

JVM 规范保证：
- **类只在被主动使用时加载**（懒加载）
- **类的 `<clinit>` 方法在多线程下由 JVM 自动加锁**（线程安全）

→ Holder 模式利用此机制，无需任何同步代码即可实现安全的懒加载。

### 5.3 JMM 三大武器对照表

| 武器 | 在单例中的作用 |
|------|----------------|
| `volatile` | DCL 中防止对象引用的发布重排序 |
| `final` | Holder/枚举中字段安全发布的兜底保证 |
| `synchronized` | 同步方法版本的互斥；类加载 `<clinit>` 的隐式互斥 |

## 六、防御性单例：抵御反射与反序列化攻击

### 6.1 反射攻击防御

```java
public class DclSingleton {
    private static volatile DclSingleton instance;

    private DclSingleton() {
        // ⭐ 在构造方法中拦截
        if (instance != null) {
            throw new IllegalStateException("单例已存在，禁止反射创建");
        }
    }

    public static DclSingleton getInstance() {
        if (instance == null) {
            synchronized (DclSingleton.class) {
                if (instance == null) instance = new DclSingleton();
            }
        }
        return instance;
    }
}
```

### 6.2 反序列化防御

实现 `readResolve()` 方法，返回已有的实例：

```java
public class HolderSingleton implements Serializable {
    private HolderSingleton() {}

    private static class Holder {
        private static final HolderSingleton INSTANCE = new HolderSingleton();
    }

    public static HolderSingleton getInstance() { return Holder.INSTANCE; }

    // ⭐ 反序列化时被调用，返回单例
    private Object readResolve() {
        return Holder.INSTANCE;
    }
}
```

> 而**枚举单例天然免疫**这两种攻击，无需额外代码。

## 七、实际工程选型建议

| 场景 | 推荐方案 |
|------|---------|
| 简单业务，无序列化/反射风险 | ⭐ **静态内部类（Holder）** |
| 需要绝对单例，对抗反射/反序列化 | ⭐ **枚举** |
| Spring 容器管理的 Bean | Spring 默认 `@Scope("singleton")`，无需自己实现 |
| 需要构造函数传参的延迟初始化 | DCL（但要严格保证 `volatile` 与双重判空） |
| 不在乎启动开销，调用频繁 | 饿汉式（非懒加载，但最简单可靠） |

## 八、面试加分要点

1. **DCL 必须 `volatile`**：因为 `new` 操作非原子（分配内存→初始化→赋值），可能发生 2↔3 指令重排序，导致其他线程读到半初始化对象。

2. **Holder 模式的"懒"由 JVM 类加载机制保证**：内部类只在被引用时才加载，且 `<clinit>` 天然线程安全。

3. **枚举单例是最佳实践**：JVM 层面防反射、防反序列化、线程安全。但严格意义上不是"懒加载"。

4. **`volatile` 在 JDK 5 之前 DCL 仍有问题**：JSR-133 在 JDK 5 重新定义了 `volatile` 的语义，才真正解决了 DCL 安全问题。

5. **构造方法私有不能防反射**：必须在构造方法内做"已存在则抛异常"的拦截。

---

## 九、面试一句话总结 ⭐

> **懒加载单例的核心是"延迟实例化 + 线程安全"**。常见有 5 种实现：普通懒汉式（线程不安全）、synchronized 方法（线程安全但每次加锁性能差）、**双重检查锁 DCL**（用 `volatile` + 双重判空，解决 `new` 操作非原子带来的指令重排序问题，是性能和安全的平衡）、**静态内部类 Holder 模式**（利用 JVM 类加载机制——内部类只在被引用时加载，且 `<clinit>` 由 JVM 隐式加锁，是最优雅的实现）、**枚举单例**（《Effective Java》推荐，JVM 天然保证线程安全、防反射、防反序列化，但严格说不是懒加载）。
>
> **工程实践中，绝大多数场景首选 Holder 模式，对单例性要求极致的场景首选枚举。**
