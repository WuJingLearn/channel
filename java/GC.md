# CMS vs G1 垃圾收集器对比

## 一、设计目标

| 维度 | CMS | G1 |
|------|-----|-----|
| 全称 | Concurrent Mark-Sweep | Garbage-First |
| 引入版本 | JDK 1.4.2 | JDK 7u4（JDK 9 起成为默认） |
| 核心目标 | **最短停顿时间**（低延迟） | **可预测的停顿时间**（在延迟和吞吐之间取平衡） |
| 当前状态 | JDK 9 标记为 Deprecated，JDK 14 移除 | 目前主流，JDK 9+ 默认收集器 |

## 二、内存布局

### CMS —— 传统分代

```
┌──────────────┬──────────────────────────────┐
│   Young Gen  │          Old Gen             │
│  (Eden + S0  │     (连续的一整块内存)         │
│    + S1)     │                              │
└──────────────┴──────────────────────────────┘
```

- 物理上分为 **Young / Old** 两块连续内存区域
- Young 区用 **ParNew** 收集（复制算法），Old 区用 CMS 收集（标记-清除）

### G1 —— Region 化分代

```
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ E  │ S  │ O  │ O  │ H  │ E  │ O  │ E  │
├────┼────┼────┼────┼────┼────┼────┼────┤
│ O  │ E  │ S  │ O  │ H  │ O  │ E  │ O  │
└────┴────┴────┴────┴────┴────┴────┴────┘
E=Eden  S=Survivor  O=Old  H=Humongous
```

- 堆被切分为大小相等的 **Region**（默认 2048 个，每个 1~32MB）
- 逻辑上仍有分代概念，但 Region 角色**动态切换**
- 大对象放入专门的 **Humongous Region**（超过 Region 大小 50% 的对象）

> **面试点**：G1 的 Region 化设计是它能做到"可预测停顿"的基础——每次 GC 不需要回收整个 Old Gen，只选择回收价值最高的若干 Region。

## 三、GC 过程对比

### CMS 四阶段

| 阶段 | STW | 说明 |
|------|-----|------|
| **1. 初始标记** | **是** | 标记 GC Roots 直接引用的对象，速度很快 |
| **2. 并发标记** | 否 | 从 GC Roots 出发遍历整个对象图，与用户线程并发执行 |
| **3. 重新标记** | **是** | 修正并发标记期间因用户线程继续运行而产生的变动（**增量更新**） |
| **4. 并发清除** | 否 | 清除不可达对象，与用户线程并发执行 |

### G1 的回收模式

G1 有三种回收模式：

**1. Young GC**：当 Eden 区 Region 用满时触发，回收所有 Young Region（STW，复制算法）

**2. Mixed GC（混合回收）**：回收所有 Young Region + **部分** Old Region

混合回收的并发标记阶段：

| 阶段 | STW | 说明 |
|------|-----|------|
| **初始标记** | **是** | 借助 Young GC 的 STW 完成，标记 GC Roots 直接引用 |
| **并发标记** | 否 | 遍历对象图，与用户线程并发 |
| **最终标记** | **是** | 处理 SATB 快照中残留的引用变更 |
| **筛选回收** | **是** | 按回收价值排序 Region，选择性回收（**复制算法**） |

**3. Full GC**：当混合回收跟不上分配速度时触发，单线程串行回收整堆（JDK 10 后改为多线程）

> **面试点**：G1 的筛选回收阶段是 STW 的（不同于 CMS 的并发清除），因为它使用复制算法移动对象，需要暂停用户线程来更新引用。

## 四、核心机制差异

### 4.1 写屏障与并发标记的正确性

并发标记时用户线程还在修改引用，可能导致**漏标**（活对象被误回收）。两者的解决方案不同：

| | CMS | G1 |
|--|-----|-----|
| 方案 | **增量更新（Incremental Update）** | **原始快照（SATB, Snapshot-At-The-Beginning）** |
| 原理 | 记录新增的引用关系，重新标记时再扫描一次 | 在并发标记开始时逻辑上对引用关系做快照，标记过程中被删除的引用也会被记录 |
| 重新标记耗时 | 相对**较长**（需要重新扫描变化） | 相对**较短**（只需处理 SATB 队列） |

> **面试点**：CMS 用增量更新，G1 用 SATB。SATB 的优势是最终标记阶段更快（停顿更可预测），代价是可能产生更多的浮动垃圾。

### 4.2 跨代引用处理

| | CMS | G1 |
|--|-----|-----|
| 方案 | **Card Table（卡表）** | **Remembered Set（RSet）** |
| 作用 | Old → Young 的引用记录在卡表中，Young GC 时不用全扫 Old 区 | 每个 Region 维护自己的 RSet，记录"谁引用了我"，避免全堆扫描 |
| 空间开销 | 小（整个堆只有一份卡表） | **大**（每个 Region 一份 RSet，通常占堆的 10%~20%） |

> **面试点**：G1 的 RSet 是实现 Region 级别独立回收的关键，但也是 G1 内存开销比 CMS 大的主要原因。

### 4.3 回收算法

| | CMS | G1 |
|--|-----|-----|
| Old 区算法 | **标记-清除** | **复制算法**（存活对象复制到空 Region） |
| 内存碎片 | **有碎片**，可能触发 Full GC 做压缩 | **无碎片**，每次回收都天然压缩 |

> **面试点**：CMS 最大的痛点就是内存碎片。当碎片严重到无法分配大对象时，会退化为 Serial Old 做 Full GC（单线程压缩整个 Old 区），这会导致极长的 STW。

## 五、停顿时间控制

| | CMS | G1 |
|--|-----|-----|
| 能否设定目标停顿 | **不能** | **能**，`-XX:MaxGCPauseMillis=200`（默认 200ms） |
| 控制方式 | 尽力而为，无量化手段 | 根据历史数据预测每个 Region 的回收耗时，在目标停顿内选择回收价值最高的 Region 组合 |

> **面试点**：G1 名字中的 "Garbage-First" 就是指优先回收垃圾最多的 Region（回收价值 = 可释放空间 / 预估耗时），这是它实现可预测停顿的核心策略。

## 六、适用场景

| 场景 | 推荐 | 原因 |
|------|------|------|
| 堆 < 4GB，对延迟敏感 | CMS（老项目）/ G1 | CMS 在小堆上表现好，但已废弃 |
| 堆 4GB~16GB | **G1** | G1 的 Region 化在中大堆上优势明显 |
| 堆 > 16GB | G1 或考虑 ZGC/Shenandoah | 超大堆 G1 的 RSet 开销增大 |
| 不能容忍碎片导致的 Full GC | **G1** | G1 复制算法天然无碎片 |
| JDK 版本 ≥ 9 | **G1** | CMS 已 deprecated，G1 是默认收集器 |

## 七、关键参数速查

```bash
# CMS 核心参数
-XX:+UseConcMarkSweepGC              # 启用 CMS
-XX:CMSInitiatingOccupancyFraction=75 # Old 区占用 75% 时触发 CMS
-XX:+UseCMSCompactAtFullCollection    # Full GC 时压缩（解决碎片）

# G1 核心参数
-XX:+UseG1GC                         # 启用 G1（JDK9+ 默认）
-XX:MaxGCPauseMillis=200             # 目标最大停顿 200ms
-XX:G1HeapRegionSize=8m              # Region 大小
-XX:InitiatingHeapOccupancyPercent=45 # 堆占用 45% 时启动并发标记
```

## 八、面试高频问题

### Q1：CMS 和 G1 最本质的区别是什么？

**内存布局不同**。CMS 是传统的连续分代，G1 是 Region 化的分代。Region 化使得 G1 可以做到：
1. 选择性回收（不用回收整个 Old 区）→ **可预测停顿**
2. 复制算法回收 Old 区 → **无碎片**

### Q2：CMS 什么情况下会退化为 Full GC？

三种情况：
1. **并发模式失败（Concurrent Mode Failure）**：CMS 并发清除还没完成，Old 区就满了 → 退化为 Serial Old
2. **晋升失败**：Young GC 时 Survivor 放不下，要晋升到 Old 区，但 Old 区碎片太多放不下连续空间
3. **元空间不足**：Metaspace 满了也会触发 Full GC

### Q3：G1 的 Mixed GC 和 Full GC 有什么区别？

- **Mixed GC**：回收所有 Young + 部分 Old Region，使用复制算法，停顿可控
- **Full GC**：回收整堆，JDK 10 前是**单线程**串行回收（非常慢），JDK 10 后改为多线程

### Q4：为什么 G1 用 SATB 而不用增量更新？

SATB 的最终标记阶段只需要处理 SATB 队列，耗时更短且更可预测，这与 G1 "可预测停顿"的设计目标一致。增量更新需要在重新标记时重新扫描脏卡，耗时波动大。

### Q5：G1 的 RSet 为什么比 CMS 的 Card Table 占用更多内存？

CMS 的 Card Table 只需要一份，记录 Old → Young 的引用方向。而 G1 每个 Region 都有自己的 RSet，记录所有"谁引用了我"，是一个**point-in**结构，粒度更细，数量更多，所以内存开销更大（通常占堆的 10%~20%）。
