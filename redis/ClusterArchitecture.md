# Redis Cluster 集群架构

## 一、Cluster 是什么，又解决了什么问题

Redis Cluster 是 Redis 官方在 3.0 版本推出的**分布式分片方案**。它最核心的两个特征是：**去中心化**和**数据分片**。

所谓"去中心化"，指的是集群里没有 Proxy、没有协调中心，所有节点地位对等，彼此通过 Gossip 协议交换状态；所谓"数据分片"，指的是数据按规则被打散到不同节点上，每个节点只保存一部分数据。

理解 Cluster 的定位，关键是把它和前两代方案对比：

- **主从复制**只解决了"读扩展"和"数据备份"，主挂了还要人肉切。
- **Sentinel** 在主从之上加了哨兵，能自动 failover，解决了"高可用"问题，但每个节点仍然持有全量数据，**容量和写入吞吐被单机上限锁死**。
- **Cluster** 才真正把数据切开放到多台机器上，容量和写 QPS 都能水平扩展，同时通过 Master-Replica 保留了 Sentinel 那套自动故障转移能力。

所以一句话总结：**Cluster = 分片（解决容量和性能） + 高可用（解决故障恢复）**。它的第一性目标是分片，高可用只是顺带做了。如果数据量不大、单机就能装下，其实用 Sentinel 更简单，不需要为了高可用强行上 Cluster。

需要注意的是，Cluster 的高可用是"有限高可用"：当某个 Master 挂掉而它没有 Replica，或者超过半数 Master 同时挂掉，集群就无法继续服务。这是分布式系统在 CAP 上的必然取舍。

## 二、数据怎么分片：哈希槽机制

Cluster 没有采用一致性哈希，而是用了一个更简单直接的方案——**哈希槽（Hash Slot）**。

整个 key 空间被预先划分为 **16384 个槽**，每个 key 通过下面这个公式确定归属：

```
slot = CRC16(key) & 16383
```

每个 Master 节点负责一段连续的槽。比如三主结构下，A 拿 0~5460，B 拿 5461~10922，C 拿 10923~16383。扩缩容的本质就是**把槽从一个节点搬到另一个节点**。

### 为什么是 16384 而不是 65536

这是个经典面试题，答案藏在 Gossip 协议的实现细节里：

1. **心跳包大小**：节点之间互发的 PING/PONG 要携带自己负责的槽位图（bitmap）。16384 位 = 2KB，65536 位 = 8KB。集群规模一大，这 6KB 差距乘以节点数 × 心跳频率，带宽浪费非常可观。
2. **集群规模上限**：作者明确说过 Cluster 不建议超过 1000 个节点，16384 个槽已经足够均匀分配，再多就是浪费。
3. **位图压缩率**：槽数少的时候，连续槽段更多，位图压缩效率更高。

记住一个关键词就够了：**Gossip 心跳包大小**。

### Hash Tag：让相关 key 落到同一个槽

分片带来一个副作用——多 key 操作会跨节点而失败。Cluster 给的解法是 **Hash Tag**：用 `{}` 把 key 中的一部分括起来，CRC16 只对花括号内的内容计算。

```
{user:1001}:profile  →  CRC16("user:1001") % 16384
{user:1001}:orders   →  CRC16("user:1001") % 16384   // 同一个槽
```

凡是需要原子操作的场景，都靠它来"绑定"相关 key：

- **MGET/MSET** 批量读写
- **MULTI/EXEC** 事务
- **EVAL** Lua 脚本（脚本里所有 key 必须同槽）
- **SINTER/SUNION/SDIFF** 集合运算
- **RENAME**、**SORT ... STORE** 等带目标 key 的命令

用 Hash Tag 时有几个常见坑：

- **粒度别太粗**。`{user}:1001` 会让所有用户挤在一个槽里，必然热点；应该细化到具体实体，写成 `{user:1001}`。
- **只识别第一对花括号**。`{a{b}}` 的 tag 是 `a{b`，不是 `b`。
- **空 tag 失效**。`{}:key` 退化为对整个 key 计算 hash。
- **天然牺牲均匀性**。同一个 tag 下 key 越多，所在 slot 就越大，迁移和负载都会受影响。

> 简单记：跨 key 操作必须靠 Hash Tag 绑定到同一槽，代价是分布不再均匀，所以 tag 粒度要尽量精细到具体实体实例。

## 三、节点角色与典型部署

集群里只有两种角色：

- **Master**：持有槽，承担所有写入和（默认情况下的）读取。
- **Replica（Slave）**：异步复制 Master 数据，平时不对外提供服务，Master 故障时顶上去。

最小高可用部署是 **3 主 3 从**：三个 Master 切分 16384 个槽，每个 Master 挂一个 Replica。少于 3 个 Master 没法做选举（多数派凑不齐），所以 3 是底线。

## 四、节点之间怎么通信：Gossip 协议

Cluster 没有中心节点，每个节点都自己维护一份"集群视图"——包括有哪些节点、各自负责哪些槽、谁是谁的主从、谁可能挂了。这份视图通过 Gossip 协议在节点间扩散同步。

Gossip 涉及四类消息：

- **PING / PONG**：节点之间相互的心跳，每次会随机带上几个其他节点的状态信息（"我所知道的 X 节点还活着"），靠这种"传话"让全网逐渐收敛。
- **MEET**：新节点入伙的握手消息。
- **FAIL**：某节点被判定为客观下线后的全网广播。

Cluster 用的是**集群总线端口**，固定为"数据端口 + 10000"。比如 Redis 监听 6379，那么 16379 就是节点间通信端口。

Gossip 协议的优劣很容易记：

- **优**：去中心化、扩展性好、容错性强，少数节点信息丢失不影响整体。
- **劣**：状态最终一致，存在收敛延迟；节点越多，心跳总量增长越快（O(N)）。这也是 Cluster 不建议超过 1000 节点的原因之一。

## 五、客户端如何找到正确的节点

客户端可以连接到集群的任意一个节点。节点收到命令后会先算槽归属，然后分三种情况处理：

**情况 1：槽就在当前节点**
直接执行并返回结果，没有任何额外开销。

**情况 2：槽在别的节点（永久）**
返回 **MOVED** 错误，告诉客户端"这个槽以后归别人了"：

```
客户端 -> Node A: GET user:1001
Node A: -MOVED 6789 192.168.1.2:6379
客户端 -> Node B: GET user:1001
Node B: "value"
```

智能客户端（JedisCluster、Lettuce 等）收到 MOVED 后会**更新本地的槽到节点的映射缓存**，下次就直连 Node B，不再走弯路。实际工程实现里，客户端通常不会只更新这一个槽——因为一次故障转移往往会让原 Master 名下所有槽同时换主，更经济的做法是直接重新拉一遍 `CLUSTER SLOTS`/`CLUSTER SHARDS`，把整张映射表刷新。

**情况 3：槽正在迁移（临时）**
返回 **ASK** 错误。客户端要先发一条 `ASKING`，再发原命令到目标节点：

```
客户端 -> Node A: GET user:1001
Node A: -ASK 6789 192.168.1.2:6379
客户端 -> Node B: ASKING
客户端 -> Node B: GET user:1001
Node B: "value"
```

ASK 的关键区别在于：**客户端不能更新本地缓存**，因为这只是一次性的"借道"，迁移没完成时大部分 key 还在源节点，下次还得先问源节点。

| 重定向 | 含义 | 客户端是否更新缓存 |
|--------|------|-------------------|
| MOVED | 槽已永久迁移 | 是 |
| ASK | 槽迁移进行中，本次临时改道 | 否 |

## 六、故障检测与自动故障转移

Cluster 的故障判定遵循"先怀疑、再公认"的两阶段流程，本质是个简化版 Raft。

### 6.1 故障检测：PFAIL → FAIL

**第一阶段：主观下线（PFAIL）**
节点 X 发现节点 Y 超过 `cluster-node-timeout`（默认 15 秒）没响应 PING，就在自己心里把 Y 标记为 PFAIL。这只是"X 单方面认为 Y 挂了"，不算数。

**第二阶段：客观下线（FAIL）**
X 通过 Gossip 收到其他节点对 Y 的状态汇报。如果发现**集群中过半数 Master**（包括自己）在最近 `NODE_TIMEOUT * FAIL_REPORT_VALIDITY_MULT`（默认 2 倍，即 2 × node-timeout）的时间窗口内都把 Y 标记成了 PFAIL，X 就把 Y 升级为 FAIL 并向集群广播 FAIL 消息，强制其他节点也把 Y 标记为 FAIL。这时 Y 才被正式宣告下线。

两个细节值得记住：

- **要求"过半 Master"** 是为了避免某个节点因网络抖动单方面误判，把好节点踢出去。
- **要求"在时间窗口内"** 是为了避免拿陈年旧报告凑数——超时的 PFAIL 报告会被丢弃。

### 6.2 故障转移：Replica 选举上位

Master 被判定 FAIL 后，它的 Replica 们开始抢着接班，整个流程类似 Raft 的 Leader 选举：

1. Replica 检测到自己的 Master 处于 FAIL 状态，先把集群的 `currentEpoch` 加 1，作为本次选举的纪元号。
2. 向集群中所有 Master 发送投票请求 `FAILOVER_AUTH_REQUEST`，请求中带上这个新的 `currentEpoch`。
3. 每个 Master 在**同一个 `currentEpoch` 内只能投一票**，投给第一个向它发起合法请求的 Replica，并回复 `FAILOVER_AUTH_ACK`。
4. 某个 Replica 在 `NODE_TIMEOUT * 2`（且不少于 2 秒）内拿到**过半 Master 的票**后晋升为新 Master：把自己的 `configEpoch` 设为本次选举的 `currentEpoch`，接管原 Master 的槽，并通过 PONG 在集群中广播新配置。

这里要分清两个 epoch，**面试容易混淆**：

- **`currentEpoch`**：集群级别的逻辑时钟，类似 Raft 的 term，约束"每个纪元每个 Master 只能投一票"，保证同一纪元内只有一个胜出者。
- **`configEpoch`**：节点级别的配置版本号，每次某节点的槽归属变更（晋升、reshard）就会刷新。集群比较"谁说的槽配置更新"靠它——`configEpoch` 大的胜出，从而在网络分区恢复后实现"last failover wins"的合并语义。

### 6.3 多个 Replica 时谁先上

如果一个 Master 挂了好几个 Replica，Cluster 不会让它们同时发起选举（那样容易选不出多数派），而是**通过延迟错峰**让数据最新的那个先抢到票。

每个 Replica 在 Master 进入 FAIL 后，按下面公式计算自己的发起延迟：

```
DELAY = 500ms + random(0~500ms) + SLAVE_RANK * 1000ms
```

其中：

- **`500ms` 固定延迟**：等 FAIL 状态在集群里扩散开。
- **`random(0~500ms)`**：打散并发，避免多个 Replica 同时发起请求互相抢票。
- **`SLAVE_RANK * 1000ms`**：核心机制。SLAVE_RANK 由所有 Replica 按"复制偏移量从大到小"排序得出，**偏移量最大的 rank = 0，次大的 rank = 1，依此类推**。rank 越小的 Replica 等待时间越短，能抢先发起选举。

效果就是：**数据最新的 Replica 大概率最先发起选举、最先拿到多数票胜出**，从而最大程度减少 failover 期间的数据丢失。

## 七、扩缩容与数据迁移

Cluster 扩缩容的本质就是**把槽从一个节点搬到另一个节点**，整个过程在线，不停服。

### 7.1 扩容步骤

```bash
# 1. 把新节点加进集群
redis-cli --cluster add-node new_host:port existing_host:port

# 2. 重新分片，把一部分槽迁过去
redis-cli --cluster reshard existing_host:port
```

新加的节点默认是"空 Master"，不持有任何槽。执行 reshard 才会真正分配数据。

### 7.2 槽迁移流程：六步法

迁移以**槽**为单位、**key**为粒度逐个搬运，整体时序如下：

```
源节点 A                                    目标节点 B
   │                                            │
   │  ① CLUSTER SETSLOT <slot> MIGRATING <B-id>
   │     "我正在把这个槽迁出去"                   │
   │                                            │
   │  ② CLUSTER SETSLOT <slot> IMPORTING <A-id>
   │                                            │  "我正在从 A 接收这个槽"
   │                                            │
   │  ③ CLUSTER GETKEYSINSLOT <slot> <count>
   │     拿到该槽下一批 key                       │
   │                                            │
   │  ④ MIGRATE <B-ip> <B-port> "" 0 timeout KEYS k1 k2 ...
   │  ───────────────────────────────────────>  │
   │     原子地把 key 搬过去                      │
   │                                            │
   │  ⑤ 重复 ③④，直到该槽内所有 key 搬完           │
   │                                            │
   │  ⑥ CLUSTER SETSLOT <slot> NODE <B-id>      │
   │     全网广播：这个槽以后归 B 了                │
```

`MIGRATE` 命令的原子性靠下面四步串起来的：

1. 源节点把 key 的 value + TTL 用 DUMP 序列化成二进制；
2. 通过 RESTORE 命令发给目标节点；
3. 目标节点写入并返回 OK；
4. 源节点删除本地 key。

Redis 主线程是单线程的，**这四步是在同一个事件循环里同步串行完成的**——也就是说，MIGRATE 期间源节点不会处理任何其他命令，所有客户端请求都会排队等待。这种"独占主线程"的执行模式天然保证了原子性：要么目标节点写入成功后源节点再删（成功），要么任何一步失败源节点保留原 key（失败回滚），**绝不会出现两边都有或都没有的中间态**。

### 7.3 迁移期间的请求怎么处理

迁移过程中，槽在源节点的状态是 MIGRATING、在目标节点是 IMPORTING。这时客户端访问该槽的 key，源节点会按"key 是否还在本地"来分流：

| key 在源节点？ | 处理方式 |
|--------------|---------|
| 在 | 直接执行（读写都正常） |
| 不在 | 返回 ASK，让客户端去目标节点 |

具体场景：

- **场景 1：key 还在源节点**——直接处理，零额外开销。
- **场景 2：key 已经搬过去了**——源节点本地查不到，回 ASK，客户端转发到目标节点。
- **场景 3：写入一个新 key**——源节点没这个 key，同样回 ASK，新 key 直接落到目标节点上，**避免迁移完还要再搬一次**。

这就是为什么迁移中用 ASK 而不是 MOVED：迁移没完成时槽的归属还没正式变更，绝大部分 key 仍在源节点，客户端缓存还不能改。

**多 key 命令在迁移期间的特殊情况**：如果一条命令带的多个 key（已经通过 Hash Tag 落到同一槽）此时**部分在源节点、部分在目标节点**，源节点既不能本地执行也不能简单 ASK（因为不是所有 key 都在目标），会直接返回 **`-TRYAGAIN`** 错误，告诉客户端"稍后重试"。等这一槽迁完，再次重试时所有 key 已经在目标节点上，命令就能正常执行了。

### 7.4 大 key 是迁移的最大敌人

`MIGRATE` 走的是 Redis 主线程的同步逻辑，序列化和传输期间整个节点都被阻塞。所以一个 500MB 的 Hash 在迁移时可能让源节点卡几秒钟，业务上会有明显抖动；如果超过 timeout，迁移还会直接失败。

应对办法只有两个：

1. **提前拆大 key**——把巨型 Hash/Set 按 hash tag 拆成多个子 key。
2. **挑业务低峰期迁移**，并把 timeout 调大一点。

### 7.5 缩容步骤

缩容是扩容的逆过程：

```bash
# 1. 把待移除节点的所有槽迁到其他节点
redis-cli --cluster reshard host:port

# 2. 槽全部清空后，从集群里移除该节点
redis-cli --cluster del-node host:port node-id
```

注意顺序：**先迁数据、后删节点**，否则会丢数据。

## 八、Cluster 的限制

由于数据被切散在多个节点，很多单机时代理所当然的能力在 Cluster 里都打了折扣：

- **多 key 操作**：只支持同一槽内的多 key 命令，跨槽会直接报错，需要靠 Hash Tag 解决。
- **事务（MULTI/EXEC）**：只能在同一节点内执行，跨节点事务不支持。
- **数据库**：只能用 db0，`SELECT 1` 会报错。
- **Lua 脚本**：脚本访问的所有 key 必须落在同一槽。
- **Pub/Sub**：消息会广播到所有节点，集群规模一大开销显著，建议改用专门的消息中间件。
- **集群规模**：官方建议不超过 1000 个节点，超过后 Gossip 开销陡增。

这些限制本质上都是分布式系统的"原罪"：数据分散意味着原本依赖单点全局视角的能力都要打折。

## 九、关键配置参数

```conf
cluster-enabled yes                # 开启集群模式
cluster-config-file nodes.conf     # 集群配置文件，由 Redis 自动维护
cluster-node-timeout 15000         # 节点超时（ms），影响故障判定速度
cluster-require-full-coverage yes  # 是否要求所有槽都有节点负责才对外服务
cluster-replica-validity-factor 10 # Replica 数据过期判定因子
cluster-migration-barrier 1        # Master 至少保留的 Replica 数
cluster-allow-reads-when-down no   # 节点下线时是否允许读
```

需要重点关注两个：

- **`cluster-node-timeout`**：故障检测的总开关。设小了网络抖动就误判，设大了故障恢复慢，生产一般 10~15 秒。
- **`cluster-require-full-coverage`**：默认 `yes`，意味着只要任何一个槽没人负责整个集群就拒绝服务。生产建议改成 `no`，让没受影响的槽继续提供服务，把损失降到最小。

## 十、和其他方案的横向对比

| 方案 | 架构 | 优点 | 缺点 |
|------|------|------|------|
| 主从 + Sentinel | 中心化哨兵 | 简单直接 | 不分片、容量受单机限制 |
| Twemproxy | Proxy | 轻量、客户端无感 | 不支持动态扩缩容 |
| Codis | Proxy + ZooKeeper | 兼容单机命令、扩缩容方便 | 需额外维护 Proxy 和 ZK |
| **Redis Cluster** | 去中心化 | **官方支持、无额外组件** | 多 key 操作受限 |

简单记忆：**功能多但运维重的是 Codis，运维轻但限制多的是 Cluster**。

## 十一、生产实践建议

最后是几条容易被问到的实战经验：

1. **至少 3 主 3 从**，少于这个规模做不到自动 failover。
2. **避免跨机房部署**，Gossip 对网络延迟很敏感，跨地域容易误判 PFAIL。
3. **`cluster-node-timeout` 控制在 10~15 秒**，平衡故障发现速度和误判率。
4. **使用智能客户端**（JedisCluster、Lettuce），让槽缓存本地化，减少 MOVED 重定向。
5. **持续监控槽分布的均匀性**，及时发现 Hash Tag 用错导致的热点。
6. **预留扩容空间**，提前规划 Hash Tag 粒度，别等数据倾斜了再补救。
7. **大 key 治理是常态工作**，不仅影响迁移，也影响主从复制和 AOF 重写。

> **一句话面试金句**：Redis Cluster 是去中心化的分片方案，它用 16384 个哈希槽切分数据，节点间靠 Gossip 维护视图，通过 Master-Replica 自动 failover 提供有限的高可用，本质是"分片为主、高可用为辅"的分布式 Redis。
